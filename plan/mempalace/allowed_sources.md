# MemPalace 首批允许采集范围清单

## 1. 目标

为 `MemPalace` 第一期内部知识记忆层冻结首批允许采集的目录、文件类型和更新节奏，只覆盖最新权威需求与场景资料，不扩展到真实客户正式数据或原始输入池。

## 2. 纳入原则

- 只纳入 `W01` 已冻结的 `L1` 权威定义层目录：`P0/`、`P1/`、`docs/事务所流程/`
- 只纳入可稳定解析、可引用、可审阅的文本或结构化文本文件
- 相邻目录或文件若未出现在本清单中，默认不进入一期白名单
- 本清单只定义“允许采集”，不处理 `_output`、`_raw` 的权重或放行

## 3. 首批白名单

| 目录 / 模式 | 允许文件类型 | 推荐更新频率 | 纳入理由 | 使用限制 |
|---|---|---|---|---|
| `docs/gyoseishoshi_saas_md/P0/*.md`、`docs/gyoseishoshi_saas_md/P0/**/*.md` | `.md` | 文档合入后同日增量刷新；每周一次全量复扫 | `P0` 是首版试点权威源，定义范围、不做项、规则、流程、页面职责与数据模型 | 以正文为主；`09-结构化总索引与交叉映射.md` 只做导航，不单独提升为规则源 |
| `docs/gyoseishoshi_saas_md/P0/06-页面规格/*-需求门禁/artifacts/` | `.json`、`.md`、`.txt` | 需求门禁工件重生成后立即增量刷新 | 该目录下工件已把页面规格边界、歧义和执行约束结构化，适合作为检索辅助上下文 | 仅作为 `P0` 正文的结构化补充，不覆盖正文口径 |
| `docs/gyoseishoshi_saas_md/P1/*.md`、`docs/gyoseishoshi_saas_md/P1/**/*.md` | `.md` | 文档合入后同日增量刷新；每周一次全量复扫 | `P1` 承接扩展范围、扩展模块和实施顺序，是经营管理签扩展的权威源 | 仅回答 P1 扩展问题，不可反向改写 `P0` 底座定义 |
| `docs/gyoseishoshi_saas_md/P1/*需求门禁/artifacts/` | `.json`、`.md`、`.txt` | 需求门禁工件重生成后立即增量刷新 | 该目录提供 P1 高仿真原型与扩展需求的结构化约束，可用于补全实施上下文 | 仅作为 `P1` 正文的结构化补充，不单独升格为规则源 |

| `docs/事务所流程/*.scenarios/*.md` | `.md` | 场景文件更新后立即增量刷新 | 场景拆分文件将资料清单按案件场景拆开，适合直接检索和回链 | 视为场景资料层，不直接定义产品实现边界 |
| `docs/事务所流程/*.config.json`、`docs/事务所流程/*.config.yaml` | `.json`、`.yaml` | 配置变更后立即增量刷新 | 配置文件可帮助理解场景拆分、字段分组和资料映射关系 | 只能辅助解析和索引，不可单独作为最终业务结论 |

## 4. 一期明确不纳入的相邻文件

以下文件虽然位于相邻目录，但未进入首批白名单：

- `docs/事务所流程/*.xlsx`
- `docs/事务所流程/*.docx`
- 任何图片、压缩包、二进制附件或导出文件
- `docs/gyoseishoshi_saas_md/_output/`
- `docs/gyoseishoshi_saas_md/_raw/`

这些范围后续若要纳入，必须由后续任务单独冻结规则，不得通过本白名单顺手放行。

## 5. 最小执行约束

1. 先扫 `P0` 与 `P1` 正文，再补充读取同层需求门禁工件。
2. 涉及场景资料时，才下探到 `docs/事务所流程/`。
3. 采集器遇到不在第 3 节中的目录或后缀，默认拒绝。
4. 同一目录下若同时出现白名单与非白名单文件，只采白名单文件，不做目录级“整包放行”。
