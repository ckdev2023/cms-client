# MemPalace 验收问题集

## 1. 目标

整理一组可直接用于首轮验收的高价值问题，验证 `MemPalace` 是否真的能支持长期使用，而不是只会做模糊检索。

本文只定义“该问什么、怎么判定是否答对”，不执行真实问答测试，不展开回答协议，也不编写评估模板。

## 2. 设计原则

- 问题必须来自真实高频追溯场景，优先验证“为什么”“依据是什么”“冲突时以谁为准”。
- 每条问题都必须能判定成功或失败，不能停留在泛泛搜索题。
- 问题分为三类：来源追溯、规则解释、历史决策。
- 验收时仍遵守固定来源优先级：`P0/P1/事务所流程 > _output > _raw`。
- 若答案只能落到 `_output` 或 `_raw`，必须显式降级，不能伪装成权威结论。

## 3. 使用方式

每条问题至少检查以下四项：

1. 是否命中了正确主题，而不是只返回关键词片段。
2. 是否给出了可回链的来源路径和来源层级。
3. 是否在冲突场景下遵守 `L1 > L2 > L3`。
4. 是否明确回答了问题本身，而不是只罗列相关文档。

建议把问题按三组依次执行：

- 先跑“来源追溯”，确认不会错用来源。
- 再跑“规则解释”，确认能回答业务为什么成立。
- 最后跑“历史决策”，确认能追到冻结、纠偏和回灌过程。

## 4. 成功判定通用规则

| 检查项 | 通过标准 | 失败信号 |
|---|---|---|
| 结论完整性 | 明确回答问题，不只贴片段 | 只返回关键词命中或文档列表 |
| 来源层级 | 至少给出一个 `L1` 来源；若引用 `_output`，同时给出对应 `L1` | 只给 `_output`、`_raw` 或不标来源层级 |
| 冲突处理 | 能说明为何以 `P0` / `P1` / `事务所流程` 为准 | 把低层来源当成最终定案 |
| 可追溯性 | 能指出具体文档或章节类型 | 只能说“文档里提到过” |
| 判定性 | 答案能直接判断对错 | 表述模糊，无法验证是否命中要求 |

## 5. 验收问题

### 5.1 来源追溯类

这组问题验证 `MemPalace` 是否能先找对来源，再给答案。

| ID | 问题 | 重点验证 | 成功信号 |
|---|---|---|---|
| `AQ-S01` | P0 案件主阶段的唯一权威定义在哪里？请指出 `S1-S9` 应该优先引用哪一层来源。 | `state-machine` + 来源优先级 | 明确以 `P0` 为首要来源，指出 `_output` 只能桥接、不能覆盖 |
| `AQ-S02` | 经营管理签的 `CaseWorkflowStep` 应该去哪里找？它为什么不能改写 `S1-S9`？ | `biz-mgmt` 与 `state-machine` 分层 | 同时回链 `P1` 与 `P0`，说明“P1 扩展、P0 底座” |
| `AQ-S03` | 某个经营管理签资料要求如果在 `事务所流程` 与 `P1` 都出现，应该先看谁？什么情况下需要同时引用两者？ | `scenario-materials` + 裁决顺序 | 先用 `事务所流程` 回答资料场景，再说明何时回链到 `P1` 落地位置 |
| `AQ-S04` | 如果 `_output` 中有一条关于状态机的整理结论，而 `P0` 已有冻结规则，最终应以谁为准？ | `_output` 降级规则 | 明确回答“以 `P0` 为准”，并把 `_output` 作为桥接来源 |
| `AQ-S05` | `P0` 门禁工件和 `P0` 正文同时命中时，哪一个负责最终规则解释？ | 正文与工件的主次关系 | 指出正文优先、工件只做结构化补充 |
| `AQ-S06` | “经营管理签认定 1 年需要哪些资料”这个问题，应该主要落在哪个 `wing/room`，为什么？ | `office-process` + `scenario-materials` 归类 | 能回答 `office-process/scenario-materials`，并解释不是 `P0` 规则题 |
| `AQ-S07` | “发 COE 前为什么会被阻断”这个问题，应该先追 `P1` 还是 `P0`？ | 通用 Gate 与专属 Gate 的裁决顺序 | 明确这是 `P1` 专属验收问题，但仍需要回链 `P0` 通用 Gate 底座 |
| `AQ-S08` | 若某结论只能在 `_raw` 中找到，而 `L1` 没有冻结定义，系统应该怎样回答？ | `_raw` 只能作为线索 | 明确输出“待编译/待确认”，不能伪装成既定规则 |

### 5.2 规则解释类

这组问题验证 `MemPalace` 是否能解释规则成立的原因、门槛和边界。

| ID | 问题 | 重点验证 | 成功信号 |
|---|---|---|---|
| `AQ-R01` | 为什么补正流程发生后，案件主阶段仍保持在 `S7`，而不是回退到未提交阶段？ | `state-machine` + `submission-audit` | 明确提到“补正属于已提交案件中的新循环”，并指出要生成新的 `SubmissionPackage` |
| `AQ-R02` | Gate-A、Gate-B、Gate-C 分别在什么时点触发？通过后各自会产生什么结果？ | `workflow-gates` | 能给出三个 Gate 的触发点和通过后动作，而不是只说“提交前要校验” |
| `AQ-R03` | 为什么 P0 的欠款通常只做风险提示，不默认阻断提交？ | P0 收费策略 | 明确引用 P0 的 `warn` 口径，并说明阻断不是首版统一规则 |
| `AQ-R04` | 为什么经营管理签到了 `COE_SENT` 前，尾款未结清时必须硬阻断？ | P1 专属收费门禁 | 能指出 `BillingPlan.gate_trigger_step=COE_SENT` 和 `gate_effect_mode=block` 的扩展口径 |
| `AQ-R05` | `source_type`、`visa_type`、`quote_amount` 各属于哪个对象？为什么不能挂错？ | `field-ownership` | 能给出 `Lead / Case` 的正确归属，并明确常见错挂项 |
| `AQ-R06` | 为什么 `Case.group` 被定义成案件归属快照，而不是永远跟着 `Customer.group` 实时变化？ | 归属继承链与审计要求 | 明确“默认继承 + 后续变更不回写历史案件 + 转组要留痕” |
| `AQ-R07` | 为什么系统把“资料项”和“附件版本”拆成两层，而不是直接在资料项上覆盖上传？ | 资料模型不可变规则 | 能说明“资料项是要求、附件是版本；新版本追加，不覆盖历史” |
| `AQ-R08` | 为什么提交后不能直接覆盖旧提交内容，而必须新建提交包？ | `submission-audit` | 明确指出“提交包锁定具体版本与字段快照”，补正必须生成新包 |
| `AQ-R09` | 为什么 `waived` 的资料项要从完成率分母里剔除，但又必须记录原因和操作人？ | 资料项治理规则 | 同时解释完成率口径与审计口径 |
| `AQ-R10` | 为什么服务端在点击“正式提交”时必须重新校验，而不能只信页面当前状态？ | 服务端重校验口径 | 明确提到关键资料/字段/文书变更后必须重新校验 |
| `AQ-R11` | 为什么 P0 不引入 `Company` 实体，而把工作类案件雇主信息沉淀在 `Case.employer_*` 字段？ | P0 范围与字段承载边界 | 能把 P0 建模策略和字段落点一起讲清楚 |
| `AQ-R12` | 为什么经营管理签的在留期间、续签提醒属于 P1 扩展，而不是 P0 默认链路？ | `biz-mgmt` + P0/P1 边界 | 明确指出 P0 手动 Reminder 兜底、P1 才启用 `ResidencePeriod + reminder_schedule_blueprint` |

| `AQ-H01` | “P0 状态机口径已冻结”这一结论是在哪些权威文档和哪些编译产出中被确认的？ | `_output` 到 `L1` 的回灌链 | 同时给出 `_output` 条目和对应 `P0/03`、`P0/04` 权威落点 |

| `AQ-H07` | “经营管理签成功结案前必须已录入在留期间并生成续签提醒”这一口径来自哪里？ | P1 成功结案前置条件 | 明确指出来自 `P1` 扩展，不可误说成 P0 通用规则 |
| `AQ-H08` | 首轮索引为什么要先跑 `L1`，再考虑 `_output` 和 `_raw`？这个顺序背后的治理理由是什么？ | 索引基线与来源治理联动 | 能把来源优先级、批次放行和低层来源降级规则串起来说明 |

## 6. 最小通过线

建议把问题集作为分层门槛使用：

| 维度 | 最低通过线 |
|---|---|
| 总体数量 | 至少 20 题 |
| 来源追溯类 | 至少通过 6 题，且 `AQ-S01`、`AQ-S04`、`AQ-S08` 必须通过 |
| 规则解释类 | 至少通过 8 题，且 `AQ-R01`、`AQ-R04`、`AQ-R08` 必须通过 |
| 历史决策类 | 至少通过 5 题，且 `AQ-H01`、`AQ-H02` 必须通过 |
| 红线 | 任何题把 `_raw` 当最终权威，或把 `P1` 扩展说成可改写 `P0` 底座，视为整轮不通过 |

## 7. 不合格答案的典型特征

- 只返回文档名，不回答“为什么”。
- 只会命中 `_output`，不给 `L1` 回链。
- 把 `事务所流程` 场景资料当成产品规则定案。
- 把经营管理签业务子步骤说成 `S1-S9` 的替代物。
- 只给模糊摘要，无法判断是否答到了具体 Gate、字段或阶段。

## 8. 当前结论

- 这组问题优先覆盖 `state-machine`、`field-ownership`、`workflow-gates`、`biz-mgmt`、`scenario-materials`、`submission-audit` 六类高价值主题。
- 题目总数为 28 条，满足“不少于 20 条”的要求。
- 每条题目都附带了明确的成功信号，便于后续做人工验收或半自动评分。
- 问题集刻意把来源追溯、规则解释、历史决策分开，避免把“能搜到”误判为“值得长期使用”。
