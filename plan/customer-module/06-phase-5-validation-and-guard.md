# Phase 5：验证、修复与门禁

## 阶段目标

- 对前面所有阶段的改动做统一验证。
- 确保客户模块改造在仓库门禁下可合入主线。

## 前置条件

- Phase 1 ~ Phase 4 的代码与测试均已完成

## 原子任务

| ID | 原子任务 | 说明 | 依赖 |
|---|---|---|---|
| `QA-001` | 跑 customers/contact-persons 相关 server 单测 | 优先最小范围执行 | `SV-012` |
| `QA-002` | 跑 admin customers 相关单测 | 必须 mock 网络 | `FE-013`、`P1-009` |
| `QA-003` | 运行 `npm run fix` | 收尾顺序第一步 | 所有改动完成 |
| `QA-004` | 运行 `npm run guard` | 收尾顺序最后一步 | `QA-003` |

## 执行顺序

1. 先跑目标单测，发现问题先修复再回归
2. 通过后运行 `npm run fix`
3. 最后运行 `npm run guard`

## 阶段验收

- 目标测试全部通过
- `npm run fix` 正常完成
- `npm run guard` 通过
- 不引入真实网络请求或新的门禁回归问题

## 记录要求

- 记录每次失败的命令、错误摘要、修复动作、回归结果
- 若 guard 阻塞，必须标出具体文件与规则类型
- 若存在遗留风险，必须回填到总计划或对应 phase 文档
