## 改动前必做

- 改动前必须先调用 codebase-retrieval 或 view 工具充分理解目标文件与相关符号，禁止"盲写"
- 改动前必须确认所涉及函数/类/接口的签名与实现均存在于仓库中，不得凭臆断使用不存在的 API
- 若需引入新依赖必须使用 `npm install`，禁止手动编辑 `package.json`

## 改动范围

- 只做用户明确要求的改动，禁止顺手"顺带优化"无关文件
- 禁止主动创建文档文件（*.md、README）或注释总结文件，除非用户明确要求
- 优先编辑现有文件；仅在功能逻辑真正需要新文件时才创建，且须符合分层规范

## 下游完整性

- 每次改动后必须用 codebase-retrieval 检查所有下游调用方、接口实现方、子类、测试，并一并更新
- API 签名（参数/返回类型）变更后，必须同步更新全部调用点；遗漏下游改动视为不完整提交

## 架构约束（强制）

- 严格遵守分层约束：feature 不得直接依赖 data/infra/tamagui；domain/data 不得依赖 shared/ui
- feature 之间不得直接互相依赖；跨 feature 协作必须通过 domain/shared 或 public 出口
- 新增/修改业务逻辑必须在 feature model 层（Hook/ViewModel），禁止在组件/页面中堆逻辑

## 测试与门禁

- 新增或修改逻辑必须补充或更新对应单测；优先覆盖 model/domain/data 层
- 禁止在测试中发起真实网络请求（必须 mock）
- 每次改动结束后须确认 `npm run guard` 通过，方可视为完成
- 单文件不得超过 500 行；方法保持单一职责，避免逻辑堆叠

## 安全与锁文件

- 禁止在代码中提交任何疑似敏感信息（token/密钥/密码）
- 仅使用 npm 管理依赖，仓库内只允许存在 `package-lock.json`
