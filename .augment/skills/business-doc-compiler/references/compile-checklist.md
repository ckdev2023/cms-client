# Business Doc Compiler — Compile Checklist

## 1. 变更定位速查

编译前先判断改动归属层级，再定位入口文档。

| 改动类型 | 首先修改 | 然后检查（是否需要同步引用） |
|---------|---------|--------------------------|
| 改产品定位 / 目标用户 / 非目标 | 01 产品定位与边界 | — |
| 改版本范围 / P0-P2 边界 / 排除项 | 02 版本范围与优先级 | 不改规则正文 |
| 改冻结口径 / 对象边界 / 不可变规则 | 03 业务规则与不变量 | 04 流程 → 06 页面规格 → 07 数据模型 |
| 改阶段 / Gate / 状态转移 | 04 核心流程与状态流转 | 03 规则 → 06 页面规格 → 07 数据模型 |
| 改导航结构 / 全局交互 | 05 信息架构与页面地图 | 06 页面规格 |
| 改某页面字段 / 交互 / 聚合口径 | 06 页面规格（对应模块） | 07 数据模型 |
| 改字段 / 实体 / 关系 / 约束 | 07 数据模型 | 06 页面规格 → 04 流程 |
| 改概念语义 / 术语定义 | 08 术语表 | 全文档引用一致性 |
| 改校验门槛项 | 03 规则 → 04 流程（Gate 触发点） | 06 页面规格 → 07 数据模型 |
| 改审计事件类型 | 03 规则 | 07 数据模型 → 04 流程 |
| 改收费 / 欠款策略 | 03 规则 | 04 流程 → 06 页面规格 |

## 2. P0 过滤

编译到 P0 目录的内容必须通过以下测试：

> **"若删掉这部分，首个试点是否仍可跑通？"**

- 答案是"不能" → 属于 P0，编译到 `docs/gyoseishoshi_saas_md/P0/`
- 答案是"能" → 不属于 P0，回主目录维护

## 3. 模板选择

| 目标文档层级 | 模板 | 必须段落 |
|------------|------|---------|
| 入口层（00、02-产品全景图、README） | 模板 A | 适合谁看 / 阅读后你会知道什么 |
| 规则/流程/数据（01–05、07） | 模板 B | 统一头部表六字段（回答什么/权威范围/不负责什么/先读/上游/下游） |
| 页面规格（06-页面规格/） | 模板 C | 模块元信息表 / 典型任务 / 列表页 / 详情页 / 关键动作 / 状态与异常 / 权限 / 规则引用 |
| 术语层（08） | 模板 B | 同上 |
| 维护层（99） | 无固定模板 | 变更日志表 |

## 4. 编译后检查

每次编译完成后逐项确认（摘自 99-文档维护与版本记录 §固定检查表）：

- [ ] 是否在 99 文档的"变更日志"中记录了本次变更？
- [ ] README 中的文档层级表是否需要更新？
- [ ] 如果新增了概念，是否先在 08-术语表中定义？
- [ ] 新增文档是否按所属层级选用了对应模板？
- [ ] 规范正文层文档是否包含完整的统一头部表？
- [ ] 页面规格文档是否包含全部必须段落？
- [ ] 00-开始这里的跳转表和阅读路径是否需要更新？
- [ ] 文件名编号、H1 标题编号、编号对照表三者一致？
- [ ] 站内交叉引用全部使用物理文件名？

## 5. 双写检测

以下概念只应在一个权威源中完整定义。编译时检查是否引入了重复：

| 概念类型 | 唯一权威源 | 常见双写陷阱 |
|---------|-----------|-----------|
| 业务规则（冻结口径/对象边界） | 03-业务规则与不变量 | 在 06 页面规格或 07 数据模型中重复定义 |
| 状态枚举值 | 03-业务规则 §状态机 | 在 04 流程或 07 数据模型中重复完整枚举 |
| 实体字段定义 | 07-数据模型设计 | 在 06 页面规格中重新定义字段类型 |
| 术语语义 | 08-术语表 | 在任何文档中用不同措辞重新解释 |
| Gate 触发条件 | 03-业务规则 + 04-流程 | 在 06 页面规格中重复触发逻辑 |

正确做法：下游文档用链接引用上游权威源的具体章节，最多附枚举值速查表。
