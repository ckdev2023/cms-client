# 任务文档模板

对每个任务文档，使用以下结构并尽量完整填写。

# [序号]-任务-[slug]

## 1. 任务目标
- 用一句话说明本任务唯一要完成的结果。

## 2. 背景上下文
- 只写执行本任务必须知道的背景。
- 不要复制整个项目背景。
- 如果有前置任务，只写与本任务直接相关的结果。

## 3. 输入材料
- 需求/设计摘要：
- 相关模块/目录：
- 相关接口/表/页面：
- 依赖前置：

## 4. 修改范围
- 明确允许修改的文件、模块、层次或资源。
- 如果具体文件未知，至少写清模块边界。

## 5. 非目标 / 禁止改动
- 明确本任务不负责什么。
- 写清不能顺手做的优化、重构、扩展。

## 6. 共享热点检查
- 本任务可能竞争的共享文件：
- 若存在共享文件，是否已拆成独立前置任务：

## 7. 实施步骤
- 步骤 1：
- 步骤 2：
- 步骤 3：
- 步骤 4：

要求：
- 步骤要能直接执行。
- 不要把多个独立子任务揉成一步。

## 8. 验收标准
- 列出可以客观检查的完成条件。
- 尽量写成接口返回、页面行为、测试结果、构建结果、数据状态等。

## 9. 自测清单
- 单测：
- 集成验证：
- 手工检查：
- lint / type check / build：

## 10. 结果工件
- 必须输出的 result json：
- 必须输出的 summary md：

## 11. 回滚 / 撤销方案
- 如果本任务失败，如何安全撤销。
- 如果需要保留中间产物，也要说明。

## 12. 交付物
- 代码变更：
- 配置变更：
- 文档变更：
- 测试变更：

## 13. 停止条件
- 遇到哪些情况必须停止，不得自行扩 scope。
- 例如：需要改 schema、需要改鉴权、需要跨模块重构、需求前提不成立。

## 14. 执行提示
- 先做最小闭环。
- 严禁顺手重构无关代码。
- 若发现设计冲突，先记录并停止。
