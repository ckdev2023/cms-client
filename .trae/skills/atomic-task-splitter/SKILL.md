# Atomic Task Splitter（原子任务拆分器）

## 目标

把用户的需求/问题拆成一组可执行的原子任务，满足：

- 每个任务在一个新窗口里只依赖该任务卡片提供的上下文即可推进
- 若任务仍偏大，递归继续拆分，直到“单人一次专注会话可完成并可验证”
- 任务之间的上下游依赖清晰可追溯（可并行、可串行、可回滚）

## 何时调用

- 需求包含多个模块/多文件改动/跨层协作/需要测试与门禁验证
- 用户希望“拆成原子任务”“分窗口完成”“明确依赖与串并行关系”
- 需要把一件大事拆成可交付、可验收、可逐步合并的小步改动

## 输入（从对话中提取）

- 业务目标：要达成什么结果（不是做什么按钮）
- 约束：平台/架构门禁/不可引入依赖/必须通过的命令/交付边界
- 现状：已有模块、入口、数据结构、已知问题、复现步骤（如果有）
- 风险点：可能影响的功能、迁移/兼容、性能、安全、权限

若信息缺失，不要停下来问用户；先做“合理默认假设”，并在输出里用“假设”段落列出，确保任务仍可落地推进。

## 原子任务判定标准（必须全部满足）

- 单一目标：只解决一个可描述清楚的变更点
- 可验证：有明确的验收口径（测试、命令、界面行为、日志/指标）
- 可回滚：能描述如何撤销或安全降级
- 低耦合：不强依赖未完成的大块工作；若有依赖，必须标注并最小化
- 上下文自足：任务卡片包含完成该任务所需的关键上下文入口（文件路径/函数名/搜索词/命令）

若任一条不满足，必须拆分或补全任务卡片信息。

## 输出格式（强制）

### A. 总览

1) 假设（如有）
2) 依赖图（DAG）：

- 用 `T1 -> T3` 表达依赖
- 标注可并行的任务集合（同一层级）

3) 里程碑（可选）：把任务按“可合并/可上线”的粒度分组

### B. 原子任务清单

对每个任务输出一张“任务卡片”，字段固定如下：

- ID：T1 / T2 / …
- 标题：动词开头，10–20 字
- 目标：1 句话说明要实现的结果
- 非目标：明确不做什么，避免范围漂移
- 依赖：列出前置任务 ID（无则写“无”）
- 影响面：可能受影响的模块/页面/流程
- 变更点：要改哪些类型的东西（类型/模型/接口/页面/测试/脚本）
- 预估文件：给出 3–8 个可能涉及的文件路径（不确定可给目录）
- 上下文包：
  - 必读入口：文件路径 + 关注点（或函数名/组件名）
  - 快速检索：3–6 个推荐搜索词（用于在新窗口快速定位）
  - 必跑命令：完成后需要跑的命令（如果仓库有统一门禁也要写）
- 验收标准（DoD）：3–6 条可验证的 check list
- 回滚方案：一句话说明如何撤销/降级
- 交接提示（新窗口 Prompt）：给“另一个窗口/另一个代理”可直接执行的指令，包含目标、上下文入口、完成标准

## 拆分策略（必须遵守）

1) 先按“价值链”拆：domain（类型/契约）→ data（实现/适配）→ feature/model（业务逻辑）→ UI（展示/交互）→ tests（覆盖/回归）→ guard（门禁）
2) 每个任务只跨 1–2 个层级；跨层太多就拆
3) 把“必须串行的依赖”压到最少，把“可并行的任务”显式列出来
4) 优先形成可合并的最小闭环：先让主路径跑通，再做增强项
5) 把风险高的点前置：权限/数据迁移/接口兼容/回归面广

## 任务过大时的递归拆分规则

如果任务满足任一条件，视为过大，必须继续拆：

- 需要修改超过 8 个文件或超过 2 个模块
- 目标包含“以及/同时/另外/顺便”
- 验收标准无法在 5 条以内描述清楚
- 涉及两个以上相互独立的用户场景

继续拆分时，保持父子关系：

- 父任务：只保留“总目标 + 依赖 + 里程碑”，不再承载实现细节
- 子任务：承载所有实现细节与 DoD
- 依赖链路必须从父任务映射到子任务（确保上下游联系）

## 示例（输出片段）

### A. 总览

- 假设：已有现成的测试框架与统一门禁命令可用
- 依赖图：
  - T1 -> T3
  - T2 -> T3
  - T3 -> T4
- 可并行：{T1, T2}

### B. 原子任务清单（示例）

- ID：T1
- 标题：补齐 domain 类型契约
- 目标：新增并导出所需的 TypeScript 类型与接口，供 data/feature 使用
- 非目标：不改 UI 行为，不接入网络
- 依赖：无
- 影响面：domain 相关模块的编译与引用
- 变更点：类型/接口、单测
- 预估文件：app/domain/**, app/domain/__tests__/**
- 上下文包：
  - 必读入口：domain 的 public 出口文件与相关类型定义
  - 快速检索：export type, interface, domain, DTO
  - 必跑命令：npm test（或项目门禁对应命令）
- 验收标准（DoD）：
  - 类型可被上层引用且严格类型检查通过
  - 单测覆盖新增的关键映射/校验逻辑
- 回滚方案：回退类型导出与测试用例
- 交接提示（新窗口 Prompt）：在新窗口只聚焦完成 T1；先定位 domain public 出口与相关类型文件，补齐契约与单测，完成后跑对应测试/门禁并给出变更摘要

## 输出约束

- 不输出实现代码；只输出任务拆分与任务卡片
- 任务卡片必须可直接复制到 issue/ticket
- 不要写空泛建议；每条 DoD 必须可验证
