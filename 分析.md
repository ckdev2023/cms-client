## 总体评价

《新规经营管理签申请全套流程Markdown文档》是一份早期输入文档，覆盖了业务流程的核心骨架，但与 P0 数据模型存在多处实质性偏差，不能直接用于研发实现。

特别是数据设计方面，该文档仍停留在“单业务场景视角”，缺失 P0 要求的大量核心实体、架构约束和租户隔离机制。

## 一、状态机对比

### 问题：两套状态机模型不兼容

流程文档定义了 18 个平铺状态，P0 用 `S1-S9` 主阶段 + `post_approval_stage` 子阶段的两层状态机覆盖。

| 流程文档状态 | P0 对应映射 | 差异说明 |
| --- | --- | --- |
| `CONSULTING` | `Lead.status = new/following/pending_sign` | ⚠️ 流程文档混入 Lead 阶段，P0 此时尚未进入 Case |
| `CONTRACTED` | `Lead.status = signed` → 转化为 `Case.stage = S1` | ⚠️ P0 已签约后立即建案，不作为 Case 内部状态 |
| `WAITING_MATERIAL` | `Case.stage = S2` | ✅ 对应 |
| `MATERIAL_PREPARING` | `Case.stage = S3 / S4` | ⚠️ P0 将资料审核（S3）与文书制作（S4）分开，且 `S3 → S4` 需通过 Gate-A |
| `REVIEWING` | `Case.stage = S5 / S6` | ⚠️ P0 明确拆分“待校验”（S5）与“待提交”（S6），以及 Gate-B 校验机制 |
| `APPLYING` | `Case.stage = S7` | ✅ 对应 |
| `UNDER_REVIEW` | `Case.stage = S7` | ⚠️ P0 不区分 `APPLYING/UNDER_REVIEW`，统一为 `S7` |
| `NEED_SUPPLEMENT` | 不是独立状态 | ❌ P0 明确规定：补资料时案件保持在 `S7`，通过新 `SubmissionPackage(supplement)` 追踪，不回退状态 |
| `SUPPLEMENT_PROCESSING` | 不是独立状态 | ❌ 同上，P0 无此状态 |
| `APPROVED` | `Case.stage = S8`，`result_outcome = approved` | ✅ 对应 |
| `REJECTED` | `Case.stage = S8/S9`，`result_outcome = rejected` | ✅ 对应 |
| `WAITING_PAYMENT` | `post_approval_stage = waiting_final_payment` | ✅ 对应 |
| `COE_SENT` | `post_approval_stage = coe_sent` | ✅ 对应 |
| `VISA_APPLYING` | `post_approval_stage = overseas_visa_applying` | ✅ 对应 |
| `SUCCESS` | `post_approval_stage = entry_success` | ✅ 对应 |
| `VISA_REJECTED` | `post_approval_stage = overseas_visa_rejected` | ✅ 对应 |
| `RESIDENCE_PERIOD_RECORDED` | 不是状态，是事件 | ❌ P0 通过 `ResidencePeriod` 记录是否存在来判断，无此状态 |
| `RENEWAL_REMINDER_SCHEDULED` | 不是状态，是事件 | ❌ P0 通过 `Reminder` 记录生成来判断，无此状态 |
| `CLOSED_SUCCESS` | `Case.stage = S9`，`result_outcome = approved` | ✅ 对应 |
| `CLOSED_FAILED` | `Case.stage = S9`，`result_outcome = rejected` | ✅ 对应 |

**关键问题**

流程文档把“补正循环”建模为两个独立状态（`NEED_SUPPLEMENT → SUPPLEMENT_PROCESSING`），P0 明确规定案件在补正期间保持在 `S7`，每次补件创建 `submission_kind=supplement` 的新 `SubmissionPackage`。这是 P0 的强规则（§15.4）。

### 2.4 `Immigration Result`（入管结果）

P0 中，`ImmigrationResult` 不是独立实体。

流程文档定义了独立的 `ImmigrationResult` 实体，但 P0 的设计不同：

- `result` → 归集到 `Case.result_outcome`（`pending/approved/rejected/withdrawn`）
- `supplement_count` → 归集到 `Case.supplement_count_cached`（缓存值，统计 `SubmissionPackage` 中 `supplement` 类型数量）
- 补正过程通过创建新的 `SubmissionPackage(submission_kind=supplement)` 记录，不需要单独实体

## 三、完全缺失的核心实体

## 四、业务规则冲突

| 规则 | 流程文档口径 | P0 口径 | 冲突性质 |
| --- | --- | --- | --- |
| 补资料状态流转 | `UNDER_REVIEW → NEED_SUPPLEMENT → SUPPLEMENT_PROCESSING → UNDER_REVIEW`（循环） | 案件保持 `S7`，每次补件创建新 `SubmissionPackage(supplement)` | ❌ 硬冲突，P0 强规则不可改 |
| 提醒天数是否可配置 | “提醒时间点应可配置” | P0 明确：预置 `180/90/30` 天，不支持配置（后置 P1） | ❌ 硬冲突 |
| 资料收集的文件上传 | 隐含为直接上传文件 | P0 明确：仅登记 `relative_path`，不存文件本体 | ❌ 架构方向不同 |
| 收费字段设计 | `deposit_paid / final_payment_paid` 为原始布尔 | 为缓存字段（`_cached`），真实状态以 `BillingPlan/PaymentRecord` 为准 | ⚠️ 需要调整 |
| 退款处理 | 系统仅标记状态 | 一致 | ✅ |
| 提醒失败兜底 | 阻断结案 | 一致（P0 §11.1 强规则） | ✅ |
| Gate-A/B/C 校验 | 完全未提及 | P0 核心机制，`S3 → S4 / S4 → S5 /` 提交前三道门槛 | ❌ 关键遗漏 |
| `SubmissionPackage` 锁定不可覆盖 | 完全未提及 | P0 最关键不变规则（§2.4 / §13） | ❌ 关键遗漏 |

## 五、架构层面缺口

### 5.1 租户隔离（`org_id`）完全缺失

流程文档所有数据结构均无 `org_id`。P0 要求所有核心实体必须具备租户隔离字段，这是多租户 SaaS 的基础安全约束。

### 5.2 审计字段缺失

所有数据结构均缺少 `created_by`、`updated_by`、`created_at`、`updated_at`。P0 规范（数据模型设计 §1）强制要求所有核心实体具备这些字段。

### 5.3 `Lead` 层完全缺失

流程文档从“创建客户”开始，跳过了 `Lead`（咨询线索）阶段。P0 的 Survey 发送、报价确认、`source_type`、`visa_plan` 等字段在 P0 设计中都属于 `Lead`，转化时才流向 `Case`。

### 5.4 收费体系降级

P0 通过 `BillingPlan`（收费节点）+ `PaymentRecord`（回款记录）支持多节点分期、作废/冲正留痕、欠款风险确认等能力。流程文档简化为 2 个布尔位，无法支撑 P0 的收费管理体系。

## 七、建议

该文档可以作为经营管理签业务流程理解的参考素材，但不能直接作为 P0 数据模型的实现依据。建议如下：

- **状态机**：研发直接以 `P0/07-数据模型设计.md` 中的 `Case.stage (S1-S9) + Case.post_approval_stage` 枚举为准，流程文档的状态列表仅作为业务流程理解参考
