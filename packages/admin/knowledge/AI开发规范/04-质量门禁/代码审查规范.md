# 代码审查规范（Vue 3版）

## 为什么需要 Review

AI 生成代码经常能“跑起来”，但仍可能存在这些问题：

- 架构分层不合理
- 类型定义不稳定
- 状态边界遗漏
- UI 细节偏差
- 可维护性差

---

## Self Review 清单

### 功能

- [ ] 验收标准都已覆盖
- [ ] 关键状态都已验证
- [ ] 手动操作主流程无异常

### 代码质量

- [ ] 无明显重复逻辑
- [ ] 组件没有承载过多职责
- [ ] 类型定义清晰
- [ ] 错误处理完整

### 架构

- [ ] 页面不直接耦合请求细节
- [ ] service 不耦合 UI 文案
- [ ] 局部状态没有错误上升为全局状态

### 体验

- [ ] 按钮有禁用态或 loading 态
- [ ] 异常文案可读
- [ ] 空态和权限态明确

---

## Reviewer 检查点

1. 需求是否真的被实现，而不是只“看起来像实现了”。
2. 状态流转是否完整。
3. 组件、store、service 是否分层合理。
4. 是否容易继续扩展和维护。
5. 是否缺少必要测试。

---

## PR 模板建议

```markdown
## Summary

- 实现帖子列表页的 loading / empty / error / success 四态
- 新增 store 测试与组件测试

## Test Plan

- [x] pnpm typecheck
- [x] pnpm lint
- [x] pnpm test
- [x] pnpm build
- [x] 手动验证列表刷新

## Screenshots

- 正常态
- 空态
- 错误态
```

---

## 评论模板

### 严重问题

```text
[Critical] 当前错误态只显示 toast，没有稳定的页面兜底，刷新失败后用户无法继续操作。
建议补 error state 组件，并增加重试入口。
```

### 一般建议

```text
[Suggestion] 这里的筛选逻辑已经超过组件职责，建议提取到 composable，便于复用和测试。
```

### 询问

```text
[Question] 这里为什么需要放到 Pinia？如果只是当前页面使用，是否保留在 composable 更合适？
```

---

## 原则

1. Review 先找风险，再谈风格。
2. 评论要指出影响，不只指出“看着不顺眼”。
3. 小 PR 比大 PR 更容易保证质量。

---

**记住：Review 的目标不是挑刺，而是防止问题进入主干。**
