# 前后端分离测试策略（Vue 3版）

## 核心理念

**Mock 后端，优先验证前端行为与状态流转。**

```text
不要让前端开发依赖真实后端进度
不要让测试依赖真实网络
先稳定页面，再替换真实接口
```

---

## 三层测试策略

### Layer 1：组件测试（70%-80%）

目标：

- UI 渲染
- 用户交互
- 状态切换

做法：

- 用 `Vue Test Utils` 挂载组件
- 用 `props` / `stub` / `mock store` 控制输入

---

### Layer 2：状态与服务单元测试（15%-20%）

目标：

- Pinia store 状态流转
- composable 行为
- service 数据转换和错误处理

做法：

- Mock `request` 返回值
- 不依赖真实 DOM

---

### Layer 3：E2E / 联调测试（5%-10%）

目标：

- 验证关键业务流程
- 验证真实接口对接

做法：

- 本地 Mock 环境先通
- 后端准备好后再跑联调用例

---

## 推荐 Mock 方案

### 接口层：MSW

```ts
import { http, HttpResponse } from "msw";

export const handlers = [
  http.get("/api/posts", () => {
    return HttpResponse.json({
      code: 0,
      data: [{ id: "1", title: "Mock Post" }],
    });
  }),
];
```

### Store / Composable 层：Vitest Mock

```ts
vi.mock("@/modules/post/services/postService", () => ({
  fetchPosts: vi.fn().mockResolvedValue([{ id: "1", title: "Mock Post" }]),
}));
```

## 目录建议

```text
src/modules/post/
  components/
  composables/
  services/
  stores/
  __tests__/

tests/
  e2e/
  msw/
```

---

## 开发阶段建议

### Phase 1：前端独立开发

1. 定义类型
2. 定义 service 接口
3. 准备 Mock 数据
4. 写组件测试与 store 测试
5. 实现页面

### Phase 2：后端联调

1. 替换 service 中的 Mock
2. 跑联调用例
3. 修正字段映射和异常处理
4. 保持组件测试不变

---

## 必测状态

- [ ] loading
- [ ] empty
- [ ] error
- [ ] success
- [ ] retry
- [ ] permission denied

---

## 原则

1. 组件不直接依赖真实 API。
2. service 不直接耦合视图文案。
3. Mock 数据结构要尽量贴近真实返回。
4. 联调问题先修 service 映射层，不要把脏逻辑扩散到组件。

---

**记住：组件测试稳定，联调才会高效。**
