### 1.2 导航返回规则

#### 1.2.1 全局入口 → 收费列表页

| 场景 | 起点 | 目标 | 返回行为 | 状态保留 |

| 导航进入 | 左侧主导航「收费与财务」 | `billing/index.html` 收费列表 | — | 初始状态（默认排序、筛选重置） |
| 列表→案件详情 | 收费列表行的「案件名称」链接 | `../case/detail.html?id=…` | 浏览器后退或面包屑返回收费列表 | **保留**进入前的筛选条件（回款状态 / Group / 负责人 / 搜索词）、当前分段视图（案件收费 / 回款流水）、滚动位置 |
| 列表→收费计划面板 | 收费列表行点击展开 `billing-plan-panel` | 页面内面板展开 | 关闭面板 | **保留**列表筛选与滚动位置 |

#### 1.2.2 仪表盘入口 → 收费列表页

| 场景 | 起点 | 目标 | 返回行为 | 状态保留 |
|------|------|------|---------|---------|
| 仪表盘跳转 | 仪表盘「待回款案件」卡片/行动作「登记回款」 | `billing/index.html`（理想情况预筛"未回款 + 逾期"） | 浏览器后退回仪表盘 | 仪表盘自身状态由仪表盘管理 |

#### 1.2.3 案件详情收费 Tab → 弹窗/面板

| 场景 | 起点 | 目标 | 返回行为 | 状态保留 |
|------|------|------|---------|---------|
| 登记回款 | 案件详情「收费」Tab 内「登记回款」按钮 | `payment-modal` overlay（在案件详情页内） | 保存或取消后关闭弹窗，**回到该案件的收费 Tab** | **保留**案件上下文（案件 ID、Tab 选中状态、收费计划面板展开状态） |
| 上传凭证 | 案件详情「收费」Tab 内回款行「上传凭证」 | 凭证上传控件（弹窗内或行内展开） | 上传完成/取消后关闭，**回到该案件的收费 Tab** | **保留**案件上下文 |
| 创建催款任务 | 案件详情「收费」Tab 内逾期节点行「创建催款任务」 | toast 反馈（同页操作） | 无页面跳转 | **保留**案件上下文 |
| 风险确认 | 案件详情「收费」Tab 内欠款风险确认入口 | `risk-ack-panel`（同页面板） | 确认后面板更新为已确认状态 | **保留**案件上下文 |

#### 1.2.4 筛选条件保留规则（生产行为）

以下规则为**生产目标行为**，原型仅演示跳转路径：

- **保留范围**：回款状态筛选值、Group 筛选值、负责人筛选值、搜索关键词、当前分段视图（案件收费列表 / 回款流水记录）
- **保留时机**：从收费列表跳转到案件详情后再返回时
- **不保留**：页面完全刷新（F5 / 重新从导航进入）时恢复默认状态
- **实现策略**（生产参考）：URL query params 或 sessionStorage 存储筛选快照，返回时恢复

#### 1.2.5 滚动位置恢复规则（生产行为）

- **保留时机**：从收费列表行跳转到案件详情后再返回时，列表页应恢复到离开时的滚动位置
- **不保留**：筛选条件变更、分段视图切换时滚动位置重置为顶部
- **实现策略**（生产参考）：`scrollRestoration` 或手动存储 `scrollTop`

> **原型演示说明**：原型中返回行为通过简单 `<a>` 导航实现，不做滚动位置 / 筛选条件 / 分段视图保持。仅演示跳转路径正确性与面包屑/返回链接的存在。弹窗关闭后页面保持不跳转（弹窗本身即在当前页面内）。

### 1.3 旧入口兼容

旧 `billing.html` 保留为过渡入口与回归对照，新模块入口为 `billing/index.html`。详见 [INVENTORY.md §12](./INVENTORY.md#12-旧入口兼容策略决策记录)。

## 3 默认排序与展示规则

### 3.1 默认排序

- 逾期置顶
- 其次按未收金额降序
- 再按下一收款节点到期日升序

### 3.2 逾期展示

- 回款状态列显示"逾期"，以风险色（`tag-red`）强调
- "下一收款节点"列展示"逾期 X 天（到期日）"，行背景使用红色高亮 `bg-[rgba(220,38,38,0.04)]`

### 3.3 金额展示

- 应收/已收/未收统一货币格式与右对齐方式，便于快速扫读
- 已收金额使用绿色 `text-green-600`；未收金额逾期时使用红色 `text-red-600`，非逾期使用橙色 `text-orange-600`；结清为 0 时使用灰色 `text-gray-400`

## 4 列表页搜索与筛选

### 4.1 搜索

- 范围：案件名称 / 客户名称 / 案件编号
- 控件：`search-input`，位于筛选栏

### 4.2 筛选器

| # | 筛选维度 | 控件类型 | 默认值 |
|---|---------|---------|-------|
| 1 | 回款状态 | select（已结清/部分回款/未回款/逾期/全部） | 全部 |
| 2 | Group | select | 全部 |
| 3 | 负责人 | select | 全部 |

> **P0 不做「按金额范围」筛选**——仅做上述 3 个维度。

### 4.3 重置

- 筛选栏末尾「重置」按钮，清空所有筛选条件

## 7 收费计划（案件维度核心区块）

### 7.1 展示

- 收费节点列表：阶段绑定、应收金额、到期日、状态
- 下一收款节点摘要：最早一个"未结清"节点（名称/阶段、到期日、该节点应收金额）
- 若不存在未结清节点：展示"无（已结清）"

### 7.2 配置入口

- 案件详情「收费」Tab
- 空状态时：主按钮「配置收费计划」
- 已存在计划且具备权限时：次按钮「编辑收费计划」

### 7.3 配置表单最小字段

| # | 字段 | 必填 |
|---|------|------|
| 1 | 收费节点名称（或阶段/里程碑） | **是** |
| 2 | 应收金额 | **是** |
| 3 | 到期日 | **是** |
| 4 | 备注 | 否 |

### 7.4 变更与留痕

- 计划的创建/编辑/删除需写入审计日志（操作人、时间、变更前后）
- 权限：管理员/财务可编辑全部；主办人仅可编辑其负责案件的计划；助理只读
- 已存在"有效回款记录"时：允许新增节点；对已结清节点**禁止**修改应收金额与到期日

| 1 | 金额 | number input | **是** | |

### 9.2 关联 BillingPlan 交互规则

- 选择器仅展示"未结清节点"：节点名称/阶段、到期日、该节点应收金额、该节点未收金额

- 金额校验（提示模式，P0 不强制阻断）：

## 11 回款更正

### 11.1 规则

- 允许对回款记录进行"作废/冲正"并留痕（原因必填，记录操作人、操作时间）
- 作废/冲正仅影响汇总口径（已收/未收）与节点结清状态，不删除原始记录
- 作废/冲正后的回款记录在列表中标识状态（valid / voided / reversed）

### 11.2 记录状态枚举

| 枚举值 | 说明 | 参与汇总 |
|--------|------|---------|
| `valid` | 有效回款 | 是 |
| `voided` | 作废 | 否 |
| `reversed` | 冲正（指向被冲正的原记录） | 否 |

## 12 金额口径与状态机

### 12.1 金额口径（案件维度）

- **应收金额**：该案件所有收费节点应收金额之和
- **已收金额**：该案件所有"有效"回款记录金额之和（不含已作废/冲正记录）
- **未收金额**：应收金额 - 已收金额（下限为 0）

### 12.2 回款状态口径（案件维度）

按优先级判定：

| 优先级 | 状态 | 判定条件 |
|--------|------|---------|
| 1（最高） | **逾期** | 存在任一收费节点到期日 < 今天，且该节点未结清 |
| 2 | **已结清** | 未收金额 = 0 |
| 3 | **部分回款** | 0 < 已收金额 < 应收金额 且不满足"逾期" |
| 4 | **未回款** | 已收金额 = 0 且不满足"逾期" |

### 12.3 收费节点状态枚举（BillingPlan.status）

| 枚举值 | 状态 | 对应规格 |
|--------|------|---------|
| `due` | 应收 | 03 §3.5 |
| `partial` | 部分回款 | 03 §3.5 |
| `paid` | 已结清 | 03 §3.5 |
| `overdue` | 欠款 | 03 §3.5 |

### 12.4 状态枚举对齐

| billing-config.js | P0 规格 §5.2 | case-detail-config.js | 数据模型 §3.20 |
|--------------------|-------------|----------------------|---------------|
| `overdue: '逾期'` | 逾期 | `arrears: '欠款'` | `overdue` |
| `paid: '已结清'` | 已结清 | `paid: '已结清'` | `paid` |
| `partial: '部分回款'` | 部分回款 | `partial: '部分回款'` | `partial` |
| `unpaid: '未回款'` | 未回款 | `unpaid: '应收'` | `due` |

> **统一规则**：`billing-config.js` 以 P0 规格 §5.2 为 label 权威（逾期/已结清/部分回款/未回款），以数据模型 §3.20 为 key 权威（overdue/paid/partial/due）。case-detail-config.js 的 `BILLING_STATUS` 需在跨模块对齐时同步更新。

### 12.5 下一收款节点

- 最早一个"未结清"的收费节点（展示节点名称/阶段、到期日、该节点应收金额）
- 若不存在未结清节点：展示"无（已结清）"

## 13 催款与风险留痕

### 13.1 催款任务生成规则

- 触发来源：收费列表批量动作 / 案件收费 Tab 单笔动作（结果一致）
- 触发条件：存在"逾期"的收费节点
- 生成结果：创建一条 `source_type=billing` 的催款跟进任务，并发送站内提醒
- 指派规则：
  - 优先：案件负责人
  - 兜底：该 Group 默认负责人
  - 若仍无可用负责人：本次创建记为失败/跳过，原因「无可用负责人」

### 13.2 催款任务去重策略

- 同一案件 + 同一收费节点，若存在 `source_type=billing` 的未完成任务（todo / doing），则不重复生成
- 去重键：`case_id + billing_plan_id + overdue_cycle_start`（或等价口径）

### 13.3 欠款风险确认

- 触发场景：案件存在欠款且需继续提交时（Gate-C 软性提示，[03 §4.3](../../../../docs/gyoseishoshi_saas_md/P0/03-业务规则与不变量.md#43-三道-gate)）
- 确认字段：确认人（`billing_risk_acknowledged_by`）、确认时间（`billing_risk_acknowledged_at`）、原因码（`billing_risk_ack_reason_code`）、原因备注（`billing_risk_ack_reason_note`）、凭证（可选）
- **持久化位置**：上述 4 个字段存储在 **Case 实体**上（[07 §3.5](../../../../docs/gyoseishoshi_saas_md/P0/07-数据模型设计.md#35-case案件)），非独立表。每次风险确认覆盖写入 Case 行，同时创建 AuditLog 记录
- 留痕：写入案件时间线与审计日志（`action_type: 'billing_risk_acknowledged'`，`object_type: 'case'`，`before_data` / `after_data` 包含 4 字段变更前后值）
- 展示：风险确认面板展示确认人、原因与凭证摘要

#### 13.3.1 页面入口

| 入口 | 页面 | 行为 |
|------|------|------|
| 财务主页 | `billing/index.html` 风险确认面板 | 查看已确认的欠款风险记录与审计摘要（只读展示） |
| 案件详情收费 Tab | `case/detail.html` 收费 Tab | 案件有欠款时展示风险确认入口；可执行确认操作 |
| Gate-C 提交检查 | `case/detail.html` 校验与提交 Tab | 提交时检测到欠款且 `billing_risk_acknowledged_at` 为空 → 报 Gate-C warning 并引导至收费 Tab 完成确认 |
| 仪表盘风险聚合 | `dashboard/index.html` | 未确认欠款案件聚合为风险案件卡片（[03 §6](../../../../docs/gyoseishoshi_saas_md/P0/03-业务规则与不变量.md#6-收费与欠款策略)） |

#### 13.3.2 Gate-C 集成规则

- Gate-C 执行时读取 Case 上的 `billing_risk_acknowledged_at` 字段
- 若案件存在欠款（`billing_unpaid_amount_cached > 0`）且 `billing_risk_acknowledged_at IS NULL`，产生软性提示（warning，不阻断）
- 用户可选择先去收费 Tab 完成风险确认再回来提交，也可直接提交（P0 不硬阻断，`gate_effect_mode` 仅 `off / warn`）
- 确认后 Gate-C 下次执行时不再报该 warning

> **P0 策略**：欠款异常以风险提示和工作台暴露为主，不统一阻断提交。`gate_effect_mode` 仅允许 `off / warn`，不支持 `block`。

## 14 列表页批量动作

| # | 动作 | 约束 |
|---|------|------|
| 1 | 批量生成催款任务 | 需至少选中 1 条逾期案件 |

### 14.1 批量选择机制

- 表头全选 checkbox（支持 indeterminate 态）
- 每行独立 checkbox（已结清行 checkbox `disabled` + `opacity-60`）
- 选中计数显示 + 清除按钮
- 批量操作栏仅在 selectedCount > 0 时可见

### 14.2 批量催款权限约束

- 仅对"当前用户具备创建催款权限"的案件生效
- 选中案件中若存在无权限项，需提示并跳过
- 同一案件在同一收费节点的逾期周期内，需避免重复生成催款任务（去重口径见 §13.2）

### 14.3 批量结果反馈

批量执行后返回"成功/跳过/失败"三段式结果：

| 段 | 说明 |
|----|------|
| 成功 | N 条催款任务已创建 |
| 跳过 | M 条已跳过（含各跳过原因统计） |
| 失败 | K 条创建失败 |

### 14.4 跳过/失败原因枚举

| # | 原因 | 说明 |
|---|------|------|
| 1 | 无权限 | 当前用户不具备该案件催款权限 |
| 2 | 已存在未完成催款任务 | 同案同节点同逾期周期 |
| 3 | 不满足逾期条件 | 节点已结清或到期日未到 |
| 4 | 无可用负责人 | 案件负责人缺失且 Group 未配置默认负责人 |
| 5 | 创建失败 | 系统错误/并发冲突 |
