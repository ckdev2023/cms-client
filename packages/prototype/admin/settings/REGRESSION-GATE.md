> 跨模块依赖详述：[MIGRATION-MAPPING.md §7](MIGRATION-MAPPING.md#7-跨模块依赖说明)

## Gate 1：无 Group（空状态）

**场景**：事务所尚未创建任何 Group，或当前筛选条件下无匹配结果。

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1.1 | 列表区域显示空状态占位 | `#groupEmptyState` 可见，表格行数为 0 |
| 1.2 | 空状态标题 | "尚未创建任何 Group" |
| 1.3 | 空状态引导按钮 | "创建第一个 Group"按钮可见且可点击 |
| 1.4 | 引导按钮触发 | 点击后弹出新建 Group 弹窗（`#groupNameModal`） |
| 1.5 | 筛选无结果 | 选择"启用"筛选而所有 Group 均已停用时，列表清空、空状态展示 |
| 1.6 | 筛选恢复 | 切回"全部"后列表恢复 |

**涉及脚本**：`settings-group-list.js`、`settings-group-actions.js`

## Gate 2：停用已引用 Group

**场景**：管理员尝试停用一个已被客户/案件引用的 Group。

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 2.1 | 停用按钮在详情面板可见 | 仅当 `group.status === 'active'` 时显示 |
| 2.2 | 有引用的 Group：确认弹窗 | `#groupDisableModal` 弹出，`#disableModalRefDetail` 显示引用计数 |
| 2.3 | 引用计数文案 | "该 Group 已被 X 个客户 / Y 个案件引用。" |
| 2.4 | 引用说明 | "停用后不可再选择该 Group，已关联对象不受影响。" |
| 2.5 | 无引用的 Group：简单确认 | `#disableModalSimpleMsg` 显示，`#disableModalRefDetail` 隐藏 |
| 2.6 | 确认后状态变更 | Group 状态变为"停用"，列表行 badge 变灰 |
| 2.7 | 停用后详情面板 | 停用按钮隐藏（已停用 Group 不可再次停用） |
| 2.8 | 列表刷新 | 停用后列表自动刷新，反映最新状态 |
| 2.9 | Toast 弹出 | "Group 已停用（示例）" |
| 2.10 | 不可物理删除 | 详情面板与列表均不出现"删除"按钮 |

**涉及脚本**：`settings-group-actions.js`、`settings-group-detail.js`、`settings-group-list.js`

**跨模块影响**（生产代码需验证，原型暂不联动）：

| 依赖方 | 影响 | 原型中的对应 |
|--------|------|------------|
| 客户新建/编辑 → Group 下拉 | 停用 Group 不出现在可选项 | 原型暂未联动 |
| 案件新建 → Group 选择 | 同上 | 原型暂未联动 |
| 客户/案件详情 → 已关联的停用 Group | 仍显示名称 + "(已停用)" 标记 | 原型暂未联动 |

> 详见 [MIGRATION-MAPPING §7.4](MIGRATION-MAPPING.md#74-停用-group--其他模块选择列表门槛)

## Gate 3：根目录未配置

**场景**：管理员尚未在系统设置中配置本地资料根目录。

### 3A. 设置页内部

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 3A.1 | 未配置警告 | `#storageNotConfiguredHint` 在根目录为空时可见 |
| 3A.2 | 警告文案 | "配置根目录后，案件/资料中心的「登记资料（本地归档）」功能才可正常使用。未配置期间该入口将被禁用。" |
| 3A.3 | 路径预览为空 | `#pathPreview` 显示占位而非拼接结果 |
| 3A.4 | 保存验证 | 根目录名称和路径均为必填，缺失时保存失败并提示 |
| 3A.5 | 配置后警告消失 | 填入名称+路径后保存，`#storageNotConfiguredHint` 隐藏 |

**涉及脚本**：`settings-storage-root.js`

### 3B. 跨模块影响（案件模块）

| # | 检查项 | 通过标准（生产代码） | 原型状态 |
|---|--------|-------------------|---------|
| 3B.1 | 案件详情 → 资料 Tab → "登记资料（本地归档）"按钮 | 按钮置灰（`disabled`） | 原型暂未实现，登记为回归项 |
| 3B.2 | 置灰提示文案 | "本地资料根目录尚未配置，请联系管理员在「系统设置 → 本地资料根目录」中完成配置" | 同上 |
| 3B.3 | 案件详情 → 已有本地资料行的路径列 | 仅显示 `relative_path`，不拼接完整路径 | 原型暂不联动 |
| 3B.4 | 路径列标注 | 追加灰色标注："(根目录未配置)" | 同上 |

### 3C. 跨模块影响（资料中心模块）

| # | 检查项 | 通过标准（生产代码） | 原型状态 |
|---|--------|-------------------|---------|
| 3C.1 | 资料中心 → "登记资料（本地归档）"按钮 | 按钮置灰 | 原型暂未实现，登记为回归项 |
| 3C.2 | 置灰提示文案 | 同案件模块 | 同上 |
| 3C.3 | 资料中心 → 已有资料行的完整路径预览 | 仅显示 `relative_path` | 原型暂不联动 |
| 3C.4 | 路径列标注 | 追加灰色标注 | 同上 |

### 3D. 生产代码检查点

| 检查点 | 位置 | 说明 |
|--------|------|------|
| 判定条件 | `OrgSetting.localFileRootPath` | `null`、空字符串或纯空白 → 视为"未配置" |
| 案件模块检查 | `useDocumentTabViewModel` | mount 时读取 `OrgSetting`，derive `isStorageRootConfigured` |
| 资料中心检查 | `useDocumentsViewModel` | 同理 |
| 跨模块协作方式 | `domain/settings/OrgSetting.ts` + `SettingsRepository` | 不直接依赖 `features/settings/`，通过 domain 接口读取 |
| 解除禁用时机 | 管理员配置根目录后 | 案件/资料中心下次加载时自动解除，非实时推送 |

> 详见 [MIGRATION-MAPPING §7.1](MIGRATION-MAPPING.md#71-根目录未配置--案件资料中心门槛)

## Gate 4：权限不足（非管理员角色）

**场景**：非管理员角色（主办人/助理/销售/财务）尝试访问系统设置。

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 4.1 | 导航隐藏 | 非管理员角色时，侧边栏和移动端导航中"系统设置"项不可见 |
| 4.2 | 角色矩阵声明 | `data/settings-config.js` 中 `ROLE_MATRIX` 完整列出 5 个角色的可见/可编辑/可导出权限 |
| 4.3 | 管理员可见可编辑 | `ROLE_MATRIX[0]` → `{ role: '管理員', visible: true, editable: true }` |
| 4.4 | 主办人不可见 | `ROLE_MATRIX[1]` → `{ visible: false }` |
| 4.5 | 助理不可见 | `ROLE_MATRIX[2]` → `{ visible: false }` |
| 4.6 | 销售不可见 | `ROLE_MATRIX[3]` → `{ visible: false }` |
| 4.7 | 财务不可见 | `ROLE_MATRIX[4]` → `{ visible: false }` |
| 4.8 | 直接 URL 访问 | 生产代码需拦截非管理员直接访问 `settings/` 路径（原型不实现） |

**涉及文件**：`data/settings-config.js`（角色矩阵声明）、shared shell 导航（生产代码条件渲染）

> 参考：[P0-CONTRACT §8 权限与可见性](P0-CONTRACT.md#8-权限与可见性)

## Gate 6：跨组审计链边界

**场景**：确认系统设置页不越界承担跨组业务动作的审计职责。

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 6.1 | 可见性面板跨模块说明 | 面板底部显示蓝色提示："此处仅控制配置开关。跨组建案/转组/线索转化改组时的原因采集与审计留痕属于业务流程域，不在此页面管理。" |
| 6.2 | `settings-config.js` 声明 | `CROSS_MODULE_NOTES.visibilityAudit` 文案与面板一致 |
| 6.3 | 无原因采集 UI | 设置页面不存在"变更原因"输入框或跨组操作审批流 |
| 6.4 | 无审计日志查看 | 设置页面不提供审计日志查看入口（P0 不做） |
| 6.5 | MIGRATION-MAPPING 声明 | §7.2 明确跨组审计归属各业务模块 |
| 6.6 | 外部依赖任务已登记 | "跨组建案/转组/线索转化改组的原因采集与审计留痕"已作为独立跨模块任务登记，不与 settings 开发合并 |

**涉及文件**：`sections/visibility-config-panel.html`、`data/settings-config.js`、`MIGRATION-MAPPING.md`

### 外部依赖任务清单

以下任务不属于 settings 模块，需由各归属模块独立实现：

| 任务 | 归属模块 | 触发条件 | settings 提供的输入 |
|------|---------|---------|-------------------|
| 跨组建案原因采集 + 审计 | 案件模块 | `OrgSetting.allowCrossGroupCaseCreateFlag === true` 时跨组建案 | 仅提供开关值 |
| 客户转组原因采集 + 审计 | 客户模块 | 管理员修改客户归属 Group | 无直接输入 |
| 线索转化改组原因采集 + 审计 | 线索模块 | 线索转化时改变归属 Group | 无直接输入 |
| 停用 Group → 下拉排除 | 客户/案件/线索模块 | Group `status=disabled` | 仅提供 Group 状态 |
| 根目录未配置 → 登记入口禁用 | 案件/资料中心模块 | `OrgSetting.localFileRootPath` 为空 | 仅提供配置值 |

> 详见 [MIGRATION-MAPPING §7.2](MIGRATION-MAPPING.md#72-跨组动作审计--客户案件线索流程域)

## 验证执行顺序

建议按以下顺序逐 Gate 验证：

1. **Gate 5**（导航文案统一）— 最快验证，打开页面即可确认
2. **Gate 1**（无 Group）— 需临时清空 demo 数据或使用筛选模拟
3. **Gate 7**（Group 动作完整性）— 覆盖主流程
4. **Gate 2**（停用已引用 Group）— 覆盖关键异常流
5. **Gate 3A**（根目录未配置 - 设置页内部）— 清空根目录字段测试
6. **Gate 4**（权限不足）— 验证角色矩阵声明
7. **Gate 6**（跨组审计链边界）— 确认边界文案和文档
8. **Gate 3B/3C**（根目录未配置 - 跨模块）— 确认文档记录完整，原型暂不联动
