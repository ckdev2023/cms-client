# 案件新建页 P0 拆分约束清单

> 权威来源：[docs/gyoseishoshi_saas_md/P0/06-页面规格/案件.md](../../../../docs/gyoseishoshi_saas_md/P0/06-页面规格/案件.md)
>
> 用途：作为 `packages/prototype/admin/case/create.html` 的唯一验收基线。

---

## 1 必保留区块

新增案件原型必须保留以下 8 个区块：

| # | 区块 | 说明 |
|---|------|------|
| 1 | 页面头部 | 面包屑、标题、P0 范围 chips |
| 2 | 转化来源上下文条 | 从 Lead / Customer 转案件进入时，显示来源对象与默认承接说明 |
| 3 | Stepper | 业务信息 / 关联人与资料 / 分派与期限 / 完成创建 |
| 4 | 模板选择 | 至少支持 `家族滞在`、`技人国` 两种模板 |
| 5 | 主申请人与关联人 | 主申请人选择、快速新建、追加关联人 |
| 6 | 资料清单模板 | 按模板展示资料项分组与必交口径 |
| 7 | 分派与初始设置 | 负责人、Group、预计提交期限、合同金额、自动动作 |
| 8 | 创建反馈 | Step 4 汇总、成功 banner、toast |

## 2 转化入口上下文

- 若 URL 带 `sourceLeadId` / `customerId` 等来源参数，页面头部下方必须显示来源上下文条
- 上下文条至少展示：`Lead` 来源、`Customer` 来源、默认承接说明
- 上下文条必须明确展示：Step 2 预填主申请人、Step 3 默认继承 `Customer.group`、以及 `REQ-P0-01` 的去重口径
- 从 Lead 转化进入且 Customer 已创建后：
  - Step 2 默认将该 `Customer` 回填为主申请人
  - Step 3 默认继承 `Customer.group` 作为 `Case.group`
- 上下文条必须明确：P0 不自动复用重复客户，若继续创建新客户仍需二次确认并留痕

## 3 建案基础字段

### 3.1 Step 1 必保留字段

| 字段 | 控件 | 必填 | 说明 |
|------|------|------|------|
| 模板 | 卡片选择 | 是 | `家族滞在` / `技人国` |
| 申请类型 | select | 是 | 认定 / 变更 / 更新 |
| 案件标题 | text input | 是 | 默认由主申请人 + 模板 + 申请类型生成，可手改 |

补充约束：

- Step 1 **不得**要求用户先决定 `Case.group`
- Step 1 需明确提示：`Group` 会在 Step 2 选定主申请人后，于 Step 3 默认继承并确认

### 3.2 Step 2 必保留字段

| 字段 | 控件 | 必填 | 说明 |
|------|------|------|------|
| 主申请人 | select / modal | 是 | 已有客户选择或快速新建 |
| 关联人 | 列表 + modal | 否 | 家族签用于配偶/子女；工作类可补充雇主联系人 |
| 资料模板 | 只读列表 + checkbox | 是 | 模板驱动生成，需区分必交/可选 |

补充约束：

- 若从转化链路进入且已携带 `customerId`，Step 2 必须预填主申请人
- 主申请人变更后，Step 3 的 `Case.group` 默认继承值必须同步更新

### 3.3 Step 3 必保留字段

| 字段 | 控件 | 必填 | 说明 |
|------|------|------|------|
| 归属 Group | select | 是 | 默认继承当前主申请人所属 Group，可人工改写 |
| 负责人 | select | 是 | 创建后案件归属主办人 |
| 预计提交期限 | date input | 是 | 用于列表风险/到期展示 |
| 合同金额 | number input | 是 | P0 仅记录总额，不展开收费节点 |
| 自动生成资料清单 | checkbox | 否 | 默认勾选 |
| 同步生成初始任务 | checkbox | 否 | 默认勾选 |
| 跨组建案原因 | textarea | 条件必填 | Group 非继承值时必须填写 |

补充约束：

- Step 3 需展示 `Group` 继承提示文案（例如：默认继承自主申请人 / 客户 Group）
- 若用户将 `Case.group` 改为非继承值，必须即时显示原因输入框
- `groupOverrideReason` 属于 `REQ-P0-01` 留痕范围

## 4 模板与批量模式

### 3.1 家族滞在模板

- 显示家族签批量建案说明卡
- 显示角色 chips：配偶 / 子女 / 扶养者 / 保证人
- 显示至少 2 条默认办理对象预览
- URL hash 为 `#family-bulk` 时默认进入家族模板，并保留批量模式提示

### 3.2 技人国模板

- 隐藏家族签批量说明卡
- 显示雇主信息沉淀说明
- 资料清单切换为主申请人 / 雇主 / 事务所内部三段

## 5 Modal 快速新建客户

### 4.1 字段

| 字段 | 必填 |
|------|------|
| 姓名 | 是 |
| 角色 | 是 |
| 所属 Group | 是 |
| 电话 | 电话/邮箱二选一 |
| 邮箱 | 电话/邮箱二选一 |
| 备注 | 否 |

### 4.2 规则

- 姓名为空时不可保存
- Group 为空时不可保存
- 电话与邮箱都为空时不可保存
- `mode=primary` 时回填为主申请人
- `mode=related` 时追加到关联人列表
- 若快速新建客户时命中电话/邮箱重复，沿用 `REQ-P0-01`：**仅提示，不自动复用已有 Customer**
- 用户选择继续创建时，必须进入二次确认态，并记录命中对象、操作者、确认时间、继续创建原因
- 二次确认态必须在 modal 内可见展示命中 Customer 摘要，不能只靠 toast
- modal 主按钮必须支持两段式动作：先展示重复命中确认态，再允许“确认继续创建并留痕”
- 从 Lead 转化进入且 Customer 已创建后，Step 2 默认将该 Customer 回填为主申请人，Step 3 默认继承其 Group 作为 Case.group

## 6 关键可用态

| 步骤 | 启用条件 |
|------|---------|
| 下一步（Step 1 → Step 2） | 模板 + 申请类型 + 案件标题完整 |
| 下一步（Step 2 → Step 3） | 已选择主申请人 |
| 下一步（Step 3 → Step 4） | 负责人 + 预计提交期限 + 合同金额完整；跨组时原因必填 |
| 创建案件 | Step 4 展示，沿用 Step 3 校验结果 |

## 7 状态与异常态

必须保留以下状态：

- 转化来源上下文态
- 家族签批量模式提示
- 技人国雇主信息说明态
- 无关联人空状态
- 成功创建 banner
- modal 校验错误提示
- modal 重复命中二次确认态

## 8 Toast / 反馈

| 场景 | title | desc |
|------|-------|------|
| 快速新建客户成功 | 客户已加入（示例） | 已回填到建案流程，可继续完善关联人与资料模板 |
| 创建案件 | 案件已创建（示例） | 已生成 Case 与资料登记清单草稿，可继续分派任务与补录资料 |

## 9 Demo-only 行为

以下行为必须保留，但需明确为原型演示：

- 模板、客户、负责人均为硬编码示例数据
- 创建案件仅显示 banner / toast，不真实落库
- 资料清单 checkbox 仅演示模板结构，不回写进度
- `#family-bulk` 仅切换默认模式，不真实批量创建多个 Case

## 10 P0 明确不做

- 不实现案件详情页 10 个 Tab 的真实创建后跳转
- 不执行 Gate-A/B/C 校验
- 不生成真实提交包、补正包、收费节点
- 不引入企业客户实体
- 不做案件草稿持久化
