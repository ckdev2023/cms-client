# Roles 目录说明

> 本目录按更接近真实 SaaS 公司的组织方式分组。
> 目标不是把每个文件都当成“一个部门”，而是区分：部门、岗位职能、外部顾问。

## 目录结构

### 01-产品与业务

- `01-产品与业务/产品设计负责人.md`
- `01-产品与业务/行政书士SaaS领域产品负责人.md`
- `01-产品与业务/个人AI开发该SaaS的最小角色配置表.md`

适用范围：

- 产品规划
- 业务规则
- MVP 边界
- 页面结构
- UX 与流程设计

### 02-工程研发

- `02-工程研发/高级Flutter架构师与高级开发专家.md`
- `02-工程研发/高级后端架构师与系统设计专家.md`

适用范围：

- 前端 / 客户端实现
- 后端架构
- 数据模型
- 工程化与交付

### 03-质量与安全

- `03-质量与安全/测试架构师与 QA 质量治理专家.md`
- `03-质量与安全/防御性攻击评估专家.md`

适用范围：

- QA
- 回归测试
- 风险评审
- 安全测试与攻击面检查

### 04-商业化与交付

- `04-商业化与交付/试点实施与客户成功负责人.md`
- `04-商业化与交付/增长销售与定价负责人.md`

适用范围：

- 首批试点
- 客户成功
- 销售推进
- 定价与商业化

### 05-外部顾问

- `05-外部顾问/日本资深行政书士与事务所实务专家.md`

适用范围：

- 行业实务校准
- 事务所 SOP 校验
- 窗口口径与现场风险判断

## 使用原则

1. 如果你要判断“做不做、先做什么”，先看 `01-产品与业务/`。
2. 如果你要落代码、改架构、定数据边界，看 `02-工程研发/`。
3. 如果你要评估回归风险、权限风险、安全风险，看 `03-质量与安全/`。
4. 如果你要推进试点、定价、销售、续费，看 `04-商业化与交付/`。
5. 如果问题涉及日本行政书士真实实务，不要只靠内部角色，必须参考 `05-外部顾问/`。

## 一句话说明

这套结构把角色库从“按专家类型平铺”调整成了“按真实公司部门分组，保留岗位文件”，更适合长期维护和调用。
