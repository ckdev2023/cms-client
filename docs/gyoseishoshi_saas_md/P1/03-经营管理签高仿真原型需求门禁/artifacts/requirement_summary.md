# Requirement Summary

## Business Goal

为后续“经营管理签 P1 高仿真原型”冻结业务边界、关键状态、门禁规则与验收口径，确保原型能够覆盖 Step 1–20 主链路，并保持 `P0 Case.stage = S1–S9` 与 P1 业务子步骤的分层不混用。

## User Story

作为行政书士、助理、负责人、财务与客户相关参与方，我需要一套可演示的高仿真原型，能够从咨询、问卷、报价、签约、建案、资料、文书、提交、补正、下签后收费、COE 发送、海外返签、在留期间记录到续签提醒与结案完整跑通经营管理签流程。

## In Scope
- 冻结经营管理签 Step 1–20 在高仿真原型中的页面承载与状态表达。
- 明确咨询、问卷报价、签约建案、资料清单、文书校验、提交入管、补正循环、下签后处理、在留期间与提醒的关键字段。
- 冻结高仿真原型必须体现的硬门禁与异常分支。
- 为后续 `packages/prototype/admin/leads-message`、`customers`、`case` 的高仿真设计提供需求基线。

## Out of Scope
- 真实后端接口、数据库持久化、真实文件上传、真实通知发送、真实定时任务。
- 非经营管理签签证类型的流程与页面。
- 退款金额试算、真实财务对账、真实 COE 文件流转。
- 生产代码实现、依赖安装、部署与联调。

## Business Rules
- P0 继续保留 `Case.stage = S1–S9` 管理层阶段；经营管理签专属流程通过 P1 业务子步骤表达。
- 问卷《2025M_C经管签信息表》回收后才能沉淀 `quote_price` 与 `visa_plan`，签约后才进入建案与资料清单阶段。
- 补资料流程可循环多次，每次必须记录 `supplement_count`。
- 下签后必须先收尾款，才允许推进 `COE_SENT`。
- 客户成功入境后必须记录新的在留期间；未记录时不得成功结案。
- 只要存在有效到期日，就必须生成 180 / 90 / 30 天提醒；提醒创建失败不得进入 `CLOSED_SUCCESS`。
- 所有路径最终必须收敛到 `CLOSED_SUCCESS` 或 `CLOSED_FAILED`。

## Edge Cases
- 问卷未回收或报价未确认时，不能演示为“已签约建案成功”。
- 材料不齐、审核退回或需要补资料时，流程停留在资料/补正路径，不得越过校验直接提交。
- 入管补正可能多轮往返，且必须在同一案件上下文中累计次数。
- 已下签但尾款未收齐时，发送 COE 必须呈现阻断态。
- 海外返签存在 `VISA_REJECTED` 分支，需直接收敛到失败结案。
- 已返签成功但缺少在留结束日时，不得生成提醒，也不得进入成功结案。

## Acceptance Criteria
- 高仿真原型可从咨询线索开始完整演示一条成功路径到 `CLOSED_SUCCESS`。
- 高仿真原型可演示入管拒签与海外返签拒签两条失败路径到 `CLOSED_FAILED`。
- 高仿真原型可演示至少一轮补正循环，并展示 `supplement_count`。
- 高仿真原型可直接看出签约前建案限制、尾款门禁、在留期间与提醒门禁。
- 原型演示只基于 mock / 本地状态，不依赖真实网络。

## Unknowns
- 无；原型阶段未决技术实现已通过 `Out of Scope` 收敛，不阻塞高仿真原型需求冻结。

## Risks
- 业务文档中的业务状态容易与 P0 `S1–S9` 混淆，若原型表达不清会误导后续实现。
- 若把“自动提醒”误画成真实系统已投递，可能掩盖仍需后端能力承接的事实。
- 若原型页面粒度偏离现有 `leads-message / customers / case` 承载方式，后续迁移成本会上升。
