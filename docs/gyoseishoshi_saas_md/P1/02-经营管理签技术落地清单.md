## 1. 落地原则

- **主状态机不加塞**：`Case.stage = S1–S9` 继续作为管理层；经管签细步骤走 P1 模板扩展。
- **业务逻辑放 model**：前端专属流程判断、按钮可用性、门禁提示放 `features/*/model`，不堆在页面组件中。
- **优先扩展现有真相源**：服务端优先对齐 `013_business_manager_visa` 已存在字段与 `residence_periods` 表；若新增字段，追加新 migration，不回写旧 migration 历史。
- **原型只作迁移输入**：`packages/prototype/admin/*/MIGRATION-MAPPING*.md` 用于确定 domain/data/model/ui 落点，不作为运行时代码。

## 3. 服务端开发任务清单

### 3.1 Lead / Intake 入口承接（M1–M2）

- 扩展 `packages/server/src/modules/portal/leads/*`
  - 标记经营管理签咨询类型
  - 固化从 `Lead → Customer → Case` 的承接字段
- 扩展 `packages/server/src/modules/portal/intake/*`
  - 承接问卷发送、回收、状态查询
  - 为 `survey_data` 提供结构化入口
- 对齐 `app.module.ts` 中已有 portal controller 注册方式

### 3.2 Case / Document 主链路扩展（M1–M5）

- 扩展 `packages/server/src/modules/core/cases/*`
  - 对外返回经管签模板字段、业务子步骤、补正计数、结果收敛信息
  - 对齐 `013_business_manager_visa.up.sql` 已有字段：`stage`、`post_approval_stage`、`visa_plan`、`close_reason`、`supplement_count`
- 扩展 `packages/server/src/modules/core/document-items/*`
  - 支持 `questionnaire` 类资料项
  - 问卷资料与普通资料统一参与完成率 / 审核 / 催办
- 扩展 `packages/server/src/modules/core/document-files/*`
  - 保持版本化附件能力
  - 为问卷产出物、理由书、补正附件提供同一归档口径

### 3.3 收费与 COE 门禁（M6）

- 扩展 `packages/server/src/modules/core/billing/{billingPlans,paymentRecords}.*`
  - 定义 `final_payment` 节点
  - 在 `billingGuards.ts` 中补 `COE_SENT` 前的 `block` 型守卫
- 扩展 `packages/server/src/modules/core/cases/*`
  - 未结清尾款时阻断推进 `COE_SENT`
  - 记录门禁失败原因，便于前端展示

### 3.4 在留期间与提醒（M8–M9）

- 复用 `packages/server/src/infra/db/migrations/013_business_manager_visa.up.sql` 已存在的 `residence_periods` 表
- 新增 `packages/server/src/modules/core/residence-periods/`
  - `controller` / `service` / `*.test.ts`
  - 查询当前在留、创建历史记录、切换 `is_current`
- 扩展 `packages/server/src/modules/core/reminders/*` 与 `core/jobs/handlers/*`
  - 基于 `valid_until` 生成 180 / 90 / 30 天提醒
  - 提醒创建失败时返回可审计错误，不自动标记成功结案

### 3.5 服务端契约同步点

- 若 `coreEntities.ts` / `documentEntities.ts` 与 API 返回不一致，必须同步更新类型与全部调用方。
- 若 `drizzle/schema.ts` 尚未完全反映 `013` 字段，必须补齐 schema 映射与校验。
- 若新增字段超出 `013`，必须新增 migration，不直接编辑已发布真相源。

## 4. 前端开发任务清单（packages/mobile）

### 4.1 新增 Lead / Customer 功能骨架（M1–M2）

- 依据原型迁移映射新增：
  - `packages/mobile/src/domain/lead/`
  - `packages/mobile/src/data/lead/`
  - `packages/mobile/src/features/lead/{model,ui}/`
  - `packages/mobile/src/domain/customer/`
  - `packages/mobile/src/data/customer/`
  - `packages/mobile/src/features/customer/{model,ui}/`
- 目标：让经营管理签能从咨询入口进入签约 / 转客户 / 转案件链路。

### 4.2 扩展 Case 域与 ViewModel（M1–M7、M9）

- 扩展 `packages/mobile/src/domain/case/*`
  - 增加经管签模板字段、业务子步骤、补正计数、COE/返签结果、结案原因
- 扩展 `packages/mobile/src/data/case/*`
  - `CaseApi` / repository 对齐服务端新增返回字段
- 扩展 `packages/mobile/src/features/case/model/*`
  - 管理按钮可用性、Step 视图、提交/补正/COE/返签门禁提示
- 扩展 `packages/mobile/src/features/case/ui/*`
  - 仅做渲染，不承接业务判断

### 4.3 问卷 / 资料 / 文书界面（M2–M4）

- 扩展 `packages/mobile/src/domain/documents/*`、`data/documents/*`、`features/documents/*`
  - 区分问卷类资料项与普通资料项
  - 展示审核状态、催办状态、补正批次
- 让案件详情页能看到：问卷结果、资料清单、文书版本、提交包与补正历史

### 4.4 提醒与在留期间（M8）

- 扩展 `packages/mobile/src/domain/reminder/Reminder.ts`
  - 从“固定 180/90/30 常量”升级到模板驱动展示口径
- 新增建议目录：`packages/mobile/src/domain/residence/`、`data/residence/`
- 在 case 相关 model 中展示当前在留、历史记录、续签提醒生成结果

### 4.5 容器与导航接入

- 同步 `packages/mobile/src/app/container/*` 注册新增 repository
- 同步 `packages/mobile/src/app/navigation/*` 注册 lead/customer/case 相关入口
- 继续通过 `shared/ui` 输出 UI，不让 feature 直接依赖 `tamagui`

## 5. 测试清单（必须项）

### 5.1 服务端

- 在对应模块补 `*.service.test.ts` / `*.controller.test.ts`
- 对 `COE_SENT` 门禁至少覆盖：尾款已结清 / 未结清 / 重复推进
- 对 `residence_periods` 至少覆盖：创建当前记录、历史切换、提醒生成失败
- 对问卷 / 补正至少覆盖：问卷资料项、补正批次关联、结果回写

### 5.2 前端

- 优先补 `model` / `domain` / `data` 单测
- ViewModel 至少覆盖：
  - 业务子步骤映射
  - 提交 / 补正 / COE 按钮禁用条件
  - 在留期间录入后提醒状态展示
- Repository / API 层必须 mock，禁止真实网络请求

### 5.3 门禁

- 涉及代码改动时，收尾顺序固定为：先 `npm run fix`，再 `npm run guard`
- 仅在对应包内做最小验证：`packages/server`、`packages/mobile`

   - 最后补在留期间、自动提醒与成功/失败结案收敛

## 7. 收敛结论

这份清单的执行口径应固定为：

- **服务端以 `packages/server` 的现有 core / portal 模块为主战场**
- **前端以 `packages/mobile` 的 domain / data / features / shared 分层为主战场**
- **原型目录只负责提供迁移映射，不直接承载生产实现**
- **经营管理签完整能力在 P1 落地，但必须持续复用 P0 已冻结的主对象与主流程**
