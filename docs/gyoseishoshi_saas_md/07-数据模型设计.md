本章以最新需求的主链路为基准：咨询转化 → 建档 → 收资料/催办 → 文书 → 校验/复核 → 提交快照 → 审理跟踪 → 结果归档 → 续签提醒 → 回款 → 留痕。

## 1. 设计原则

- 以案件为中心，但把"阶段（Stage）""门槛（Gate）""快照（Submission Package）"拆开落地，便于自动化与审计
- 客户资料可复用：一个客户可多案；多对象/多案件场景可共享材料"版本"，但引用必须可锁定
- 不引入"家庭/户"主数据实体：客户关系用于导航与复用（CustomerRelation），关键关系门槛用于案件质量控制与审计（CaseParty）
- 资料项是"要求"，附件是"版本"：每次上传生成新版本，不覆盖历史；提交包只能引用具体版本
- 文书与模板版本不可变：导出/提交引用固定版本，保证历史可追溯
- 客户可见域与内部域隔离：沟通、资料备注、校验结果需明确可见性与字段范围
- 全量关键事件可留痕：状态流转、资料审核、文书定稿、校验/复核、提交包生成、收费节点确认等

## 1.1 跨实体通用字段与治理约束

- 作为 SaaS 系统，核心业务主实体应统一具备 `org_id/workspace_id` 一类租户隔离字段，避免后期补租户成本过高
- 高频写入实体建议统一具备 `created_by`、`updated_by`、`created_at`、`updated_at`，必要时增加 `deleted_at` 或 `archived_at` 支持软删除/归档
- 涉及客户门户展示的实体应带可见性字段（如 `visible_scope`、`client_visible_flag`），不要依赖前端页面自行约束
- 编号类字段（lead_no、customer_no、case_no、submission_no）建议按组织内唯一生成，并保留人工可读格式
- Group 作为治理对象而非自由标签，建议具备组织内唯一名称/编号、停用状态与成员关系维护能力，不建议被物理删除

## 1.2 索引与唯一约束建议

- 唯一约束：`lead_no`、`customer_no`、`case_no`、`submission_no` 建议在组织内唯一；`company_no` 仅在启用企业主数据时需要组织内唯一
- 检索索引：客户的 `group_id/owner_user_id`、案件的 `group_id/stage/principal_user_id/next_action_due_at`、资料项的 `status/due_date`、任务的 `assignee_user_id/status/due_at`、提醒的 `recipient_id/remind_at/send_status`
- 工作台聚合索引（建议）：案件的 `latest_validation_status/has_blocking_issue_flag/next_deadline_due_at/billing_unpaid_amount`，便于快速聚合"风险/待提交/待回款/即将到期"
- 去重辅助：联系方式类字段建议保存规范化值（如 `normalized_phone`、`normalized_email`），配合姓名/生日进行重复提示
- 快照查询：提交包与附件版本建议按 `case_id + submitted_at`、`requirement_id + version_no` 建索引，保证时间线与追溯查询效率
- 协作与可见性：CaseCollaborator 建议按 `case_id + user_id` 建唯一约束，并按 `user_id + active_flag` 建索引用于"我参与的案件"查询

## 1.3 P0 策略落模原则

> 完整的 P0 版本策略与能力开关表见 [03 §1](03-业务规则与不变量.md#1-p0-版本策略冻结)。以下仅列出对数据模型设计有直接影响的要点：

- 模型层需要预留"可开启但不强制"的能力（如双人复核、客户门户、外部提醒），不应把这些写死为所有案件必选
- 会影响业务策略的门槛建议显式建模（如 `billing_gate_mode`、`review_required_flag`），避免把规则散落在前端或人工约定里
- P0 仅要求支撑 2 个高频模板闭环：家族滞在、工作签证（技人国）；模板模型先保证可扩展
- P0 把 Group 作为强依赖维度；工作类案件的雇主/受入机构信息可先作为 Case 字段沉淀

## 2. 核心实体总览（MVP）

- Lead（咨询线索）
- Group（归属团队/项目组）
- UserGroupMembership（用户与 Group 关系：用于最小可见范围与协作边界）
- Customer（个人客户）
- CustomerRelation（客户关系映射：配偶/子女/父母/代理人等，用于检索与快速跳转，不作为案件门槛口径）
- Company（企业客户，P1/P2 可选：如需企业主数据、企业侧协作与批量能力）
- ContactPerson（联系人/企业担当，P1/P2 可选）
- Case（案件）
- CaseCollaborator（案件协作成员：负责人/协作者/显式授权可见范围）
- CaseParty（案件关联人：主申请人/扶养者/保证人/家属/企业联系人等）
- CaseStageHistory（案件阶段流转记录）
- CaseTemplate（案件模板：生成资料清单/默认任务/默认校验规则集）
- DocumentRequirement（资料项：材料要求）
- DocumentFileVersion（资料附件版本：每次上传生成一条不可变版本）
- CommunicationLog（沟通记录：内部/客户可见分域）
- Task（任务：推动案件流转的驱动单元）
- CaseDeadline（期限：在留到期/补件截止/预约日等统一管理）
- Template（文书模板：版本化）
- GeneratedDocumentVersion（生成文书版本：初稿/审核中/定稿等）
- ValidationRun（提交前校验运行：结果分层，含硬性阻断/软性提示）
- ReviewRecord（双人复核：通过/驳回原因）
- SubmissionPackage（提交包：一次对外提交快照）
- SubmissionPackageItem（提交包锁定引用：资料版本/文书版本/关键字段快照）
- BillingPlan（收费计划/节点）
- PaymentRecord（回款记录）
- Reminder/Notification（提醒与触达记录：内部/客户）
- AuditLog（审计日志：全对象通用）
- User / Role（内部用户与角色；客户侧用户建议独立为 ClientUser）

## 4. 关键关系说明

- 所有主实体建议都携带 `org_id/workspace_id`，先天满足租户隔离与后续多组织扩展
- 一个 User 可以通过 UserGroupMembership 关联一个或多个 Group，其中一个可标记为主 Group
- 一个 Group 可以关联多个 Customer 与 Case，用于表达归属维度（团队/项目组），支撑筛选、协作与可见性口径
- 一个 Lead 可转化为一个 Customer（或匹配既有 Customer）并可创建一个或多个 Case；Lead.group_id 作为默认归属来源
- 一个 Customer 可以对应多个 Case；Case 默认继承 Customer.group_id 作为归属快照，若发生转组应记录审计日志并保留变更原因
- 若启用企业主数据：一个 Company 也可以对应多个 Case；否则工作类案件的雇主/受入机构信息建议先沉淀在 Case.employer_* 字段
- 一个 Case 可以有多个 CaseParty，用于表达多对象场景的关键关系；门槛校验依赖其完整性
- 一个 Case 可以有多个 CaseCollaborator，用于表达协作者与显式授权可见范围；负责人建议也落一条 principal 记录以便统一查询
- 一个 Case 可基于 CaseTemplate 生成多条 DocumentRequirement 与默认 Task
- 一个 DocumentRequirement 可以有多个 DocumentFileVersion；提交包只能引用具体版本
- 一个 Case 可以有多个 GeneratedDocumentVersion；提交包只能引用具体版本
- 一个 Case 可以有多条 ValidationRun 与 ReviewRecord；提交包应锁定当次引用的校验报告与复核结果
- 一个 Case 可以有多个 SubmissionPackage；补正属于新的 SubmissionPackage，并关联原包
- 一个 Case 可以有多个 CaseDeadline，并由 Reminder/Notification 产生提醒留痕
- BillingPlan 与 PaymentRecord 关联到 Case，并可与阶段/门槛联动（如提交前必须满足某收费节点）
