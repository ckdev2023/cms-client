# 05 核心流程与状态流转

## 1. 总体流程

1. 咨询录入
2. 客户建档
3. 正式签约
4. 案件创建
5. 资料清单生成
6. 客户提交资料
7. 内部审核资料
8. 缺件催办 / 补件循环
9. 文书生成
10. 提交前校验
11. 正式提交
12. 审理跟踪
13. 出结果
14. 归档
15. 续签 / 派生案件提醒

### 1.1 落地共识：阶段、门槛、快照
为了让流程能被系统自动化（而不是靠人记忆），需要把“流程步骤”落地为以下三类可执行机制：
- 案件阶段（Stage）：用于组织工作与看板统计，驱动默认任务/提醒
- 门槛校验（Gate）：用于阻断关键错误，集中在“进入文书/进入提交/提交”三道关卡
- 提交包（Submission Package）：用于固化每一次对外提交快照（材料版本、文书版本、关键字段、校验报告、复核签名）

### 1.2 案件阶段（Stage）建议
建议用少量阶段覆盖 80% 场景，并允许配置化扩展：
- S1 已建档（已签约/已立案）
- S2 资料收集中（对客户发出资料请求）
- S3 资料审核中（内部审核/退回补正）
- S4 文书制作中（模板生成、整理、审核、定稿）
- S5 待校验（进入提交前校验关卡）
- S6 待提交（校验通过，且满足复核要求（如启用）后等待对外提交）
- S7 已提交审理中（含补正/追加提交）
- S8 已出结果（许可/不许可）
- S9 已归档（结案）

阶段与关键动作的关系（原则）：
- 阶段用于“管理与协作”，不直接等于“资料是否齐备”
- 阻断由门槛校验负责；例如允许在 S2/S3 创建与流转，但进入 S4/S5 必须满足最低门槛

### 1.3 阶段进入门槛速查
- S1 已建档：已签约/已立案，主申请人、案件类型、负责人已明确
- S2 资料收集中：资料清单已生成并已发出至少一轮资料请求
- S3 资料审核中：已有客户上传资料，内部开始审核/退回补正
- S4 文书制作中：关键资料项达到最低可做文书门槛，开始生成或整理文书
- S5 待校验：关键文书已定稿，进入正式提交前校验
- S6 待提交：最新一次校验通过，且复核要求已满足
- S7 已提交审理中：已生成提交包并记录提交日期/回执
- S8 已出结果：已登记结果与结果日期
- S9 已归档：案件已结案，后续提醒/派生动作已确认

### 1.4 P0 默认策略口径
- 双人复核默认不是全局强制门槛；仅在事务所或案件模板明确开启时生效
- 收费节点默认用于提醒与风险提示，不默认阻断提交动作
- 客户门户默认不纳入首版主流程，P0 先保证内部用户可完成全链路
- P0 提醒以内部提醒为主；客户侧外部触达若启用，也应控制在最小能力范围
- 首版模板固定为：家族滞在、工作签证（技人国）

## 2. 咨询转案件流程

### 流程步骤
1. 新建咨询线索
2. 录入基础信息
3. 判断业务类型
4. 记录报价
5. 标记签约状态
6. 已签约后创建正式案件
7. 指派负责人
8. 自动生成资料清单和初始任务

### 状态建议
- 新咨询
- 跟进中
- 待签约
- 已签约
- 已转案件
- 已流失

补充规则（去重与转化）：
- 线索转客户：按电话/邮箱优先匹配，其次姓名+生日（或证件号）；支持人工确认合并
- 线索转案件：必须选择案件类型/签证类型、申请类型（认定/变更/更新）、主申请人、负责人；生成案件编号、资料清单与初始任务
- 已流失：不进入案件统计，但保留来源与原因用于报表

## 3. 资料收集流程

### 流程步骤
1. 系统按模板生成资料清单
   - 触发点：案件创建完成后，系统根据“签证类型 × 申请类型”匹配案件模板并生成通用资料项
   - 特殊需求：允许在案件内手动追加资料项，或标记某资料项无需提供（需原因留痕）
2. 客户上传资料
   - 上传后系统自动将该资料项更新为“已提交待审核”，并在清单中展示“已上传”标识与最新版本
3. 助理审核资料
4. 审核通过则进入已通过
5. 不通过则填写原因并进入补件状态
6. 系统触发催办提醒
7. 客户再次补交
8. 循环直到资料齐备

### 资料项状态
- 未发送
- 待提交
- 已提交待审核
- 审核通过
- 需补正
- 无需提供
- 已过期需重交

状态解释与推荐转移：
- 未发送：资料项已生成但尚未对客户发出请求
- 待提交：已请求客户提交（可触发催办）
- 已提交待审核：客户已上传最新版本，等待内部审核
- 审核通过：本资料项满足要求，可用于文书/提交包引用
- 需补正：审核不通过，必须填写原因并回到待提交
- 无需提供：因客户情况/申请策略变化而豁免（需记录豁免原因与操作人）
- 已过期需重交：超过时效或证件有效期不足，需要重传新版本

关键规则（版本与引用）：
- 资料项是“要求”，附件是“版本”；每次上传形成新版本，不覆盖历史版本
- 文书与提交包引用必须指向“具体版本”，保证审计成立
- 家庭/多案件共用材料：允许多个案件引用同一版本；提交后若需更新材料，应创建新版本并用于新的提交包/补正包

### 3.1 资料清单进度（完成率）口径
为了支持案件列表聚合展示与客户侧“清楚知道还差多少”，需要统一完成率口径：
- 默认口径：完成率 = 已审核通过的必交资料项数 / 必交资料项总数
- 分母剔除：标记为“无需提供/不适用”的资料项不计入总数，但必须记录原因与操作人
- 分组口径：支持按提供方/按类别输出子完成率（家庭场景优先用提供方）

### 3.2 资料项标准转移规则
- 未发送 → 待提交：对客户发出资料请求后进入可催办状态
- 待提交 → 已提交待审核：客户或内部上传了新的附件版本
- 已提交待审核 → 审核通过：审核通过并可用于文书/提交包引用
- 已提交待审核 → 需补正：审核不通过，必须填写补正原因
- 需补正 → 待提交：客户按意见重新上传后回到待提交/待审核链路
- 任一待处理状态 → 无需提供：因本案策略或申请条件变化被豁免，但需留痕原因
- 审核通过 → 已过期需重交：因证件有效期或材料时效不足而失效，进入重新收集流程

## 4. 文书制作流程

### 流程步骤
1. 资料达到最低门槛
2. 生成模板文书初稿
3. 助理整理
4. 行政书士审核
5. 修改迭代
6. 定稿
7. 进入提交前校验

### 文书状态
- 未生成
- 初稿
- 审核中
- 待修改
- 已定稿
- 已导出

补充规则（与校验/提交关系）：
- 进入提交前校验的最低门槛：关键资料项达到“审核通过”且关键文书为“已定稿”
- 文书版本不可变：导出与提交包应关联到某个版本，避免“导出后又改动”导致不一致

## 5. 提交前校验流程

### 校验项
- 主字段完整性
- 必需资料是否齐全
- 日期逻辑
- 有效期
- 费用节点
- 文书定稿状态
- 内部复核完成情况

### 结果状态
- 待校验
- 校验不通过
- 校验通过
- 已提交

### 5.1 校验分层与阻断口径
校验项建议分为两类，避免系统过度阻断影响效率：
- 硬性阻断：必填字段、关键关系（主申请人/扶养者绑定）、证件有效期、关键资料项齐备、文书已定稿
- 软性提示：风险项与建议补充（例如地址变更未确认、附件过旧但未过期、历史信息需二次确认、收费节点异常等）

### 5.2 双人复核（最小可行）
复核的目标是降低低级错误与争议风险，建议先做最小闭环：
- 复核触发：校验通过后进入待提交前
- 复核内容：校验报告 + 关键材料清单 + 文书版本号
- 复核结果：通过/驳回（驳回必须填写原因并回到“校验不通过”或“资料/文书处理中”）
- 默认口径：是否启用复核由事务所/案件模板决定；未启用复核时，校验通过即可进入 S6 待提交

### 5.3 提交包（Submission Package）
提交动作必须生成提交包，用于快照与追溯：
- 固化：提交日期、受理机关/窗口、提交凭证/回执
- 锁定：本次提交引用的资料版本、文书版本、关键字段快照、校验报告、复核记录
- 补正：收到补正通知后，创建“补正提交包”并关联原提交包，形成可追溯链路

### 5.4 校验、复核与提交的串联规则
- 提交前必须引用“最新一次有效校验结果”；若关键资料版本、关键字段或定稿文书发生变化，应重新校验
- 若启用双人复核，则“校验通过”并不等于“可提交”，还需复核通过后才能进入 S6 待提交
- 复核驳回后，应根据原因回退到 S3 资料审核中、S4 文书制作中或 S5 待校验，不应停留在 S6
- 无论首次提交还是补正提交，只要生成了新的对外提交，就必须创建新的提交包

## 6. 提交与审理流程

### 流程步骤
1. 标记提交日期
2. 上传提交凭证/回执
3. 案件进入审理中
4. 如收到补正通知则创建补正任务
5. 更新补正截止日
6. 再次提交补正材料
7. 等待结果
8. 记录许可/不许可结果

### 状态建议
- 待提交
- 已提交审理中
- 待补正
- 补正处理中
- 已许可
- 不许可
- 已归档

补充规则（补正闭环）：
- 待补正：由“补正通知”触发生成补正任务，并写入补正截止日
- 补正处理中：补正资料审核与文书更新完成后进入“待提交”，提交时生成补正提交包
- 审理中信息更新：支持记录来函、追加材料、面谈预约等事件，统一沉淀到案件时间线

### 6.1 异常结案/终止处理
- 对于客户撤回、改委托、长期中止等异常情况，建议不额外膨胀主阶段，而是记录结果状态、结案原因、归档原因与时间线事件
- 异常结案后应自动关闭未完成的内部任务，并决定是否保留或取消后续提醒
- 若未来重新启动，应基于原案件复制或派生新案件，而不是直接覆盖历史阶段与提交记录

## 7. 续签提醒流程

### 流程步骤
1. 系统识别在留到期日
2. 在 90 / 60 / 30 / 7 天节点触发提醒
3. 创建续签跟进任务
4. 联系客户确认是否委托
5. 如确认，则复制历史资料和基础信息创建新案件

### 续签价值
- 降低流失
- 提高复购
- 减少重复采集资料

补充规则（聚合与去重）：
- 同一扶养者/企业在留期限对应多个案件时，提醒在内部侧聚合展示，并提供“展开关联案件”
- 客户侧提醒默认仅发送主提醒，避免同日多条轰炸；所有触达必须留痕
- P0 渠道策略建议：优先站内/内部提醒；外部邮件或 IM 触达作为可选扩展，不作为首版依赖前提

## 8. 收费流程

### 流程步骤
1. 创建报价单
2. 设定收费节点
3. 记录应收金额
4. 到节点触发收款提醒
5. 财务登记实收
6. 上传凭证
7. 标记是否开票

### 收费状态
- 未报价
- 已报价待确认
- 已确认待付款
- 部分回款
- 已结清
- 退款中
- 已退款

补充规则（与阶段联动）：
- 收费节点与案件阶段绑定：例如“受理前/提交前/结果后”等节点到达触发内部提醒
- 对客户的欠款提醒需显式开启，并可配置是否需要负责人确认后发送
- 默认门槛口径：收费异常在 P0 先以风险提示和工作台暴露为主，不统一阻断提交
