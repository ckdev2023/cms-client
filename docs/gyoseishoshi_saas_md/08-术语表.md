## 先读概念

### 新人必读对象关系图

```
Lead ──转化──→ Customer ──建案──→ Case
(咨询线索)     (个人客户)          (案件)
                  │                  │
          CustomerRelation    ┌──────┼──────┐
          (客户关系映射)       │      │      │
                        CaseParty  DocReq  BillingPlan
                        (案件关联人)(资料项)(收费节点)
```

### 新人第一轮必须理解的 7 个概念

| 概念 | 一句话定义 | 详细条目 |
|------|-----------|---------|
| **Customer** | 个人客户主数据，一个客户可关联多个案件 | [Customer](#customer) |
| **Case** | 系统核心工作单元，一案一单 | [Case](#case) |
| **Group** | 内部归属团队，非企业客户 | [Group](#group) |
| **Stage** | 案件在主流程中的阶段（S1–S9） | [Stage（案件阶段）](#stage案件阶段) |
| **Gate** | 阶段间的质量关卡（A/B/C 三道） | [Gate（门槛校验）](#gate门槛校验) |
| **DocumentRequirement** | "需要什么材料"的要求，不是文件本身 | [DocumentRequirement](#documentrequirement) |
| **SubmissionPackage** | 一次对外提交的完整快照 | [SubmissionPackage](#submissionpackage) |

### 最常混淆的概念

| 经常混淆的一对 | 区别要点 |
|---------------|---------|
| DocumentRequirement vs DocumentFileVersion | 资料项是"要什么"；附件版本是"实际交的文件" |
| CaseParty vs CustomerRelation | CaseParty 参与案件校验与提交锁定；CustomerRelation 仅用于客户侧导航 |
| Group vs Company | Group 是内部团队归属；Company 是外部企业主体 |
| Stage vs Gate | Stage 标识案件在哪个阶段；Gate 是阶段间的质量关卡 |

完整的易混概念速查见文末 [附：易混概念速查](#附易混概念速查)。

## 核心业务对象

### Lead

| 项目 | 内容 |
|------|------|
| 中文名 | 咨询线索 |
| 定义 | 承接"未签约/未立案"的早期信息与跟进动作的记录。转化时生成 Customer 与 Case，避免不完整信息污染主数据 |
| 生命周期 | 新咨询 → 跟进中 → 待签约 → 已签约 → 已转案件；或 → 已流失 |
| 归属 | Lead.group → 转化时默认继承至 Customer.group → Case.group |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-核心对象语义)、[03 §3.6](03-业务规则与不变量.md#36-线索状态) |

### Customer

| 项目 | 内容 |
|------|------|
| 中文名 | 个人客户 |
| 定义 | P0 以个人客户为主体的客户记录。一个 Customer 可关联多个 Case |
| 易混辨析 | ≠ Company（企业客户，P1/P2）；≠ ClientUser（客户门户登录账号） |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-核心对象语义)、[07 §3.2](07-数据模型设计.md#32-customer个人客户) |

### Company

| 项目 | 内容 |
|------|------|
| 中文名 | 企业客户 |
| 定义 | 企业主数据实体（P1/P2 可选）。P0 阶段工作类案件的雇主/受入机构信息先作为 Case 字段沉淀，不引入独立企业实体 |
| 易混辨析 | ≠ Group（内部归属团队）；≠ Customer（个人客户） |
| 权威来源 | [03 §1.1](03-业务规则与不变量.md#11-版本边界策略)、[07 §3.3](07-数据模型设计.md#33-company企业客户) |

### Case

| 项目 | 内容 |
|------|------|
| 中文名 | 案件 |
| 定义 | 系统核心工作单元。一案一单，拥有独立的阶段、资料清单、任务、期限、收费与审计记录 |
| 归属 | Case.group 默认继承 Customer.group，为案件归属快照；Customer.group 后续变更不回写覆盖历史案件 |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-核心对象语义)、[07 §3.5](07-数据模型设计.md#35-case案件) |

### Group

| 项目 | 内容 |
|------|------|
| 中文名 | 归属团队 / 项目组 |
| 定义 | 内部归属维度，用于客户/案件的协作归属、检索筛选、工作台统计与对象级可见性边界 |
| 易混辨析 | **≠ 企业客户**（Company 是外部主体）；**≠ 来源渠道**（source_channel 是客户获取来源）；**≠ 普通标签**（Group 是治理对象，不应由业务用户自由输入） |
| P0 范围 | 单层治理与最小可见性边界；不展开字段级权限、审批流、层级树 |
| 继承链 | `Lead.group → Customer.group → Case.group`（默认继承，跨组建案/转组须留痕） |
| 权威来源 | [03 §13](03-业务规则与不变量.md#13-group-治理规则) |

### CaseParty

| 项目 | 内容 |
|------|------|
| 中文名 | 案件关联人 |
| 定义 | 案件维度的关键关系人记录，用于门槛校验、文书变量填充、责任边界与审计。角色包括主申请人、扶养者、保证人、配偶、子女、企业担当等 |
| 易混辨析 | **≠ CustomerRelation**（客户侧关联人关系仅用于检索与跳转，不作为案件门槛口径）。案件所需关键关系必须落在 CaseParty 并随提交包快照锁定 |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-核心对象语义)、[07 §3.7](07-数据模型设计.md#37-caseparty案件关联人) |

### CustomerRelation

| 项目 | 内容 |
|------|------|
| 中文名 | 客户关系映射 |
| 定义 | 客户侧的关联人关系（配偶/子女/父母/代理人等），用于展示、快速跳转、批量建案入口聚合与去重辅助 |
| 易混辨析 | **≠ CaseParty**。CustomerRelation 不作为案件门槛口径，也不随提交包锁定 |
| 权威来源 | [07 §3.2A](07-数据模型设计.md#32a-customerrelation客户关系映射) |

### ContactPerson

| 项目 | 内容 |
|------|------|
| 中文名 | 联系人 / 企业担当 |
| 定义 | 企业侧联系人或个人关系人（法定代表人、人事担当、紧急联系人等），P1/P2 可选 |
| 权威来源 | [07 §3.4](07-数据模型设计.md#34-contactperson联系人企业担当) |

## 资料与文书

### DocumentRequirement

| 项目 | 内容 |
|------|------|
| 中文名 | 资料项（材料要求） |
| 定义 | "案件需要的材料要求"，描述需要提交什么材料、由谁提供、截止日期等。资料项是"要什么"，不是文件本身 |
| 易混辨析 | **≠ DocumentFileVersion**（附件版本 = 实际提交的证据文件版本）。资料项是要求，附件是响应 |
| 权威来源 | [03 §2.3](03-业务规则与不变量.md#23-资料项与附件版本分离)、[03 §7](03-业务规则与不变量.md#7-资料项治理规则) |

### DocumentFileVersion

| 项目 | 内容 |
|------|------|
| 中文名 | 资料附件版本 |
| 定义 | 客户或内部上传的实际文件版本。每次上传生成新版本，不覆盖历史。提交包只能引用具体版本 |
| 易混辨析 | **≠ DocumentRequirement**（资料项）；**≠ GeneratedDocumentVersion**（文书版本） |
| 不可变性 | 版本一旦创建不可覆盖；审核结果写在版本上 |
| 权威来源 | [03 §2.3](03-业务规则与不变量.md#23-资料项与附件版本分离)、[07 §3.10](07-数据模型设计.md#310-documentfileversion资料附件版本) |

### Template

| 项目 | 内容 |
|------|------|
| 中文名 | 文书模板 |
| 定义 | 版本化的文书模板，包含变量字典、格式化规则与缺失策略。用于生成 GeneratedDocumentVersion |
| 权威来源 | [03 §8](03-业务规则与不变量.md#8-文书与模板规则)、[07 §3.14](07-数据模型设计.md#314-template文书模板) |

### GeneratedDocumentVersion

| 项目 | 内容 |
|------|------|
| 中文名 | 生成文书版本 |
| 定义 | 由文书模板生成的实际文书产物。每次生成/修改产出新版本，提交包引用具体版本 |
| 状态 | 草稿 → 审核中 → 待修改 → 已定稿 → 已导出 |
| 不可变性 | 导出/提交引用固定版本，保证历史可追溯 |
| 权威来源 | [03 §3.3](03-业务规则与不变量.md#33-文书状态)、[07 §3.15](07-数据模型设计.md#315-generateddocumentversion生成文书版本) |

### SubmissionPackage

| 项目 | 内容 |
|------|------|
| 中文名 | 提交包 |
| 定义 | 一次对外提交的快照，固化提交日期、受理机关、回执，并锁定本次引用的资料版本、文书版本、关键字段快照、校验报告与复核记录 |
| 不可变性 | 提交后不允许覆盖式替换历史引用；后续补正或更新通过"新版本 + 新提交包"完成 |
| 易混辨析 | 补正属于新一轮提交，形成"补正提交包"并关联原提交包（related_submission_id） |
| 权威来源 | [03 §2.4](03-业务规则与不变量.md#24-提交包锁定与不可覆盖)、[03 §14](03-业务规则与不变量.md#14-提交动作强规则) |

### CaseTemplate

| 项目 | 内容 |
|------|------|
| 中文名 | 案件模板 |
| 定义 | 按"签证类型 × 申请类型"驱动，用于生成资料清单、默认任务、校验规则集与文书模板建议。P0 预置家族滞在与工作签证（技人国）两类模板 |
| 权威来源 | [07 §3.8](07-数据模型设计.md#38-casetemplate案件模板) |

### 材料字典

| 项目 | 内容 |
|------|------|
| 中文名 | 通用材料字典 |
| 定义 | 系统内置的资料项命名标准库，包含默认说明/样例与时效建议。案件模板从材料字典中引用标准项，事务所级可覆盖 |
| 权威来源 | [03 §7.3](03-业务规则与不变量.md#73-材料字典与模板治理) |

## 流程与校验机制

### Stage（案件阶段）

| 项目 | 内容 |
|------|------|
| 中文名 | 案件阶段 |
| 定义 | 案件在主流程中所处的阶段，用于组织工作、看板统计与驱动默认任务/提醒。P0 冻结 S1–S9 共 9 个阶段 |
| 枚举 | S1 已建档、S2 资料收集中、S3 资料审核中、S4 文书制作中、S5 待校验、S6 待提交、S7 已提交审理中、S8 已出结果、S9 已归档 |
| 易混辨析 | 阶段用于"管理与协作"，不直接等于"资料是否齐备"。阻断由 Gate 负责 |
| 权威来源 | [03 §3.1](03-业务规则与不变量.md#31-案件阶段)、[04 §1.2](04-核心流程与状态流转.md#12-案件阶段stage) |

### Gate（门槛校验）

| 项目 | 内容 |
|------|------|
| 中文名 | 门槛校验 |
| 定义 | 流程中的关键关卡，用于阻断关键错误。P0 冻结 3 道 Gate，每次执行沉淀为 ValidationRun |
| 枚举 | Gate-A（进入文书）、Gate-B（进入提交前校验）、Gate-C（提交/生成提交包前） |
| 权威来源 | [03 §4.3](03-业务规则与不变量.md#43-三道门槛gate规则定义)、[04 §1.5](04-核心流程与状态流转.md#15-门槛校验gate触发点与流程串联) |

### 硬性阻断 / 软性提示

| 项目 | 内容 |
|------|------|
| 定义 | 校验结果的两个严重级别。**硬性阻断**：必须修复才可继续（如必填字段缺失、关键资料项未通过）。**软性提示**：风险提示但不阻断流程（如日期逻辑风险、欠款风险） |
| 权威来源 | [03 §4.1](03-业务规则与不变量.md#41-硬性阻断p0-最小集合)、[03 §4.2](03-业务规则与不变量.md#42-软性提示p0-最小集合) |

### ValidationRun

| 项目 | 内容 |
|------|------|
| 中文名 | 校验运行记录 |
| 定义 | 每次 Gate 执行产生的校验结果记录，区分硬性阻断与软性提示，沉淀报告供复核与提交包引用 |
| 状态 | pending / failed / passed |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-核心对象语义)、[07 §3.16](07-数据模型设计.md#316-validationrun提交前校验) |

### ReviewRecord

| 项目 | 内容 |
|------|------|
| 中文名 | 双人复核记录 |
| 定义 | 校验通过后进入待提交前的复核环节。驳回必须给出原因。属于"按策略启用"的能力，P0 默认不强制 |
| 结果 | approved / rejected |
| 权威来源 | [03 §1.2](03-业务规则与不变量.md#12-能力预留但默认关闭)、[07 §3.17](07-数据模型设计.md#317-reviewrecord双人复核) |

### waived（无需提供/不适用）

| 项目 | 内容 |
|------|------|
| 定义 | 资料项的一种状态：因策略或条件变化被豁免。必须记录原因（结构化原因码 + 自由文本）、操作人与时间。被标记为 waived 的资料项从完成率分母中剔除，但仍可追溯 |
| 权威来源 | [03 §7.2](03-业务规则与不变量.md#72-无需提供不适用waived治理规则) |

## 协作与权限

| 项目 | 内容 |
|------|------|
| 中文名 | 案件协作成员 |
| 定义 | 案件绑定的协作成员记录，支持多人协作与跨组授权。角色类型包括 principal（负责人）、collaborator（协作者）、viewer（仅查看） |

### visible_scope（可见性标记）

| 项目 | 内容 |
|------|------|
| 定义 | 沟通记录、附件、文书等必须携带的标记字段，区分 `internal_only`（仅内部可见）与 `client_visible`（客户可见） |
| 易混辨析 | **可见 ≠ 可导出**：即使客户或内部用户可见某数据，导出/下载/分享仍需独立权限控制并留痕 |
| 权威来源 | [03 §2.5](03-业务规则与不变量.md#25-可见性标记)、[03 §10.4](03-业务规则与不变量.md#104-可见--可导出) |

### Task

|------|------|

## 附：易混概念速查

| 概念 A | 概念 B | 区别要点 |
|--------|--------|---------|
| DocumentRequirement（资料项） | DocumentFileVersion（附件版本） | 资料项是"要什么材料"；附件版本是"实际提交的文件版本" |
| CaseParty（案件关联人） | CustomerRelation（客户关系映射） | CaseParty 用于案件门槛校验与提交包锁定；CustomerRelation 仅用于客户侧检索跳转，不参与校验 |
| Group（归属团队） | Company（企业客户） | Group 是内部协作归属维度；Company 是外部企业主体 |
| Group | 来源渠道（source_channel） | Group 表达归属团队；来源渠道表达客户获取途径 |
| Group | 普通标签 | Group 是治理对象（不建议自由输入，不可物理删除被引用的记录）；标签是自由分类 |
| Stage（阶段） | Gate（门槛） | Stage 用于管理与协作，标识案件所处环节；Gate 用于阻断关键错误，是阶段间的质量关卡 |
| 硬性阻断 | 软性提示 | 硬性阻断必须修复才可继续；软性提示仅风险提示，不阻断流程 |
| 可见（view） | 可导出（export） | 可见不等于可导出；导出/下载/分享属于敏感动作，需独立权限控制并留痕 |
| SubmissionPackage（提交包） | 补正提交包 | 首次提交与补正都生成独立提交包；补正包通过 related_submission_id 关联原包 |
| Template（文书模板） | CaseTemplate（案件模板） | Template 是文书生成模板（变量字典/格式规则）；CaseTemplate 是案件类型模板（驱动资料清单/默认任务/校验规则集） |
| Customer（个人客户） | ClientUser（客户门户用户） | Customer 是业务主数据；ClientUser 是客户门户的登录账号，绑定 Customer |
