## 1. P0 版本策略冻结

P0 采用 **"宽链路、浅能力、强留痕"** 策略：主链路模块保留，但每个模块只做最小闭环，不在首版追求高度配置化。

### 1.1 版本边界策略

- 客户门户定位为 P1，P0 不纳入首版核心范围。
- 双人复核在模型和流程上预留，但**默认不作为全局强制门槛**；由事务所或案件模板按需开启。
- 收费节点与案件联动是必须能力，但"未付款是否阻断提交"在 P0 采用**提示优先**，不作为统一硬阻断规则。
- 首版模板范围明确为 2 类高频案件模板：**家族滞在**、**工作签证（技人国）**。
- P0 提醒能力以内部提醒与站内提醒为主，客户侧外部触达后置。
- P0 客户实体以"个人客户"为主；工作类案件的雇主/受入机构信息先作为案件字段沉淀，企业主数据（企业客户实体、企业侧工作台、批量导入）后置到 P1/P2。
- P0 先保证内部用户可完成全链路，客户门户默认不纳入首版主流程。

### 1.2 能力预留但默认关闭

以下能力在模型层预留"可开启但不强制"的开关，P0 默认关闭或降级：

| 能力 | P0 默认状态 | 开关位置 |
|------|-----------|---------|
| 双人复核 | 关闭；事务所或案件模板可按需开启 | CaseTemplate.review_required_flag |
| 收费阻断 | 提示模式（warn） | CaseTemplate.billing_gate_mode / BillingPlan.gate_effect_mode |
| 客户门户 | 不纳入 P0 | 版本范围 |
| 外部触达（邮件/短信/IM） | 可选扩展，不作为首版依赖 | 提醒渠道配置 |
| 企业客户主数据 | 不引入；雇主信息沉淀在 Case 字段 | 版本范围 |
| Group 层级树 / 字段级权限 / 审批流 | 不做 | 版本范围 |

## 2. 对象边界与不可变规则

### 2.1 核心对象语义

| 对象 | 定义 | 补充 |
|------|------|------|
| Customer | 个人客户。P0 以个人客户为主体 | 客户可关联多个案件 |
| Case | 案件。一案一单，拥有独立的阶段、资料清单、任务、期限、收费、审计 | — |
| CaseParty | 案件关联人。用于门槛校验、文书变量、责任边界与审计 | ≠ CustomerRelation（客户侧关联人关系仅用于检索与跳转） |
| Group | 内部归属团队/项目组。用于协作归属、检索口径、对象级可见性边界 | ≠ 企业客户 ≠ 来源渠道 ≠ 普通标签 |
| Lead | 咨询线索。承接未签约/未立案的早期信息 | 转化时生成 Customer 与 Case |
| DocumentRequirement | 资料项 = 材料要求 | ≠ 附件（DocumentFileVersion） |
| DocumentFileVersion | 附件版本 = 提交的证据版本 | 每次上传生成新版本，不覆盖历史 |
| SubmissionPackage | 提交包 = 一次对外提交的快照 | 引用具体版本并锁定 |
| ValidationRun | 校验运行记录 | 区分硬性阻断与软性提示 |

### 2.2 归属继承链

```
Lead.group → Customer.group → Case.group（默认继承）
```

- **Case.group 是案件归属快照**：Customer.group 的后续变更不回写覆盖历史案件。
- 跨组建案 / 转组必须留痕（操作人、时间、原因），并写入审计日志。
- 线索转客户时默认继承 Lead.group；如转化时改组，必须记录原因、操作人与时间。

### 2.3 资料项与附件版本分离

- 资料项是"材料要求"，附件是"提交的证据版本"。
- 同一资料项允许多版本追加；每次上传生成新版本，不覆盖历史版本。
- 文书与提交包引用必须指向"具体版本"，保证审计成立。
- 多对象 / 多案件共用材料：允许多个案件引用同一版本；提交后若需更新材料，应创建新版本并用于新的提交包 / 补正包。

### 2.4 提交包锁定与不可覆盖

- 提交包引用"具体附件版本 / 文书版本 / 关键字段快照"。
- 提交后不允许覆盖式替换历史，只能新增版本并生成新提交包。
- 补正属于新一轮提交：形成"补正提交包"并与原提交包关联。
- 无论首次提交还是补正提交，只要生成了新的对外提交，就必须创建新的提交包。

### 2.5 可见性标记

- 沟通记录、附件、文书都**必须标记**"内部可见 / 客户可见"（`visible_scope`：`internal_only` / `client_visible`）。
- 客户侧与内部侧需分离权限域（字段、附件、沟通记录可见性）。
- 客户侧默认只可见客户门户定义范围内的数据。

### 2.6 归并与去重

- 归并与去重**不做物理覆盖删除**：保留原记录，建立归并关系，历史沟通、附件、审计日志不断链。
- 线索转客户、客户转案件、多对象批量建档 / 批量建案时都应触发去重检查。
- 去重匹配优先级：电话/邮箱优先，其次姓名+生日（或证件号），支持人工确认合并。

## 3. 最小状态机定义（P0 固定）

### 3.1 案件阶段

| 阶段 | 代号 | 说明 |
|------|------|------|
| 已建档 | S1 | 已签约/已立案 |
| 资料收集中 | S2 | 对客户发出资料请求 |
| 资料审核中 | S3 | 内部审核/退回补正 |
| 文书制作中 | S4 | 模板生成、整理、审核、定稿 |
| 待校验 | S5 | 进入提交前校验关卡 |
| 待提交 | S6 | 校验通过，满足复核要求（如启用）后等待对外提交 |
| 已提交审理中 | S7 | 含补正/追加提交 |
| 已出结果 | S8 | 许可/不许可 |
| 已归档 | S9 | 结案 |

### 3.2 资料项状态

| 状态 | 枚举值 | 说明 |
|------|--------|------|
| 未发送 | `not_sent` | 资料项已生成但尚未对客户发出请求 |

| 审核通过 | `approved` | 满足要求，可用于文书/提交包引用 |

| 无需提供 | `waived` | 因策略/条件变化被豁免；必须记录原因与操作人 |
| 已过期需重交 | `expired` | 超过时效或证件有效期不足 |

|------|------|
| 新咨询 | — |

## 4. 校验门槛分层（硬阻断 / 软提示）

校验分为两层，避免系统过度阻断影响效率，也避免完全不阻断导致质量失控。

### 4.1 硬性阻断（P0 最小集合）

必须修复才可继续：

- 必填字段完整
- 关键关系人（CaseParty）齐备（主申请人 / 扶养者绑定）
- 必交资料项齐备且通过（`approved`）
- 关键文书已定稿（`final`）
- （如启用）双人复核签名通过

### 4.2 软性提示（P0 最小集合）

风险提示，不阻断流程：

- 日期逻辑 / 有效期风险
- 收费欠款风险
- 建议补充材料与风险提示
- 附件版本过旧但未过期
- 关键信息疑似未更新（地址/在留期限等）

### 4.3 三道门槛（Gate）规则定义

P0 冻结 3 个 Gate。每次 Gate 执行都沉淀为一条校验运行记录（ValidationRun）与报告。

> Gate 的触发点、阶段串联与通过后动作详见 [核心流程与状态流转](04-核心流程与状态流转.md) §1.5。

#### Gate-A：进入文书

- **硬性阻断**：关键关系人（CaseParty）齐备；关键资料项达到可做文书的最低门槛（按模板定义，且状态为 `approved`）
- **软性提示**：资料项附件版本过旧但未过期、关键信息疑似未更新（地址/在留期限等）、欠款风险提示

#### Gate-B：进入提交前校验

- **硬性阻断**：关键文书为已定稿；关键资料项齐备且通过（必交项）；关键字段必填完成
- **软性提示**：日期逻辑/有效期风险、材料时效建议、收费节点异常提示

#### Gate-C：提交（生成提交包前）

- **硬性阻断**：最新一次校验结果为通过；如启用双人复核则复核通过；提交信息字段（提交日期/机关/回执等）满足最小要求
- **软性提示**：欠款风险、非关键材料建议补充
- **欠款特别规则**：若存在欠款风险且仍要提交，必须先完成风险确认留痕（确认人、原因，可选上传凭证）

### 4.4 重新校验触发条件

以下变化发生后，必须重新生成 ValidationRun，避免提交引用与校验结果不一致：

- 任一必交资料项新增了附件版本且被审核通过 / 退回补正
- 任一关键关系人（CaseParty）新增 / 删除 / 角色变更
- 任一关键字段发生变更（如姓名、在留期限、住址、雇主信息等按模板定义）
- 任一关键文书发生变更或重新定稿（生成新版本）
- 复核启用策略发生变化（本案从不启用变为启用，或模板要求变更）

## 5. 审计事件最小集合

案件时间线用于"单页可扫读"，审计日志用于"可追责"，两者事件口径一致但展示层次可不同。

### 5.1 事件类型（P0 必须覆盖）

| 类别 | 事件 |
|------|------|
| 案件 | 创建、阶段变更、负责人/协作者变更、Group 变更、跨组建案/转组 |
| 资料 | 资料项新增/删除（受限）、标记无需提供/不适用（含原因）、状态变更、审核意见变更 |
| 附件 | 附件版本新增、附件版本替换/撤回（如允许）、下载/导出/分享 |
| 催办与沟通 | 催办发送、沟通记录新增、沟通记录可见性变更 |
| 任务与期限 | 任务创建/完成、期限创建/变更/到期 |
| 提交与风控 | 校验执行与结果、复核签名、提交包生成/导出、收费节点确认、回款记录新增 |
| 敏感动作 | 导出/下载/分享、批量发送、负责人交接、欠款风险确认 |

### 5.2 审计日志治理规则

- 审计日志**默认不可删**。
- 敏感导出与跨案件批量导出需权限控制。
- 审计日志字段最小集合：对象类型、对象 ID、操作类型、操作原因（可选）、变更前后数据、操作人、操作时间。

## 6. 收费与欠款策略

### 6.1 核心规则

- P0 默认**不将欠款作为统一提交硬阻断**。
- 收费节点与案件阶段绑定（例如"受理前/提交前/结果后"），到达节点触发内部提醒。
- 收费异常在 P0 先以**风险提示和工作台暴露**为主，不统一阻断提交。

### 6.2 欠款继续提交的留痕要求

在欠款状态下继续提交时，必须：

1. 记录风险确认（确认人、原因，可选上传凭证）
2. 风险确认进入案件时间线与审计日志
3. 工作台应将未确认的欠款案件聚合为风险案件并提示补录

### 6.3 阶段联动

- 如存在收费节点，案件阶段流转与提交包生成时应关联到对应收费节点的当前状态，便于事后对账与追溯。
- 对客户的欠款提醒需显式开启，并可配置是否需要负责人确认后发送。

## 7. 资料项治理规则

### 7.1 完成率口径

- 默认口径：`完成率 = 已审核通过的必交资料项数 / 必交资料项总数`
- **分母剔除**：标记为"无需提供/不适用"（`waived`）的资料项不计入总数，但必须记录原因、操作人与操作时间。
- 分组口径：支持按提供方 / 按类别输出子完成率（多对象/关联人场景优先用提供方）。

### 7.2 无需提供/不适用（waived）治理规则

- **必填留痕**：原因（结构化原因码 + 自由文本补充）、操作人、操作时间。
- **权限控制**：默认仅允许助理/主办人及以上角色执行；对"必交资料项"的豁免可按模板或事务所配置要求主办人确认。
- **生命周期约束**：资料项一旦被提交包引用并完成对外提交，不应删除或覆盖历史记录；后续若策略变化需再次索取，创建新版本或新增资料项继续追踪。
- **指标口径**：被标记为 `waived` 的资料项从分母剔除，但仍需在案件时间线与审计日志中可追溯。

### 7.3 材料字典与模板治理

- 系统内置"通用材料字典"：统一资料项命名、默认说明/样例、时效建议。
- 案件模板由"签证类型 × 申请类型"驱动，可在事务所级进行覆盖（增删改资料项、文案与样例、截止日规则）。
- 案件内允许本案级调整：新增资料项、标记无需提供/不适用（必须写明原因并留痕）。

### 7.4 资料项标记要求

- 资料项必须可区分"材料要求"与"附件版本"。
- 资料项必须标记提供方（主申请人 / 扶养者或保证人 / 事务所内部）、责任人、截止日与退回原因。
- 多对象/多案件场景支持"同一材料多案件引用"：引用指向具体版本，避免共享可变文件。

## 8. 文书与模板规则

- **变量字典**：每个变量需定义来源字段、格式化规则、缺失时策略（阻断/提示/留空）。
- **版本不可变**：文书导出应关联到某个版本，便于审计与提交包锁定。导出后又改动不影响已锁定版本。
- 进入提交前校验的最低门槛：关键资料项达到"审核通过"且关键文书为"已定稿"。

---

## 9. 沟通记录规则

- **最小标准**：渠道、对象、摘要、结论、下一步动作（可一键转任务）。
- 内部备注与客户可见内容**必须分离**，避免误发敏感信息。
- 沟通记录可见性变更属于审计事件。

---

## 10. 权限与可见性规则

### 10.1 角色权限模型

P0 最小角色集合：管理员 / 主办人 / 助理 / 销售 / 财务。

### 10.2 可见性口径

- **管理员**：默认可见全量对象。
- **普通用户**：默认仅可见本组对象、本人负责对象及被显式协作授权的案件。
- **客户侧**：默认只展示当前阶段、下一步动作、资料清单状态、付款状态；不展示内部风险与审计细节。

### 10.3 三维访问控制

内部侧对象级访问控制至少同时考虑三个维度：

1. **角色**
2. **Group**（归属团队）
3. **负责人/协作者**

### 10.4 可见 ≠ 可导出

- 权限表达拆分为：**可见（view）**、**可编辑（edit）**、**可导出/下载（export）**。
- 即使可见也不默认等于可导出。
- 导出、下载、分享链接、批量发送等动作**必须记录到审计日志**。

### 10.5 影响可见范围的变更

Group 变更、跨组建案、负责人交接等影响可见范围的操作必须落审计日志，并能追溯变更前后归属。

---

## 11. 提醒与触达规则

### 11.1 P0 渠道策略

- P0 以内部提醒与站内提醒为主。
- 外部邮件或 IM 触达作为可选扩展，不作为首版依赖前提。
- 客户侧提醒需支持开关与渠道偏好（邮件/短信/IM），并记录触达留痕。

### 11.2 聚合与去重

- 同一扶养者/企业在留期限对应多个案件时，提醒在内部侧**聚合展示**。
- 客户侧提醒默认仅发送主提醒，避免同日多条轰炸。
- 所有触达必须留痕。

### 11.3 催办升级

- 催办需可配置升级链路：客户未响应 → 提醒负责人 → 提醒主管/主办人。
- 同一客户/扶养者的多案件共用材料需聚合催办，避免重复打扰。

---

## 12. 归并与去重规则

- 归并不是物理覆盖删除，而是保留原记录与归并关系，避免历史沟通、附件、审计日志断链。
- 去重检查触发点：线索转客户、客户转案件、多对象批量建档/批量建案。
- 去重匹配策略：按电话/邮箱优先，其次姓名+生日（或证件号）；支持人工确认合并。

---

## 13. Group 治理规则

### 13.1 语义定义

Group 统一定义为**"内部归属团队/项目组"**，用于客户/案件的协作归属、检索筛选、工作台统计与对象级可见性。

**不等同于**：企业客户、来源渠道、普通标签。

### 13.2 P0 治理范围

- P0 只做**单层治理**与最小可见性边界。
- 不展开字段级权限、审批流、Group 层级树等高级能力。
- Group 是治理对象，不建议由业务用户在建档页面自由输入。
- 被 Customer/Case 引用过的 Group 建议仅允许停用，不做物理删除。

### 13.3 归属与继承

- Lead 创建时应确定 owner 与 Group；Group 作为后续 Customer/Case 的默认归属来源。
- Customer.group 用于表达客户的默认归属，不等于其所有历史案件都必须跟随变更。
- Case.group 是案件归属快照：默认继承 Customer.group，后续 Customer.group 变更不回写覆盖历史案件。

### 13.4 跨组建案与转组

- 跨组建案/转组是例外动作，应由组织策略明确是否允许。
- 执行时必须留痕：操作人、时间、原因。
- 历史案件的 Case.group 不因 Customer.group 调整而被回写覆盖。
- 若主申请人与负责人分属不同 Group，应由组织策略明确"是否允许跨组协作建案"，P0 默认允许授权后协作，但不自动放开全部跨组可见。

### 13.5 可见性口径

- 普通用户默认查看"本人负责 + 本组对象 + 显式协作对象"。
- 管理员可见全量。
- Group 变更、跨组建案等操作写入审计日志。

---

## 14. 提交动作强规则

以下规则旨在避免历史失真，P0 冻结为强规则：

- 提交包必须锁定引用：资料附件版本、文书版本、校验报告与复核记录（如存在）。
- 提交后禁止覆盖式替换历史引用；后续补正或更新必须通过"新版本 + 新提交包"完成。
- 提交包导出、回执下载、对外分享链接等属于敏感动作，必须权限控制并写入审计日志。
- 提交前必须引用"最新一次有效校验结果"；若关键资料版本、关键字段或定稿文书发生变化，应重新校验。
