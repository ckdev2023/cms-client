## 新增文档前先问三件事

1. **这是不是已有权威源的内容？** — 如果是，在已有文档中增量修改，不新建文档。
2. **这篇文档给谁读？** — 明确读者角色（新人 / 开发 / 设计 / QA / 维护者）。
3. **它属于哪一层？** — 入口、术语、规则、流程、页面、数据、还是维护文档？

## 编号体系规则

以下规则自 2026-04-08 编号收口后永久生效，取代此前所有过渡映射表和"迁移中"状态。

| # | 规则 | 说明 |
|---|------|------|
| N-1 | **物理文件名即目标编号** | 文件名的两位数字前缀就是该文档的唯一编号。不再存在"README 说新编号、文件名还是旧编号"的过渡状态 |
| N-2 | **H1 标题与文件名一致** | 每篇文档的 `# XX …` 标题编号必须与文件名前缀一致。例如 `07-数据模型设计.md` 的 H1 必须是 `# 07 数据模型设计` |
| N-3 | **站内引用使用物理文件名** | 交叉引用一律使用当前物理文件名（含编号前缀），不使用历史别名或旧编号路径 |
| N-4 | **编号不复用** | 文件被删除或归档后，其编号不再分配给新文档，避免历史引用歧义 |
| N-5 | **新增文档取下一个空闲编号** | 规范正文层新增文档时取当前最大编号 +1。入口层（`00`、`02-产品全景图`）和维护层（`99`）使用固定编号 |

## 文档模板规范

新增或重写文档时，按文档所属层级选用对应模板。模板定义的是**必须包含的结构骨架**，具体段落可根据内容增减，但骨架中标记为"必须"的段落不可省略。

### 模板 A：入口文档

适用于：`00-开始这里`、`02-产品全景图` 等入口层文档。

```markdown
# {文档标题}

> {一句话定位：这篇文档是什么、为什么需要它}

## 这份文档适合谁看                <!-- 必须 -->

- {角色 1}
- {角色 2}
- ...

## 阅读这篇后你会知道什么            <!-- 必须 -->

- {收获 1}
- {收获 2}
- ...

---

## {正文段落}                      <!-- 至少一个正文段落 -->

{根据文档职责自由组织。以下为常用段落示例：}

### 推荐阅读顺序                    <!-- 建议 -->

| 步骤 | 文档 | 时间 | 你会了解 |
|------|------|------|---------|
| ...  | ...  | ...  | ...     |

### 关键概念索引                    <!-- 建议 -->

{列出核心对象或概念，附术语表链接}

### 常见任务跳转表                  <!-- 建议 -->

| 我想了解… | 去读 |
|-----------|------|
| ...       | ...  |
```

**写作要求**：

- 不承载规则定义或流程细节，只承载导航与概览。
- 所有具体定义必须链接到权威文档，不在入口文档中双写。
- 阅读顺序须注明预计耗时，帮助读者规划时间。

---

### 模板 B：规则 / 流程 / 数据文档

适用于：`03-业务规则与不变量`、`04-核心流程与状态流转`、`07-数据模型设计` 等规范正文层文档。

```markdown
# {文档标题}

> {权威声明：本文档是 XX 的唯一权威定义 / 权威文档。}
> {引用声明：其他文档引用本文档中的定义，不得双写。}

| | |                               <!-- 必须：统一头部表 -->
|---|---|
| **本文回答什么问题** | {一到两句话} |
| **权威范围** | {本文负责定义的内容边界} |
| **不负责什么** | {明确排除的内容，附链接到对应权威文档} |
| **新人应先读** | {前置阅读建议，附链接} |
| **上游文档** | {本文依赖的输入文档} |
| **下游文档** | {引用本文的文档} |

---

## {正文章节}

{按文档职责组织正文。}
```

**写作要求**：

- 统一头部表六个字段全部必填，不可省略。
- "不负责什么"必须明确列出排除项并给出链接，避免读者在本文中找不到时产生困惑。
- 状态枚举、门槛定义等跨文档共用概念只在权威文档中完整定义，下游文档用链接引用+枚举值速查。
- 正文章节标题使用数字编号（§1、§2…），便于跨文档引用定位。

---

### 模板 C：页面规格文档

适用于：`06-页面规格/` 目录下各模块文档。

```markdown
# {模块名称}

> 规则引用：{与本模块相关的规则段落引用，附链接到 03-业务规则}
> 流程引用：{与本模块相关的流程段落引用，附链接到 04-核心流程}

| | |                               <!-- 必须：模块元信息表 -->
|---|---|
| **页面目标** | {这个模块/页面帮助用户完成什么} |
| **面向角色** | {主要使用者角色} |
| **入口** | {用户从哪里进入这个页面} |
| **MVP / P1 边界** | {当前版本做什么、不做什么} |

---

## 1. 典型任务                      <!-- 必须 -->

- {任务 1}：{一句话描述用户目标}
- {任务 2}：…
- ...

## 2. 列表页                        <!-- 如有列表页则必须 -->

### 2.1 字段

- {字段 1}
- {字段 2}
- ...

### 2.2 默认筛选

- {筛选维度 1}
- ...

### 2.3 批量动作

- {批量动作 1}
- ...

## 3. 详情页 / 核心区块              <!-- 如有详情页则必须 -->

### Tab N：{Tab 名称}

- {字段 / 区块 / 交互说明}
- ...

## 4. 关键动作                      <!-- 必须 -->

| 动作 | 触发条件 | 结果 | 权限 |
|------|---------|------|------|
| ...  | ...     | ...  | ...  |

## 5. 状态与异常                    <!-- 必须 -->

- {状态 / 异常场景}：{处理方式}
- ...

## 6. 权限与可见性                  <!-- 必须 -->

| 角色 | 可见范围 | 可编辑范围 | 可导出 |
|------|---------|-----------|--------|
| ...  | ...     | ...       | ...    |

## 7. 规则与流程引用                <!-- 必须 -->

| 引用内容 | 权威文档 |
|----------|---------|
| ...      | ...     |
```

**写作要求**：

- 页面规格只写界面职责（字段、Tab、交互、聚合口径），不重新定义业务规则。
- 规则引用必须用链接指向 03-业务规则或 04-核心流程的具体章节，不在本文中双写规则全文。
- 列表页必须列出字段、默认筛选和批量动作。
- 详情页按 Tab 组织，每个 Tab 列出字段或区块说明。
- "关键动作"表必须包含触发条件、结果和权限列，确保研发与 QA 可直接作为验收依据。
- "状态与异常"覆盖空状态、错误状态、边界条件的处理方式。
- 标注 MVP / P1 边界时，对 P1 功能使用"P1"前缀标记，避免混入 P0 实现范围。
