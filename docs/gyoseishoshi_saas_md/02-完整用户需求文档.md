# 02 完整用户需求文档

## 0. 产品范围

- 本产品首先要解决的是**事务所内部作业闭环**，即从咨询转化到提交、回款、归档的主链路在线化。
- 完整需求文档覆盖的是中长期目标，但版本实施必须分层：**MVP 优先内部后台闭环**。
- 所有能力都应尽量落到可执行机制上：模板、状态机、门槛校验、任务、提醒、提交包、审计日志。

> **冻结口径与业务规则**：版本策略、对象边界、不可变规则、最小状态机、校验门槛分层、审计事件集合等规则定义已统一归入 [03-业务规则与不变量](03-业务规则与不变量.md) 作为唯一权威来源，本文不再重复。

## 1. 用户角色

### 1.1 行政书士 / 主办人
需求：
- 掌握案件全局
- 控制案件质量
- 降低遗漏和退件风险
- 快速审核资料和文书
- 查看全所案件进展和即将到期项目

### 1.2 助理 / 事务员
需求：
- 建档
- 维护资料清单
- 催办缺件
- 更新案件状态
- 记录沟通
- 推动案件流转

### 1.3 咨询顾问 / 销售 / 前台
需求：
- 记录咨询线索
- 识别业务类型
- 跟进签约
- 将咨询转正式案件

### 1.4 财务人员
需求：
- 管理应收与实收
- 查看案件收费节点
- 记录回款和发票信息
- 统计欠款和回款情况

### 1.5 客户本人 / 企业联系人
需求：
- 在线填写资料
- 上传材料
- 查看缺件
- 接收提醒
- 查看案件进度
- 查看费用与付款状态

## 2. 核心业务对象

> 各对象的语义定义、不可变规则与归属继承链详见 [03-业务规则与不变量 §2](03-业务规则与不变量.md#2-对象边界与不可变规则)。

### 2.0 咨询线索（Lead）
咨询线索用于承接"未签约、未立案"的早期信息与跟进动作，避免把不完整信息直接污染客户主数据与案件数据。
典型字段：
- 联系方式（电话/邮箱/IM）
- 意向业务类型（签证种类/申请类型）
- 来源/介绍人
- 归属团队/项目组（group）
- 跟进状态与下一步动作
- 报价与签约状态

### 2.1 客户
- 个人客户（P0 主体）
- Group（归属团队/项目组：P0 强依赖）
- 雇主/受入机构信息（工作类案件必需，P0 先沉淀在案件字段）
- 关联人（配偶、子女、担保人、法定代表人、人事担当）

### 2.2 案件
案件是系统的核心工作单元。一个客户可以对应多个案件，一个案件可以绑定多个关联人和多个文档对象。

### 2.3 资料项
资料项是"案件需要的材料要求"，不是单纯附件。

### 2.4 沟通记录
沟通必须沉淀为正式业务记录，而不是散落在备注里。

### 2.5 文书
文书需支持模板化、变量填充、版本管理和导出。

### 2.6 任务
案件推进依赖任务驱动，而不是仅靠状态。

### 2.7 期限
在留期限、补件期限、付款期限、预约日期等都需要统一管理。

### 2.8 收费记录
收费要和案件阶段联动，并支持分阶段收款。

### 2.9 提交包（Submission Package）
提交包用于固化"某一次对外提交"的快照，确保可审计、可追溯、可复盘。
建议能力：
- 固化：提交日期、受理机关/窗口、提交凭证/回执
- 锁定引用：本次提交所使用的资料项版本、文书版本、关键字段快照
- 关联：校验报告、复核记录（双人复核签名）

## 3. 功能需求总览

> 各功能涉及的业务规则（校验门槛、审计事件、欠款策略、资料项治理、权限可见性等）统一定义在 [03-业务规则与不变量](03-业务规则与不变量.md)，本节仅描述功能目标与需求清单。

### 3.1 客户资料采集
目标：标准化采集客户资料，减少重复录入。

需求：
- 支持个人客户创建（P0）
- 支持客户自定义识别名（对内显示名/简称）
- 支持フリガナ（カナ）与多语言姓名
- 支持客户写真/头像
- 支持介绍人/来源记录与客户归属团队/项目组（group）
- 支持线索转客户时默认继承 Lead.group，并允许按权限调整默认归属
- 支持客户备注（可检索）
- 支持家族签/多对象场景的批量建档与批量建案（个人客户为主体；通过"关联人关系 + 案件关联人"表达关系）
- 支持自定义表单
- 支持客户在线填写
- 支持事务所代录
- 支持多语言
- 支持字段校验
- 支持证件上传
- 支持历史变更记录
- 支持客户关系映射

家族签/扶养类办理（典型：扶养者已有工作签证，配偶/子女办理家族滞在）建议流程：
- 创建咨询线索：以咨询人（通常为扶养者/保证人）登记联系方式、来源、group 与备注
- 创建主申请人客户：为老婆/孩子分别建立个人客户档案，并默认继承该线索的 group
- 创建案件：为每个主申请人分别创建案件，并强制绑定扶养者/保证人作为案件关联人
- 必须补齐扶养者信息：在留资格、在留卡号、在留期限到期日、联系方式与证件影像等作为进入做文书/提交前校验的门槛
- 资料清单按提供方拆分：主申请人提供、扶养者提供、事务所内部产出，并支持分别催办留痕
- 结案后续跟进：基于扶养者在留期限自动生成续签/变更的提醒与跟进任务，支持一键创建后续案件
- 复用路径：若扶养者已是本所历史客户或已有历史案件，支持一键带入其在留信息与证件附件，并对关键字段触发更新确认（在留期限、住址、雇主/收入等）

### 3.2 案件建档
目标：把咨询转为标准化案件。

需求：
- 从客户档案一键建案
- 自动生成案件编号
- 绑定主申请人、雇主/受入机构信息（工作类案件）、关联人
- 指派负责人和协作人
- 设置优先级、标签、风险等级
- 上传委托合同、报价单
- 支持案件模板
- 创建案件时必须选择：案件类型/签证类型、申请类型（认定/变更/更新）
- 系统根据"签证类型 × 申请类型"自动匹配案件模板，并在建案完成后自动生成资料清单

> Group 归属继承与跨组建案规则 → [03 §13](03-业务规则与不变量.md#13-group-治理规则)

### 3.3 资料清单管理
目标：清楚知道要什么资料、谁提交、是否通过。

需求：
- 按案件模板自动生成资料清单
- 支持内部资料和客户资料分离
- 支持状态管理
- 支持备注、样例、审核意见
- 支持多版本上传
- 支持到期提醒
- 支持完成率展示
- 客户上传后系统自动更新状态为"已提交待审核"
- 审核通过后自动计入完成率；退回补正后显示退回原因

> 完成率口径、waived 治理规则、材料字典与模板治理 → [03 §7](03-业务规则与不变量.md#7-资料项治理规则)
>
> 资料项与附件版本分离规则 → [03 §2.3](03-业务规则与不变量.md#23-资料项与附件版本分离)

### 3.4 缺件催办
目标：系统化催办，不依赖人工记忆。

需求：
- 自动识别缺件、过期件、不合格件
- 生成待催办列表
- 支持自动提醒与手动提醒
- 支持多语言消息模板
- 支持催办记录留痕
- 支持逾期升级提醒

> 催办升级链路与聚合去重规则 → [03 §11.3](03-业务规则与不变量.md#113-催办升级)

### 3.5 与客户来回沟通
目标：统一沉淀所有案件沟通。

需求：
- 记录电话、邮件、面谈、微信/LINE 等沟通
- 支持案件时间线
- 支持内部备注与客户可见备注分离
- 支持@同事
- 支持沟通转待办
- 支持消息模板
- 支持附件关联

> 沟通记录最小标准与可见性分离规则 → [03 §9](03-业务规则与不变量.md#9-沟通记录规则)

### 3.6 模板文书输出
目标：提升文书产出效率与一致性。

需求：
- 模板库管理
- 变量自动填充
- 多案件类型模板配置
- 条件段落
- 多语言模板
- 版本管理
- 导出 Word / PDF
- 定稿留痕

> 变量字典与版本不可变规则 → [03 §8](03-业务规则与不变量.md#8-文书与模板规则)

### 3.7 提交前校验
目标：降低提交错误和补正率。

需求：
- 校验必填字段
- 校验必需资料是否齐全
- 校验日期逻辑
- 校验证件有效期
- 校验收费风险提示（P0 默认不作为统一硬阻断）
- 支持双人复核（默认按模板/案件启用）
- 生成校验报告

> 校验门槛分层（硬阻断/软提示）与三道 Gate 定义 → [03 §4](03-业务规则与不变量.md#4-校验门槛分层硬阻断--软提示)
>
> 提交动作强规则 → [03 §14](03-业务规则与不变量.md#14-提交动作强规则)

### 3.8 期限 / 更新提醒
目标：避免逾期，挖掘续签机会。

需求：
- 管理多种期限
- 自定义提醒规则
- 提醒负责人与客户
- 日历视图
- 90/60/30/7 天多节点提醒
- 自动创建续签待办或新线索
- 支持家庭办理场景的扶养者期限跟进

> 提醒聚合去重与渠道策略 → [03 §11](03-业务规则与不变量.md#11-提醒与触达规则)

### 3.9 进度可视化
目标：让内部和客户都能看懂案件进度。

需求：
- 看板视图
- 时间线视图
- 阶段进度条
- 当前待办
- 下一步动作
- 风险提示
- 客户侧简化进度页

> 客户侧可见性口径 → [03 §10.2](03-业务规则与不变量.md#102-可见性口径)

### 3.10 收费与回款
目标：让财务和案件协同运作。

需求：
- 报价单
- 分阶段收费
- 应收/实收/欠款管理
- 上传支付凭证
- 欠款提醒
- 发票记录
- 财务报表

> 收费与欠款策略（欠款不阻断提交、强留痕要求、阶段联动） → [03 §6](03-业务规则与不变量.md#6-收费与欠款策略)

### 3.11 留痕与审计
目标：满足内控、追责、风控和纠纷防范。

需求：
- 记录关键操作日志
- 记录操作人、时间、对象、变更前后
- 查看案件审计时间线
- 支持敏感操作审批
- 支持日志导出

> 审计事件最小集合与治理规则 → [03 §5](03-业务规则与不变量.md#5-审计事件最小集合)

### 3.12 权限与协作控制
目标：让多人协作可控，避免误看、误改、误发。

需求：
- 支持最小角色权限模型（主办人/助理/销售/财务/管理员/客户侧用户）
- 支持页面级、对象级的访问控制
- 支持内部可见 / 客户可见的字段与记录范围控制
- 支持敏感操作权限（导出、删除、批量修改、提交包导出）
- 支持案件负责人交接与协作留痕

> 权限模型、三维访问控制与可见≠可导出规则 → [03 §10](03-业务规则与不变量.md#10-权限与可见性规则)

### 3.13 全局搜索、去重与归并
目标：减少重复建档，提升日常检索效率。

需求：
- 支持按案件编号、客户名、フリガナ、电话、邮箱、在留卡号、法人番号等全局搜索
- 支持线索、客户、企业、案件的统一检索入口
- 支持按 group（归属团队/项目组）过滤与展示，保证同名客户/案件可快速区分
- 支持重复数据提示与人工归并
- 支持归并后的历史保留与映射追溯

> 归并与去重规则 → [03 §12](03-业务规则与不变量.md#12-归并与去重规则)

## 5. 非功能需求

### 5.1 安全性
- 数据加密
- 附件权限控制
- 审计日志不可随意删除
- 二步验证
- 敏感字段脱敏

> 可见性标记与导出留痕规则 → [03 §10](03-业务规则与不变量.md#10-权限与可见性规则)

### 5.2 稳定性
- 大文件上传
- 自动保存
- 误删防护
- 历史版本恢复

### 5.3 易用性
- 步骤化表单
- 批量操作
- 搜索与筛选
- 移动端查看提醒与上传资料

### 5.4 可扩展性
- 自定义字段
- 自定义模板
- 新案件类型扩展
- 开放 API
- 多组织支持

### 5.5 性能、可观测与数据保全
- 常用列表页与案件详情页在常规数据量下应保持可接受响应速度，避免因附件和时间线过长导致页面不可用
- 大文件上传需支持断点续传/失败重试/明确失败提示，避免客户与助理反复上传
- 自动保存、定时任务、提醒发送、导出生成等后台任务需有状态追踪与失败告警
- 关键业务对象需支持备份与恢复策略，尤其是附件版本、提交包、审计日志、回款记录
- 需要有基础可观测性：至少能追踪提醒是否发出、校验是否执行、导出是否成功、提交包是否生成
