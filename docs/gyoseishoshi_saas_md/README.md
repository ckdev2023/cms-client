## 核心原则

1. **一处定义，多处引用** — 同一概念只在一个文档写完整定义，其他文档用链接引用。
2. **按层分工** — 规则在"业务规则与不变量"定义，流程在"核心流程与状态机"定义，页面在"页面规格"定义，数据在"数据模型"定义。不跨层双写。
3. **术语统一** — 易混概念（Group、CaseParty、DocumentRequirement 等）以术语表为准，避免同义词漂移。
4. **页面只写界面** — 页面规格只写界面字段、交互和聚合口径，不承载规则全文。
5. **数据只写结构** — 数据模型只写实体、字段、关系和约束，不重复业务策略全文。

## 变更定位规则

**在修改任何需求前，先判断改动属于哪一层，再按下表定位入口文档。**

| 改动类型 | 首先修改 | 然后检查（是否需要同步引用） |
|----------|---------|--------------------------|
| 改产品定位 / 目标用户 / 非目标 | 01 产品定位与边界 | — |
| 改版本范围 / P0-P2 边界 / 排除项 | 02 版本范围与优先级 | 不改规则正文 |
| 改冻结口径 / 对象边界 / 不可变规则 | 03 业务规则与不变量 | 04 流程（门槛触发点）→ 06 页面规格 → 07 数据模型 |
| 改阶段 / Gate / 状态转移 | 04 核心流程与状态机 | 03 规则（如涉及门槛条件）→ 06 页面规格 → 07 数据模型（枚举） |
| 改导航结构 / 全局交互 | 05 信息架构与页面地图 | 06 页面规格（受影响模块） |
| 改某页面字段 / 交互 / 聚合口径 | 06 页面规格（对应模块） | 07 数据模型（如涉及新字段） |
| 改字段 / 实体 / 关系 / 约束 | 07 数据模型 | 06 页面规格（引用该字段的页面）→ 04 流程（引用该实体的转移规则） |
| 改概念语义 / 术语定义 | 08 术语表 | 全文档检查该术语的引用一致性 |
| 改校验门槛项（硬阻断/软提示） | 03 业务规则与不变量 → 04 流程（Gate 触发点） | 06 页面规格（校验结果展示）→ 07 数据模型（ValidationRun） |
| 改审计事件类型 | 03 业务规则与不变量 | 07 数据模型（AuditLog）→ 04 流程（时间线） |
| 改收费 / 欠款策略 | 03 业务规则与不变量 | 04 流程（收费流程步骤）→ 06 页面规格（收费页面） |

### 变更检查清单

每次变更完成后，回答以下问题确认无遗漏：

- [ ] 改动是否只修改了对应层的权威文档？
- [ ] 其他文档中引用该概念的段落是否仍然指向正确的权威源？
- [ ] 是否有文档重新双写了本次改动的内容（应避免）？
- [ ] 术语表中是否有需要更新或新增的词条？

## 维护约定

- **新增需求**时，先用"变更定位规则"判断归属文档，再增量修改。不要在任意文档中随意追加段落。
- **跨层变更**（如同时涉及规则和页面）先改上游（03），再改下游（06），最后检查引用一致性。
- **新增概念或实体**时，先在术语表（08）定义，再在对应层文档中使用。
- **文档拆分**：页面规格（06）按模块拆为独立文件，便于局部增量维护。其他文档应保持单文件。
- **审计工具**：`AUDIT-doc-boundaries.md` 记录了文档间重复内容的完整审计。迁移期间可作为参考。
