## 0. 阅读说明

| 标记 | 含义 |
|------|------|
| **UI ✅** | 当前原型有可点击页面/按钮覆盖 |
| **UI ⚠️** | 原型有部分 UI 但功能不完整（如 toast-only） |
| **UI ❌** | 原型无对应页面，仅有业务定义 |
| **P0 映射** | 该步骤对应的 P0 状态机阶段 / 子阶段 / 实体 |
| **可测** | 是否可在原型中构造并验证 |

## 2. 逐步拆解

### 阶段 A：咨询（CONSULTING）

| 维度 | 内容 |
|------|------|
| **对应步骤** | Step 1–2 |
| **角色** | 行政书士 / 助理 |
| **动作** | 创建咨询线索；录入客户基础信息 |
| **输入字段** | `name`、`phone`、`email`、`location`（海外/日本）、`source_type`（介绍/广告/自然）、`referrer_name`、`business_type`（经营管理签）、`group`、`owner` |
| **输出** | Lead 记录创建，状态 = `new` |
| **异常分支** | 重复线索检测（同名同电话）；必填项缺失校验 |
| **P0 映射** | `Lead` 实体；leads-message 列表 + 新建弹窗 |
| **原型页面** | `leads-message/index.html` → 「新增咨询」弹窗 |
| **可测** | **UI ✅** — 可点击新增按钮，填写表单，确认创建，验证列表出现新行 |

---

### 阶段 B：问卷 & 报价（CONSULTING 持续）

| 维度 | 内容 |
|------|------|
| **对应步骤** | Step 3–4 |
| **角色** | 行政书士 |
| **动作** | 发送问卷《2025M_C经管签信息表》；回收后生成报价 |
| **输入字段** | `personal_info`、`education`、`work_experience`、`financial_info`、`visa_history`、`business_plan`、`family_info` → 汇总为 `quote_price` + `visa_plan`（1年 / 4+1年） |
| **输出** | Survey 记录；报价金额确认 |
| **异常分支** | 问卷未回收 → Lead 状态保持 following；报价被客户拒绝 → 标记为 lost |
| **P0 映射** | `Survey` 实体 (P0 §1 Lead 阶段)；`Lead` 跟进记录 |
| **原型页面** | `leads-message/detail.html` 跟进 Tab |
| **可测** | **UI ⚠️** — 原型 leads 详情可查看跟进记录和基础信息，但没有独立的问卷发送/回收 UI；测试时需用"文档模拟步骤" |

---

### 阶段 C：签约（CONSULTING → CONTRACTED）

| 维度 | 内容 |
|------|------|
| **对应步骤** | Step 5 |
| **角色** | 行政书士 / 客户 |
| **动作** | 合同签署 + 收取定金（约 50%） |
| **输入字段** | `contract_signed = true`、`deposit_amount`（≈ quote_price × 50%） |
| **输出** | Lead 状态 → `signed`；触发转化（创建 Customer + Case） |
| **异常分支** | 客户放弃签约 → Lead 标记 lost；定金金额与报价不符 → 需确认 |
| **P0 映射** | `Lead.status = signed` → 转客户/转案件 |
| **原型页面** | `leads-message/detail.html` → 「转为客户」按钮、「转为案件」按钮 |
| **可测** | **UI ⚠️** — 转化按钮存在且可点击，但签约合同和定金收取无专用 UI；转化操作会跳转到 customers 或 case 创建页面 |

### 阶段 D：建案（→ S1 初始）

| 维度 | 内容 |
|------|------|
| **对应步骤** | Step 5 → 6 的桥接 |
| **角色** | 行政书士 / 助理 |
| **动作** | 创建案件：选择签证类型 + 申请类型、指定负责人/Group、关联客户 |
| **输入字段** | `case_type`（経営・管理）、`application_type`（认定/更新/变更）、`group`、`owner`、`customer_id`、`source_lead_id`、`due_date`、`amount` |
| **输出** | Case 记录创建，stage = S1；资料清单模板驱动生成 |
| **异常分支** | 跨组建案需填写原因 + 审计留痕；重复建案检测 |
| **P0 映射** | `Case` 实体 + `CaseParty` + `DocumentRequirement` 模板展开 |
| **原型页面** | `case/create.html` — 多步向导（模板选择 → 基础信息 → 关联人 → 资料清单确认） |
| **可测** | **UI ✅** — 可选择"経営・管理"模板卡片，填写申请类型/Group/负责人，走完向导确认建案，验证跳转到案件列表且出现新行 |

|------|------|

### 阶段 K：补正循环（S7 内闭环）

| 维度 | 内容 |
|------|------|
| **对应步骤** | Step 14 |
| **角色** | 行政书士 / 助理 |
| **动作** | 登记补正通知 → 收集补正材料 → 重新经过 Gate-B/C → 生成补正提交包 |
| **输入字段** | 补正截止日、补正材料项、Gate-B 校验结果 |
| **输出** | `SubmissionPackage(submission_kind=supplement)` 关联原始提交包 |
| **异常分支** | Gate-B 未通过 → 阻断补正包生成；欠款 + 补正 → 需风险确认；补正循环可多次（`supplement_count` 递增） |
| **P0 映射** | 案件保持 S7；`SubmissionPackage` + `CorrectionPackage` 概念；P0 §15.4 强规则 |
| **原型页面** | `case/detail.html` — 「登记补正通知」按钮 → 补正包状态显示 → 「提交补正包」按钮 |
| **可测** | **UI ✅** — correction 样本数据展示补正处理中状态；可验证补正通知登记 → 补正包提交全流程；日志记录补正次数 |

**关键校核点**：
- 补正循环在 S7 内闭环，不回退至 S2–S6（P0 冻结口径）
- 补正完成后只允许进入 `approved` 或 `rejected`
- 每次补正自动创建补正任务，带截止日

## 7. 业务文档 vs P0 关键偏差备忘

以下偏差在生成测试脚本时需特别注意，测试以 P0 口径为准：

| # | 偏差点 | 业务文档口径 | P0/原型口径 | 测试应遵循 |
|---|-------|-------------|-------------|-----------|
| Δ1 | 补正状态 | 独立状态 `NEED_SUPPLEMENT` / `SUPPLEMENT_PROCESSING` | S7 内闭环，不回退 | P0 |
| Δ2 | 资料上传 | 隐含直接上传文件 | 登记本地归档路径（`relative_path`） | P0 |
| Δ3 | 提醒天数 | "可配置" | P0 固定 180/90/30 天 | P0 |
| Δ4 | 收费字段 | 布尔位 `deposit_paid` | 缓存值 `_cached`，以 BillingPlan/PaymentRecord 为准 | P0 |
| Δ5 | Lead 阶段 | `CONSULTING` 混入 Case | Lead 独立实体，签约后转化才进入 Case | P0 |
| Δ6 | 入管审查中 | 区分 `APPLYING` / `UNDER_REVIEW` | 统一为 S7 | P0 |
