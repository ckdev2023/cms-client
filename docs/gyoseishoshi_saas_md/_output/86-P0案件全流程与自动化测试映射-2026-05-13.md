## 1. 主链路（咨询 → 归档）

与 [04 §0.1](../P0/04-核心流程与状态流转.md#01-p0-主链路) 一致：

```text
咨询录入 → 客户建档 → 案件创建 → 资料收集与审核 → 文书登记与确认
    → 提交前校验 → 正式提交 → 审理跟踪 / 补正 → 出结果 → 归档
```

**事务所实务理解**：前台多为「受付・线索管理 → 客户主档 → 一案一单（Case）」；中台在案件内驱动「资料一齐 → 文书齐备 → 校验通过 → 提交包锁定提交 → 结果登记 → 收尾归档」。P0 产品用 **管理层阶段 S1–S9** 与 **Gate-A/B/C** 把上述节奏固化在系统里。

---

## 2. 管理层阶段（S1–S9）

正式定义见 [03 §3.1](../P0/03-业务规则与不变量.md#31-案件阶段) / [04 §2](../P0/04-核心流程与状态流转.md#2-案件阶段s1s9)。

| 代号 | 阶段 | 进入条件（摘要） |
|------|------|------------------|
| `S1` | 已建档 | 已签约/已立案，主申请人、案件类型、负责人已明确 |
| `S2` | 资料收集中 | 资料清单已生成并发出至少一轮资料请求 |
| `S3` | 资料审核中 | 已有客户上传/登记资料，内部开始审核 |
| `S4` | 文书制作中 | **Gate-A 通过** |
| `S5` | 待校验 | **Gate-B 通过** |
| `S6` | 待提交 | 最新校验通过；复核满足（若启用） |
| `S7` | 已提交审理中 | **Gate-C 通过**且已生成提交包 |
| `S8` | 已出结果 | 已登记结果与结果日期 |
| `S9` | 已归档 | 结案并完成归档动作 |

**P0 明确不做（流程层）**：业务子步骤 `CaseWorkflowStep`、客户门户、系统内 Word/PDF 文书引擎等见 [04 §6](../P0/04-核心流程与状态流转.md#6-p0-明确不做流程层)。

## 3. 三道 Gate（与阶段的关系）

摘要见 [04 §0.3](../P0/04-核心流程与状态流转.md#03-gate-触发摘要)；硬性/软性明细见 [03 §4](../P0/03-业务规则与不变量.md#4-校验门槛gate规则)。

| Gate | 典型触发 | 通过后（摘要） |
|------|----------|----------------|
| **Gate-A** | `S3 → S4` 或首次登记关键文书前 | 可进入文书侧作业（S4） |
| **Gate-B** | `S4 → S5` 或主动进入校验 | 生成 `ValidationRun` |
| **Gate-C** | 「正式提交」 | 生成 `SubmissionPackage`，阶段 **→ S7** |

**重新校验**：资料版本、关系人、关键字段、关键文书、复核策略等变更后须重新跑校验——见 [03 §4.4](../P0/03-业务规则与不变量.md#44-重新校验触发条件)。**自动化测试应覆盖「前端以为可提交，但服务端因数据已变更而拒绝」的路径**（[04 §5](../P0/04-核心流程与状态流转.md#5-服务端执行口径ai--开发实现必须遵守)）。

## 4. 分阶段作业剧本（便于写用例场景）

以下内容对应 [04 §4](../P0/04-核心流程与状态流转.md#4-分阶段流程剧本)。

### 4.1 咨询转案件

线索 → 客户去重与合并 → 签约后建案：`Customer`、`Case`、`CaseParty`、资料清单模板展开。家族签证等需绑定扶养者/保证人为 `CaseParty`（[04 §4.1](../P0/04-核心流程与状态流转.md#41-咨询转案件)）。

### 4.2 资料收集与审核

- 资料项状态机：`not_sent → waiting_upload → uploaded_reviewing → approved`，以及 `revision_required`、`waived`、`expired`（[03 §3.2](../P0/03-业务规则与不变量.md#32-资料项状态)、[04 §4.2](../P0/04-核心流程与状态流转.md#42-资料收集与审核)）。
- P0 默认：**本地资料服务器登记**（`storage=local_server` + 相对路径），非 SaaS 存证文件体（[03 §2.3](../P0/03-业务规则与不变量.md#23-资料项与附件版本分离)）。

### 4.3 文书

- P0：**草稿 `draft` → 已确认 `final`**，系统登记名称与外链 `file_url`（[03 §3.3](../P0/03-业务规则与不变量.md#33-文书状态)、[03 §8](../P0/03-业务规则与不变量.md#8-文书规则)）。详细 UI 口径见 [文书中心](../P0/06-页面规格/文书中心.md)。

### 4.4 校验与提交

- `ValidationRun`：`pending` / `failed` / `passed`（[03 §3.7](../P0/03-业务规则与不变量.md#37-校验结果状态)）。
- `SubmissionPackage`：锁定引用版本，**不可覆盖**，补正须新包并关联原包（[03 §2.4](../P0/03-业务规则与不变量.md#24-提交包锁定与不可覆盖)、[04 §4.4–4.5](../P0/04-核心流程与状态流转.md#44-提交前校验与提交)）。

### 4.5 补正循环

保持在 **S7** 内：补正完成后重新 **Gate-B → Gate-C**，新提交包 `related_submission_id` 关联原包（[04 §4.5](../P0/04-核心流程与状态流转.md#45-审理跟踪与补正)）。

### 4.6 归档、续签提醒、收费

- 归档：S8 → S9（[04 §4.6](../P0/04-核心流程与状态流转.md#46-归档与续签提醒)）。
- 收费：P0 **默认不硬阻断提交**，欠款继续提交须**风险确认留痕**（[03 §4.3](../P0/03-业务规则与不变量.md#43-三道-gate)、[03 §6](../P0/03-业务规则与不变量.md#6-收费与欠款策略)）。

## 6. 自动化测试落地（本仓库）

### 6.1 收尾门禁（交付前）

与 AGENTS / 门禁一致：仓库根目录先 `npm run fix`，再 `npm run guard`。

### 6.2 按包的测试命令

| 包 | 命令 | 说明 |
|----|------|------|
| **server** | `npm run server:test` | `node --import tsx --test 'src/**/*.test.ts'` |
| **server** | `npm run server:guard` | 含 `lint`、`typecheck`、架构/迁移检查、**单测 + integration-pg:ci** |
| **admin** | `npm run test -w @cms/admin`（或进入 `packages/admin` 执行 `npm test`） | `vitest run --coverage` |
| **admin** | `npm run admin:guard` | 含 `typecheck`、`lint`、`test`、`build` |

本地全栈/队列（文书导出等）见仓库说明：`npm run local:dev`（server + admin + worker）。

### 6.4 写自动化用例时的推荐顺序

1. **固定断言**：阶段枚举、`ValidationRun` 状态、提交包锁定语义与 [03/04] 一致。  
2. **Gate 负例**：缺 `final` 文书、缺必交资料、校验过期后仍请求提交等。  
3. **补正**：仍在 **S7**，新 `SubmissionPackage` + 关联字段。  
4. **收费**：欠款 + 风险确认留痕（不默认阻断 vs 确认后可继续）。  
5. **集成**：`server:guard` 中的 PostgreSQL 集成任务会覆盖迁移与真实 DB 路径，改 schema 后必跑。
