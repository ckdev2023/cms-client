## 2. 已修复 P0

### P0-1 建案 Owner Picker 显示 fixture，提交触发 `CASE_OWNER_NOT_FOUND`

- **现象**：建案 Step3 「负责人」下拉只能看到 `铃木 / 田中 / 佐藤 / 山本 / 中村`
  等 fixture 名字；默认值为 `viewer.ownerId = "suzuki"`，提交时后端报：

  ```
  CASE_OWNER_NOT_FOUND: ownerUserId "suzuki" does not match any active user
  ```

- **根因**：`packages/admin/src/views/cases/CaseCreateView.vue` 仍在使用
  `getOwnerOptions(locale.value)`（fixture 数据 `OWNER_CATALOG`）+
  `withCurrentUserOwnerOption(...)`，并把 `viewer.ownerId`（fixture slug）作为默认值传给
  `useCreateCaseModel`。
- **修复**：改为消费真实组织用户数据，并把当前登录管理员的 UUID 作为默认负责人。
  关键 diff：

  ```ts
  // CaseCreateView.vue
  - import {
  -   getOwnerOptions,
  -   withCurrentUserOwnerOption,
  - } from "../../shared/model/useOwnerOptions";
  + import { toApiOwnerOption } from "../../shared/model/useOwnerOptions";
  + import { getActiveUserOptions } from "../../shared/model/useOrgUserOptions";

  - const ownerOptions = computed(() =>
  -   withCurrentUserOwnerOption(getOwnerOptions(locale.value), currentUser.value),
  - );
  + const ownerOptions = computed<CaseOwnerOption[]>(() =>
  +   getActiveUserOptions().map((u) =>
  +     toApiOwnerOption({ id: u.value, displayName: u.label }),
  +   ),
  + );
  + const defaultOwnerId = computed(() => currentUser.value?.id ?? viewer.ownerId);

  - defaultOwner: viewer.ownerId,
  + defaultOwner: defaultOwnerId.value,
  ```

- **验证**：走查中 owner picker 默认显示 `Local Admin`，
  最终 `POST /api/cases` 请求体 `ownerUserId = "00000000-0000-4000-8000-000000000011"`
  对应 `users.id`，HTTP 201 通过。

### P0-2 BMV 子类型 case 创建后「资料清单 0/0」

- **现象**：建案选择「经营管理（认定 4 个月）」子模板（Step1 卡片预览 18 项资料）后
  提交，案件创建成功但案件详情页：

  - 概览卡 `资料完成率 0%`、`0/0`
  - 「资料清单」tab 标签 `资料清单 0/0`
  - 切到 tab 内容仅有「资料登记清单」标题与空表格

- **根因**：

  1. `case_templates` 表只 seeds 了 3 行：`dependent_visa` / `work` /
     `business_manager_visa`（`packages/server/src/scripts/seedCaseTemplates.ts`）；
  2. 前端 `useCreateCaseModel` 把所选子模板 `templateId = "biz_mgmt_cert_4m"`
     原样作为 `caseTypeCode` 发送（`CaseAdapterWriteBuilders.ts L95`）；
  3. 服务端 `findActiveCaseTemplateByCaseType` 严格按 `case_type = $1` 查询，
     `biz_mgmt_cert_4m` 找不到任何 `case_templates` 行 → `{found: false}`；
  4. 回退到 legacy `TemplatesResolver` 也无对应 key → 返回空 checklist；
  5. 事务里 `insertDocumentItems(tx, orgId, caseId, [])` 写入 0 行。

- **修复**：

  - 在 `cases.template.repository.ts` 引入 BMV 子类型回退：
    若 `caseTypeCode` 属于 `isBmvCaseTypeCode(...)` 范围（与 migration 038
    `LIKE 'biz_mgmt%'` 对齐），先按原 code 精确查询，未命中再用
    `case_type = 'business_manager_visa'` 兜底。
  - 同步更新 backfill 脚本 `backfillCaseDocumentItems.ts` 的
    `BACKFILL_QUERY`：精确匹配 `t_exact` 不命中时 BMV 系列 LEFT JOIN
    `t_bmv (case_type='business_manager_visa')`，确保存量 case 也能补齐资料清单。
  - 新增单测：

    - `cases.template.repository.test.ts`：`falls back to business_manager_visa
template for BMV subtype codes` / `does not fall back for non-BMV unknown
codes`。

- **验证**：

  - 跑 `DRY_RUN=1 backfillCaseDocumentItems` 命中 23 个存量 case，
    包含本次走查创建的 `CASE-202605-0019`（`biz_mgmt_cert_4m`），10 项资料。
  - 实际执行 backfill：`23 updated, 4 skipped`。
  - 重新加载案件详情页，「资料清单 0/10」tab 下展示 10 项 BMV 资料
    （主申请人 3、事务所内部 2、扶养者/保证人 5）。

## 3. 新发现 P3（一致性 / UX 提示）

### P3-1 BMV 承接卡片：发送问卷后下一步提示与门禁脱节

- **现象**：「发送问卷」成功后，UI 提示文案：

  ```
  下一步：等待客户回收问卷
  ```

  但「生成报价」按钮立即可点击，且服务端 `generateBmvQuote()` 隐式把
  `questionnaireStatus` 从 `sent` 推进到 `returned`。
  即「等待客户回收问卷」对管理员来说没有可执行动作，
  和后端「点击生成报价 = 自动确认问卷已回收」的实际行为不一致。

- **风险**：管理员误以为需要等待客户在外部确认问卷，造成停滞；
  也会让规格 `04-页面规格 §Step3 questionnaire_returned` 的「确认回收」
  独立动作语义被覆盖。

- **建议**（待方案讨论）：

  - 方案 A：UI 增加「确认问卷已回收」独立动作，用户点击后状态再转 `returned`，
    同时 `generateBmvQuote()` 不再隐式推进；
  - 方案 B：保持现有自动推进语义，但把「等待客户回收问卷」改为「问卷已发送，
    可继续生成报价」并直接显示报价行为可执行。

### P3-2 案件详情概览卡 `资料完成率` 在 backfill 后 tab 与卡片仍需手动 reload

- **现象**：当通过 backfill 脚本补齐 `document_items` 后，
  已打开的案件详情页 tab 计数与「资料完成率」卡片仍展示 `0/0`，
  必须 `location.reload()` 才更新。

- **风险**：低；只在罕见的「先建案后 backfill」场景出现，
  正常建案流程在新链路下已不再依赖 backfill。

- **建议**：日常 P0 修复后无需立即处理；
  如后续支持模板热更新，可通过 SWR 失效时间或显式刷新按钮覆盖。
