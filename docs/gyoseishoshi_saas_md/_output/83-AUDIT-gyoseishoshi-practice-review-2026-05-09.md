## 0. 评审维度

| 维度 | 关注点 |
|---|---|
| **業務範囲** | 行政書士可受任業務 vs 弁護士独占業務（绝对边界）；P0 主战场是「在留資格 + 会社設立 + 翻訳証明」 |
| **文書類型** | 行政書士实务文書远不止「申請理由書」，含委任状 / 同意書 / 誓約書 / 翻訳証明書 / 報酬請求書 / 領収書等 |
| **法規遵守** | 行政書士法 §1-2（業務範囲）/ §9（業務帳簿）/ §10（守秘義務）；行政書士法施行規則 §11（職印）等 |
| **業務帳簿** | 行政書士法 §9 業務帳簿 = 2 年保存；契約関係文書 = 5 年保存；個人情報 = PIA 級别管理 |
| **入管実務** | 申請書（入管定式 PDF） vs 添付書類（行政書士作成 docx）；2024 育成就労法 / 永住要件改正等条文版本追跡 |
| **PII 保護** | パスポート / 在留カード / 戸籍 / 住民票 — 字段级权限；個人情報保護法 + GDPR 类似要求 |
| **責任分担** | 行政書士本人对最终文書负民事/行政责任（行政書士法 §1-2）；平台模板 = 参考素材，不替代行政書士判断 |

## 2. P0 评审条目（必改 / 已落地）

### P0-1 申請書（入管定式 PDF）必须明确不在 D3 范围

- **症状**：80 RFC 通篇只讲「申請理由書 / 身元保証書 / 雇用契約書サマリ」等 docx 模板，**完全没提行政書士最核心的工作之一 — 入管申請書**
- **行业依据**：在留資格申請的「在留資格認定証明書交付申請書」「在留資格変更許可申請書」「在留期間更新許可申請書」「永住許可申請書」均为入管局发布的**定式 PDF 表格**（带固定字段的可填表 PDF），行政書士实务中不能用 docx 模板替代——这是入管局**强制使用样式**
- **影响**：如果产品/法务误以为 P1 docx 模板覆盖了"入管申請"，会出现「客户申请被拒（格式不符）」的事故。范围必须早于评审就钉死
- **修改建议**：80 §11「不做的事」明确加一行「申請書（入管定式 PDF）= P3 单独 RFC，需 PDF 表格自动填表能力（pdf-lib + AcroForm）」；82 §4.1 必备模板矩阵的 doc_type 列**禁止**包含 `application_form_*`
- **落点**: ✅ 已回填 80 §11 / 82 §4.1
- **关联 v1 字典**: SPEC-081 §0 增加一行「**Schema 不覆盖 PDF AcroForm 字段映射**」

### P0-2 委任状（共通 doc_type）— 每案必备，缺失则模板矩阵不完整

- **症状**：82 §4.1 必备模板矩阵列出 `dependent_visa / work / business_manager` 三个 caseType 的 7 份模板，**完全没有「委任状（依頼者 → 行政書士の委任）」**
- **行业依据**：
  - 行政書士法 §1-2 — 行政書士の業務は本人または法定代理人からの依頼に基づく
  - 实务规范 — 任何受任案件第一份产出都是「委任状」（依頼者本人押印 + 行政書士事務所控え保管）
  - 入管申請時、行政書士が提出代行する場合は「申請取次行政書士」資格 + 申請取次の届出 + 委任状必携
- **影响**：所有事务所 P1 上线后第一天就会发现「为什么没有委任状模板」
- **修改建议**：82 §4.1 矩阵新增 `caseType=common`（跨 caseType 共通）`doc_type=power_of_attorney`，至少 1 份 published；上下文 schema 新增 `customer.sealRegistrationOrSignature`（自署 / 印鑑）
- **落点**: ✅ 已回填 82 §4.1 + 81 §3 (新增 `case.delegationScope` / `case.delegationStartDate`)

### P0-3 在留カード氏名・英文表記の欠落（Critical for 入管文書）

- **症状**：81 §2 customer.* 字段表只有 `fullName`（漢字）/ `fullNameKana`（カナ），**缺英文表记 + 在留カード公式記載**
- **行业依据**：
  - 入管申請書は「氏名は在留カードに記載されている英字綴り」を要求
  - 在留カード上の表記は **アルファベット（パスポート綴り）+ 漢字（漢字圏出身者のみ任意併記）**
  - 中国系客户の場合、漢字「王」は日本漢字「王」と簡体字「王」が同形だが「劉」/「刘」、「張」/「张」のような差异が頻繁；在留カード上の漢字表記は**日本漢字統一**
  - 通称名（在留カード or 住民票登録の通称）は本名と別物、書類分けが必要
- **影响**：申請理由書 / 身元保証書で氏名が在留カードと不一致 → 入管審査で形式不備差戻し
- **修改建议**：customer.* 新增 4 字段
  - `customer.fullNameEn` (string, required, ISO 拉丁拼写、パスポート綴り)
  - `customer.fullNameOnResidenceCard` (string, applicable when 在日, 在留カード公式記載)
  - `customer.fullNameAlt` (string, optional, 通称名)
  - `customer.familyNameRomaji` / `customer.givenNameRomaji` (string, パスポート単独綴り)
- **落点**: ✅ 已回填 81 §2

### P0-4 中国系客户 — nationality 細分 + 漢字処理

- **症状**：81 §2 `customer.nationality` 只是 ISO 3166-1 alpha-2，**未涵盖大陸 / 台湾 / 香港の区分**与漢字氏名の処理規約
- **行业依据**：
  - 行政書士事務所の中国系顧客が顧客総数の **40-60%**（特に経営管理 / 家族滞在）
  - 入管申請書の「国籍・地域」欄は「中華人民共和国 / 台湾 / 香港 / マカオ」の 4 区分（中国大陸は CN、台湾は TW、香港は HK、マカオは MO）
  - 漢字処理：簡体字（中国大陸） vs 繁体字（台湾 / 香港） vs 日本漢字統一の規則（在留カード = 日本漢字統一、戸籍提出書 = 出身地原文）
- **影响**：「国籍」表記ミスで入管差戻し（経験ベース：年間 5-10% の中国客户案件）
- **修改建议**：
  - `customer.nationality` 値域明確化：ISO 3166-1 alpha-2 + 特殊コード `CN` (中国大陸) / `TW` (台湾) / `HK` (香港) / `MO` (マカオ) 区分
  - `customer.nationalityJP` 派生映射表特殊处理: CN→「中華人民共和国」/ TW→「台湾」/ HK→「中華人民共和国（香港）」
  - 派生 `customer.nameScript`: enum<`japanese_kanji` | `simplified_chinese` | `traditional_chinese`>，模板按需引用
- **落点**: ✅ 已回填 81 §2 + 派生字段表

### P0-5 申請理由書叙述性内容 — narrative 字段族缺失

- **症状**：81 整体 schema **没有自由文本字段**。但「申請理由書」实务中除了变量填充，**还需要 50-200 字案件特殊事情の自由叙述**（如「家族滞在 — 子女学業の都合で配偶者と離れて暮らす理由」）
- **行业依据**：
  - 申請理由書 = 行政書士の核心成果物
  - 入管審査官が最も重視するのは「定型変数」より「個別事情の説明」
  - 模板に `{{narrative.applicationReason}}` がないと、行政書士は出力後 docx を再編集 → 二重作業
- **影响**：用户拿到的 docx 只有变量填充、缺关键叙述部分 → 行政書士需要再开 Word 编辑 → D3 价值大打折扣
- **修改建议**：新增 `narrative.*` 字段族
  - `narrative.applicationReason` (string, optional, 申請理由 free text)
  - `narrative.familyBackground` (string, optional, 家族構成 / 関係性叙述)
  - `narrative.businessPlan` (string, optional, 事業計画 / 経営方針叙述)
  - `narrative.additionalRemarks` (string, optional, 補足事項)
  - 前端 `CaseFormFinalizeModal` 提供富文本/多行编辑入口
- **落点**: ✅ 已回填 81 新增 §11 「narrative 字段族」+ 80 §3.2 加 narrative 步骤

### P0-6 行政書士印影 = 法律要求（行政書士法施行規則 §11）

- **症状**：80 §4.5「不做服务端 PDF 转换 / 不嵌入字体 / 不嵌入印影 PNG」OK，但**没在文档与产品 UX 明确告知行政書士印影是法律要求**
- **行业依据**：
  - 行政書士法施行規則 §11 — 行政書士は業務上作成する書類に職印を押印しなければならない
  - 職印 = 都道府県行政書士会へ届出済みの印影（法定印鑑）
  - P1 阶段不内嵌 PNG OK，但 admin UX **必须**在 export 后提示「请自行加盖職印」+ 模板 docx 末尾留印影位置占位
- **影响**：用户拿到 docx 不盖印就提交 → 入管局形式不備差戻し → 法律风险
- **修改建议**：
  - 80 §4.5 加章节「印影法律要求」+ 模板末尾必有「行政書士　印」文字+空白印影位置
  - admin UX：export 完成 toast「请自行加盖職印 / 自署」
  - 82 §3 审核维度加「模板末尾是否有印影位置占位」
- **落点**: ✅ 已回填 80 §4.5 + 82 §3.1

### P0-7 模板法律责任分担 — 必须明确「最终责任在行政書士本人」

- **症状**：82 §3 审核流程提到「法务签字」，但**没明确平台 vs 事务所的法律责任分担**
- **行业依据**：
  - 行政書士法 §1-2 — 行政書士の作成書類は本人責任（業務代理人責任）
  - 「平台提供的模板」≠「行政書士が確認した文書」；行政書士法上、最终署名 + 押印した行政書士本人が法律責任を負う
  - 同业平台（例：legalon / cloud-sign / freee 行政書士）約款：模板提供方仅承担「テンプレート作成過失」級别；客户最终文書责任在事务所
- **影响**：未明确时，万一某模板因条文过期导致客户被拒否，事务所可能向平台索赔，纠纷高
- **修改建议**：82 新增 §10「法律免責 / 責任分担」
  - 平台责任：模板素材提供 + 法务定期复核
  - 事务所责任：模板使用前自行确认 + 出具最终文書责任
  - admin UX：首次使用 D3 必须勾选「我已了解模板仅供参考，最终责任由本所行政書士承担」
  - ToS / 利用規約 同步加条款
- **落点**: ✅ 已回填 82 §10

### P0-8 PII 字段级权限（passport / residence card 不是所有 staff 都能看）

- **症状**：现有 RLS 是 org 级隔离，**没有字段级权限**。但 `customer.passportNo` / `customer.residenceCardNo` 是高敏 PII
- **行业依据**：
  - 個人情報保護法 §3 — 取扱事業者は必要最小限の範囲で個人情報を取り扱うべき
  - 行政書士事務所内 staff 区分：行政書士本人 / 補助者（法定登録あり） / 事務員（一般 staff） / 派遣者
  - パスポート / 在留カード番号 = 「特定個人情報」相当の保護要求；補助者以下のアクセスは記録必要
- **影响**：D3 渲染时如果一般 staff 触发了「生成包含 passport」的 export，没访问审计 → 个人情报保护法违反风险
- **修改建议**：
  - `users.role` 已存在；新增 `users.can_view_pii: boolean`（默认 false，仅 owner / 主任行政書士 = true）
  - renderer 在加载 RenderContext 时，按 `ctx.user.can_view_pii` 决定是否填入 passport / residence card / 戸籍関連字段；缺失则 preflight 抛 `pii_access_denied`
  - 所有 PII 字段的访问写入审计 log（audit_logs 已有，扩展 `accessed_fields` 数组）
  - 81 §13 PII 字段级权限分类
- **落点**: ✅ 已回填 81 新增 §13

## 3. P1 评审条目（强建议）

### P1-1 客户確認 = 案件 phase，不是 generated_document state

- **症状**：80 §0.3「draft → final → exporting → exported」流程**缺了「行政書士起案 → 客户確認 → 修改循環」**
- **行业实务**：行政書士事務所典型业务流程：①受任 → ②資料収集 → ③**起案 (草稿)** → ④**客户確認**（修正循环 1-N 次）→ ⑤定稿 → ⑥**事務所内決裁** → ⑦印影付与 → ⑧客户最终签字 → ⑨提出
- **修改建议**：明确「客户確認」属于 **case phase**（不是 generated_document status）；案件状态机文档 04 加补充；generated_document 仅承载渲染产物，不承载 review workflow
- **落点**: ✅ 已回填 82 §1 加流程图说明 + 待回灌到 04-核心流程

### P1-2 supporter.statusOfResidence applicable 时必填

- **症状**：81 §4 supporter.statusOfResidence 写「在日支援者必填」但未具体定义 applicableWhen
- **行业实务**：扶養者 = 永住者 / 日本人配偶者 / 経営管理 / 技人国等异なる在留資格 → 家族滞在許可審査で「扶養者の在留資格 + 在留期限」是核心審査項目
- **修改建议**：81 §4 supporter.statusOfResidence 改 required + applicableWhen=`supporter.fullName != ''`
- **落点**: ✅ 已回填 81 §4

### P1-3 翻訳証明 — 行政書士の核心業務、文書渲染要预留架构

- **症状**：D3 完全没考虑「翻訳証明書」（戸口本 / 出生証明書 / 婚姻証明書 → 日本語訳 + 訳者証明）
- **行业依据**：中国系客户必备文書；行政書士事務所收入の 15-25%（特に家族滞在 / 結婚ビザ案件）
- **修改建议**：P1 不实施翻訳証明（涉及人工翻译 + 翻訳者署名），但 **caseType / doc_type 枚举要预留** `doc_type=translation_certificate`（empty implementation 即可）；P2 RFC 加 OCR + 自動翻訳 + 訳者証明嵌入
- **落点**: ✅ 已回填 80 §11

### P1-4 業務帳簿保管期限 = 2 年（行政書士法 §9）

- **症状**：82 §8 storage 备份策略没提保管期限的法律要求
- **行业依据**：
  - 行政書士法 §9 — 業務帳簿（受任記録）保存 2 年
  - 戸籍関連文書の正本 — 1 年保管
  - 法人関連文書 — 5 年保管
  - 完全削除前需確認「現在進行中案件 / 紛争 / 監査対応」
- **修改建议**：document_templates 表与 generated_documents 表加 `retention_policy: enum<2y, 5y, indefinite>`；定期清理脚本按 retention 自动归档（不删除）
- **落点**: ✅ 已回填 82 §8

### P1-5 入管法令版本字段 — 模板老化追跡

- **症状**：模板 docx 内容引用了入管法条文（如「在留資格認定証明書 (留学) — 出入国管理及び難民認定法 §7-2」）；条文更新后模板老化没有跟踪机制
- **行业实务**：2024 年「育成就労法案」「永住要件改正」等持续在变；模板内文引用必须**显式标注法令版本**
- **修改建议**：
  - `document_templates.references_law_version: jsonb` — 例：`{"immigration_law": "2024-04-01", "guideline": "2024-06-15"}`
  - 法务季度复核必查项：本模板引用法令是否更新
- **落点**: ✅ 已回填 82 §3.1 + 82 §6 季度复核

### P1-6 報酬請求書 / 領収書 — 行政書士財務文書类型

- **症状**：D3 完全没覆盖「報酬請求書」「領収書」「業務報告書」类财务/业务文書
- **行业实务**：
  - 着手金請求 → 中間報告書 → 完了報告書 + 報酬請求書 → 領収書（印紙税要求あり）
  - 国税庁要求：消費税適格請求書（インボイス制度 2023.10〜）
  - 同业 SaaS（freee / マネーフォワード行政書士）已支援
- **修改建议**：P1 不实施财务文書，但 **架构预留** `case_type=common, doc_type=invoice / receipt / business_report`；P2 RFC 加インボイス制度対応
- **落点**: ✅ 已回填 80 §11 + 82 §4.1

### P1-7 案件日期三者：作成日 / 申請日 / 提出日

- **症状**：81 §7 today.* 只有渲染时间戳；但行政書士实务的「申請書類の日付」要区分
- **行业实务**：
  - 作成日 (`document_creation_date`) — 文書作成した日
  - 申請日 (`application_date`) — 入管に申請する予定日
  - 提出日 (`submission_date`) — 実際に提出した日
- **修改建议**：81 §3 case.* 加 `case.applicationDate` / `case.submissionDate`（optional）；今 today.* 仍然作为渲染时点的兜底
- **落点**: ✅ 已回填 81 §3 (附「日期三者」说明)

### P1-8 同意書 / 誓約書 / 念書 — 共通模板要兜底

- **症状**：82 §4.1 必备模板矩阵只有 caseType-specific 模板；缺「跨案件共通」类
- **行业实务**：
  - 個人情報取扱同意書（個人情報保護法対応）
  - 申請内容真実性誓約書
  - 連絡先・住所変更届
  - 这些是**任何 caseType 都可能用**的共通模板
- **修改建议**：82 §4.1 矩阵加「caseType=common」一行，至少 3 份共通模板（同意書 / 誓約書 / 連絡先変更届）
- **落点**: ✅ 已回填 82 §4.1

### P1-9 「会社設立」「許認可」类业务范围扩展
