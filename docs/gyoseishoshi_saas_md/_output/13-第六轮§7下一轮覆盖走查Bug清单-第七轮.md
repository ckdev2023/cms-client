## 1. P0 — 业务真相源缺失（拒签结案无法区分成功失败）

### BUG-117 [P0][API] `POST /api/cases/:id/phase-transition` 不接受 closeReason，CLOSED_FAILED 落地后 close_reason / resultOutcome / stage 全空

- **位置**：
  - `packages/server/src/modules/core/cases/cases.controller-bodies.ts:87`：`export type PhaseTransitionBody = { toPhase: unknown };`
  - `packages/server/src/modules/core/cases/cases.controller.ts:363-378`：仅读 `body.toPhase`
  - `packages/server/src/modules/core/cases/cases.service.ts:2267-2302`（`executePhaseTransitionUpdate`）：UPDATE SQL 完全不写 `close_reason / result_outcome / stage`
- **业务规范**（MemPalace P1 `01-经营管理签扩展范围与落地计划.md#M9 结案与异常兜底`）：
  > - `S9` 仍作为统一归档阶段
  > - **通过 `result_outcome / close_reason` 区分成功与失败**
- **现象**（端到端 reproduction）：

  ```bash
  TOKEN=$(cat /tmp/cms_r7_token.txt)
  NEW=$(curl -s -X POST .../api/cases -d '{"customerId":"...","caseTypeCode":"biz_mgmt_4m",...}' | jq -r .id)
  for TO in CONTRACTED WAITING_MATERIAL MATERIAL_PREPARING REVIEWING APPLYING UNDER_REVIEW; do
    curl ... /api/cases/$NEW/phase-transition -d "{\"toPhase\":\"$TO\"}"
  done
  curl ... /api/cases/$NEW/phase-transition -d '{"toPhase":"REJECTED"}'         # 201
  curl ... /api/cases/$NEW/phase-transition -d '{"toPhase":"CLOSED_FAILED"}'    # 201, close_reason=null
  curl ... /api/cases/$NEW
  # → phase=CLOSED_FAILED stage=S1 closeReason=null resultOutcome=null
  ```

- **DB 验证**：
  ```sql
  SELECT id, stage, business_phase, close_reason, result_outcome
    FROM cases WHERE id='3ede349e-3220-440c-8815-854fad48e81d';
  -- S1 | CLOSED_FAILED | (null) | (null)
  ```
- **对照旧路径**（仅供参考）：`cases.service.ts:1867-1891`（`transition`，老 stage 入口）依赖 `assertCloseReasonForNonCompletionArchive`（cases.service.ts:929-934）强制 `S?→S9` 必须传 closeReason；新 phase 入口完全绕过这个口径。
- **业务影响**：
  - **统计 / 风控完全失真**：BI 看板"成功率 vs 拒签率"、续签提醒模块、客户 LTV 全部依赖 `result_outcome` 与 `close_reason`，phase-only 路径下两字段永远 null。
  - **CLOSED_SUCCESS 与 CLOSED_FAILED 在 UI 之外不可区分**（仅 phase 枚举不同，但 close_reason / result_outcome 两个真相源全空）。
  - **审计回溯失败**：监管 / 内审来追问"为何此案件归档为失败"时，系统无 close_reason 可答。
- **建议修复**：
  1. `cases.controller-bodies.ts`：`type PhaseTransitionBody = { toPhase: unknown; closeReason?: unknown; resultOutcome?: unknown; }`。
  2. `cases.controller.ts:phaseTransition`：传入 `closeReason / resultOutcome` 给 service。
  3. `cases.service.ts:transitionPhase`：当 `toPhase === 'CLOSED_FAILED'` 时强制 closeReason 必填（参考 `assertCloseReasonForNonCompletionArchive`）；当 `toPhase === 'CLOSED_FAILED'` 时 SQL UPDATE 写入 `close_reason = $X, result_outcome = coalesce($Y, 'failure')`；当 `toPhase === 'CLOSED_SUCCESS'` 时强制 `result_outcome = 'success'` 且 closeReason = null。
  4. **必要时同步落 stage（参考 BUG-116）**：终态 phase 时联动 `stage = 'S9'`，确保单元一致。
  5. 补 `cases.phase-transition-closed-failed.focused.test.ts` 真跑 PG 验证 close_reason / result_outcome 落地。
- **优先级**：**P0**（业务真相源缺失，影响所有下游统计）。
- **关联**：BUG-116（stage 不联动）共因，建议同一 PR 修。

- **建议修复**：
  1. `main.ts`：抽出一个 `applyDocumentTitle(route)` helper（沿用现有 logic）；
  2. `router.afterEach((to) => applyDocumentTitle(to))`；
  3. 新增对 locale 的 watch：
     ```ts
     watch(
       () => i18n.global.locale.value,
       () => {
         const current = router.currentRoute.value;
         applyDocumentTitle(current);
       },
     );
     ```
     注意 `i18n.global.locale` 在 Vue I18n v9 是 `WritableComputedRef<string>`，watch source 用 `() => i18n.global.locale.value`。
  4. 新增 `main.title-locale-sync.test.ts`：mount 后切 locale，断言 `document.title` 文案变化。
- **优先级**：P2（次要 UX；浏览器 tab title 不影响业务流，但是国际化承诺的一部分）。
- **关联**：BUG-112（Tasks i18n 已修）；BUG-119 / 124（同样属于 i18n 三语切换体系，统一在「locale change → re-emit」收口）。
