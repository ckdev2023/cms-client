# Admin H5 响应式走查（第一轮 / R-H5-1 — 移动端 viewport 适配现状清查）

> 生成日期：2026-05-05
>
> 命题：Admin Web 仅按桌面 1280–1440 设计，无系统性 H5 适配。本轮在 chrome-devtools-mcp
> 控制下把 viewport 切到 **iPhone 12 (390×844, DPR=3, mobile, touch)**，再覆盖
> **iPhone SE (375×667)** 校验最坏情况，对 8 个代表页面逐项采样横向溢出、文本截断、
> 表格塌陷、tab/header/抽屉等问题，量化并归类。
>
> 工具：chrome-devtools-mcp（list_pages / navigate_page / emulate / take_snapshot /
> take_screenshot / evaluate_script / list_console_messages / click）
>
> 数据集：Local Demo Office（admin@local.test / Admin123!）；UI locale = ja-JP；
> 站点 `http://localhost:5173/`。

## 0. 总结

### 0.1 一句话结论

**Admin Web 当前没有任何系统性 H5 适配，从顶部导航到案件详情、创建向导、设置页几乎
所有核心路径都在 390×844 下出现横向溢出（`scrollWidth - clientWidth ≥ 14 ~ 383px`）、
文本被字符级截断、表格塌陷或 grid 列宽固化。** 走查覆盖 8 个代表页面，发现 **P0×4 / P1×5 /
P2×4 / P3×3 / P4×2 / 体系性 ×3** 问题，**没有一个核心页面在 H5 下属于 "可用" 状态**。
此外抓到 **2 处硬编码中文 `aria-label` 在 ja-JP locale 下原样泄露**（属 i18n 兼回归），
以及 console 16 条 intlify 空 key 警告（疑似 R42-A 残留 / 其他 builder fallthrough）。

### 0.4 体系性观察（最关键 3 条）

1. **「整个 admin web 没有 mobile-first / responsive breakpoint 体系」** — 全仓搜索
   `@media (max-width:`、`@media (min-width:` 仅在 `shell.css` 出现 4 处（640/1024/1023），
   且仅作 **TopBar 局部小调** 与 **侧边栏显隐切换**；没有一个 view 文件（`CaseDetailView.vue` /
   `CaseListView.vue` / `BillingView.vue` / `SettingsView.vue` / 模板卡片 grid 等）配置
   H5 断点。**任何 desktop grid / 多列布局都会以固化列宽外溢出现在 H5。**

2. **「`ui-page-header` + 多 action button 的横向布局没有 H5 收纳策略」** — `案件詳細`
   页 `情報を編集 / ZIP エクスポート / ステータス遷移` 三个按钮总宽 `108+142+134+gap*2=404px`，
   超过 iPhone 12 视口 390px；`flex-wrap: nowrap` 强制不换行；title 被压到 125px 后字符级
   断成 4 行（h1）。**应在 H5 下：① actions 折叠到 `…` 菜单 / ② 用 sticky bottom action bar /
   ③ wrap 到次行 + title min-width 0 释放空间。**

3. **「TopBar 同时挂 5 个按钮 + 1 个搜索 + 1 头像，没有 H5 折叠策略」** — `topbar-actions`
   含 locale 下拉 / `相談を新規作成` / `案件を新規作成` / `ログアウト` + 头像，**5 项
   同时常驻**。`@media (max-width: 1023px)` 的局部规则只是把搜索框换行，没有对 actions
   做任何收纳。**应该把 `相談を新規作成` / `ログアウト` / 头像菜单收到右上汉堡或用户菜单。**

### 0.5 本轮最严重的根因 1 处（P0×3 共因）

```883:886:packages/admin/src/views/cases/CaseDetailView.vue
.case-detail-view {
  display: grid;
  gap: 20px;
}
```

- `display: grid` 隐式 `grid-template-columns: 1fr`，但子元素 `ui-page-header`（含 actions
  flex-row nowrap）、`case-detail-view__tabs`（flex-wrap wrap 但子 `flex-shrink: 0`）、
  `case-detail-view__panel`（含 sidebar/aggregate 内表）的 **min-content** 全部 ≥ 545px。
- 浏览器把 `1fr` 解析为 `minmax(auto, 1fr)`，其中 `auto` 取 max-content min-width，
  最终列宽固化为 **545.672px**（与视口 375/390 无关）。
- 改法：`grid-template-columns: minmax(0, 1fr)` 或 `display: block`，**并在所有内层
  flex/grid 容器上加 `min-width: 0`**。
- 这一行 CSS 是 `案件详情` 全部 5 个 tab（overview/forms/log/timeline/billing-tab）的
  共因，修了它能一次性解决 P0×1 + P1×3 + P2×1。

### 1.3 案件详情 `/#/cases/5d38aaac…?tab=overview`（**最严重，P0**）

**root cause 量化**（chrome-devtools-mcp `evaluate_script`）：

```javascript
// .case-detail-view 元素的 grid layout
{
  width: 350,                     // 容器宽度（视口 390 减 padding）
  scrollWidth: 546,                // 内容必需宽度
  gridTemplateColumns: "545.672px" // ← 关键：computed 之后变成固定 545.672px
}
```

**全链路尺寸**（从 `<html>` 一路下钻到 `.case-detail-view`）：

```
html       cw=390  sw=565
.app-shell w=390   sw=566
main       w=390   sw=566
.content-inner w=390 sw=566
.case-detail-view w=350 sw=546   ← 这里开始溢出
.case-detail-view__tabs w=546 (flex-wrap: wrap, overflow-x: auto)
```

**h1 标题压缩**：

```javascript
{
  text: "R23-AUDIT-TITLE-TEST",
  fontSize: "28px",
  width: 125,    // ← 仅 125px，被同行 actions 挤压
  height: 134,   // ← 高度 4 行
  wordBreak: "normal",
  overflowWrap: "normal"
}
```

虽然 CSS 没显式 `word-break: break-all`，但 `R23-AUDIT-TITLE-TEST` 的连字符 `-` 是
**有效断行点**，浏览器在 125px 宽度下按 `R23- / AUDIT- / TITLE- / TEST` 拆 4 行。

**actions 总宽超视口**：

```javascript
{
  width: 405,
  flexWrap: "nowrap",
  items: [
    { text: "情報を編集", width: 108 },
    { text: "ZIP エクスポート", width: 142 },
    { text: "ステータス遷移", width: 134 }
  ]
  // total = 108 + 142 + 134 + (10 + 10) gap = 404
}
```

**tabs 行为**：

```javascript
// 10 个 tab，第 1 行 6 个延伸到 right=520（超出 390），第 2 行 4 个
[{ text: "概要", left: 20, right: 46 },
 { text: "提出前チェック ブロック 1", left: 66, right: 232 },
 { text: "必要書類 0/1", left: 252, right: 343 },
 { text: "タスク", left: 363, right: 402 },        // ← 部分超出
 { text: "基本情報", left: 422, right: 474 },      // ← 完全在视口外
 { text: "文書", left: 494, right: 520 },          // ← 完全在视口外
 // ↓ 第 2 行
 { text: "期限", top: 487, ... },
 { text: "請求", top: 487, ... },
 { text: "連絡記録 1", top: 487, ... },
 { text: "ログ", top: 487, ... }]
```

→ 第 1 行有 3 个 tab（`タスク` / `基本情報` / `文書`）落在 390 视口外，但因为 `flex-wrap: wrap`
没把它们换到第 2 行，原因是父容器 `.case-detail-view__tabs` 自身 `width=546` 已经是固化的
（被 `.case-detail-view` 的 `545.672px` 列宽决定），所以 wrap 也不会触发到 390 这个值。

**修复建议**：见 §3.1。

来自 `<input type="date">` 的浏览器/扩展内置 UI（多半是 Chrome 简体中文按系统语言渲染），
受 page `<html lang="zh-CN">` 或浏览器语言驱动，与应用 i18n 框架无关。**修复**：
顶层 `<html lang>` 应跟随当前 locale 切换（ja-JP locale → `lang="ja"`），让浏览器 date
input 文案与页面语言保持一致。

#### R-H5-W1 · `[intlify] Not found '' key in 'ja' locale messages.` 共 16 条

console 实测：

```
[warn] [intlify] Not found '' key in 'ja' locale messages.
[warn] [intlify] Fall back to translate '' key with 'zh-CN' locale.
[warn] [intlify] Fall back to translate '' key with 'zh' locale.
[warn] [intlify] Not found '' key in 'zh' locale messages.
```

每组 4 条 × 4 = 16 条。空字符串作为 i18n key 被传入 `t("")`，疑似某个 builder
（可能是 R42-A 修复后 `suffixKey` 默认 `""` 时被 resolver 直接 t 调用）的派生问题。
**待补查**：在 `CaseTimelineTextResolver.resolveKeyParam` 处加 `if (key) i18n.t(key) else return null`
保护。

## 5. 体系性沉淀（编译式入库候选 — 待回灌到权威文档）

本轮发现的 18 条 H5 问题与 1 处共因 CSS bug 指向 4 条原则：

1. **「Admin Web 必须有 mobile-first 设计 token + breakpoint 体系」（新原则）**
   建议沉淀到 `docs/gyoseishoshi_saas_md/00-原则与约定/ADR-009-h5-responsive-strategy.md`，
   明确：① 4 个 breakpoint（sm/md/lg/xl）② 关键 view 的 H5 验收清单 ③ 通用 `<ResponsiveTable>` /
   `<ResponsiveActions>` 组件契约。

2. **「`display: grid` 与 `1fr` 的 min-content blowup 必须显式 `minmax(0, 1fr)`」（CSS 共因原则）**
   本轮 R-H5-1 的根因是 `1fr` 默认变成 `minmax(auto, 1fr)` 时，`auto` 取 max-content
   于是被内部 `flex-wrap: nowrap` 的 actions 撑大。**所有 grid 容器都应用 `minmax(0, 1fr)`**，
   避免子项 min-content 倒推父列宽。

3. **「PR review checklist 必须包含 H5 viewport 走查」（流程原则）** — 本轮发现的所有
   P0/P1 问题在 desktop（≥ 1280px）下完全不可见；只有切到 H5 viewport 才暴露。
   建议把 `Cypress / playwright`（或 chrome-devtools-mcp 脚本）的 H5 走查写入 PR check：
   `npm run test:e2e:h5` 在 390×844 下挂载关键页面，断言 `scrollWidth ≤ clientWidth + 2`。

4. **「i18n 走查应该常态化覆盖 ja/en/zh 三语 + aria-label」（i18n 原则加固）** —
   R-H5-I1（中文 aria-label 在 ja-JP 下泄露）说明：仅检查可见 UI 文案不够，**aria-label /
   placeholder / title 等 attribute 也必须经 i18n**。建议在 `i18n-key-consistency.test.ts`
   里加一条扫描规则：`grep -nE 'aria-label="[^"]*[\u4e00-\u9fff]'` 在 packages/admin/src 下
   不应有命中。
