# 试点页面迁移顺序建议

> Foundation 骨架（design tokens / app shell / shared primitives）完成后，将 8 个原型模块迁移到 `packages/admin` 的推荐顺序与依据。

## 评估维度

| 维度 | 说明 |
|------|------|
| UI 区块数 | split-manifest.json 中 `sections` 条目数量 |
| 脚本数 | `scripts` 条目数量，反映交互复杂度 |
| 需全新构建 | 原型缺失、需从零构建的区块占比 |
| 跨模块依赖 | 该模块对其他业务模块数据的读/写依赖 |
| 共享组件需求 | 迁移该模块后需要新增到 shared/ui 的通用组件 |

## 模块复杂度对比

| # | 模块 | 区块数 | 脚本数 | 数据文件 | 需全新构建比例 | 跨模块依赖 |
|---|------|--------|--------|----------|---------------|-----------|
| 1 | dashboard | 6 | 1 | 1 | 低 | 读：cases, billing, tasks, customers |
| 2 | customers | 6 | 4 | 1 | 低 | 无 |
| 3 | leads-message | 5 | 2 | 1 | 中（列表重构） | 无 |
| 4 | case | 7 | 2 | 1 | 中 | 读：customers, templates |
| 5 | tasks | 8 | 5 | 2 | 高（筛选/详情/提醒/弹窗全新） | 读：cases |
| 6 | billing | 9 | 5 | 2 | 高（回款/计划/风险全新） | 读写：cases, audit |
| 7 | documents | 10 | 5 | 2 | 极高（摘要卡/审核/waive/引用全新） | 读写：cases, audit |
| 8 | settings | 11 | 8 | 2 | 极高（几乎全新） | 提供：Group/OrgSetting |

## 推荐迁移顺序

### Wave 1 — 壳层验证

#### ① Dashboard（仪表盘）

**理由**：交互最轻量，仅包含聚合卡片和只读列表，无 CRUD、无弹窗、无批量操作。最适合验证 Foundation 的 AppShell + TopBar + SideNav + Card + PageHeader + Chip 是否能稳定撑起完整页面。

**关键产出**：
- 确认 shell slot / content area 布局在真实页面下表现正常
- 验证 design tokens 在多状态卡片场景下的可用性
- 定义 `features/dashboard/` 目录结构范式供后续模块参照

**需新增 shared 组件**：SegmentedControl（scope 切换）、SkeletonBlock（骨架屏）

**预计工作量**：小

---

### Wave 2 — 标准列表范式

#### ② Customers（客户）

**理由**：定义标准 CRUD 列表页范式（表格 + 搜索 + 筛选 + 批量操作 + 新建弹窗 + 分页）。该模块无跨模块写依赖、原型完成度高、是后续所有列表页的金样本。迁移它会驱动 Table、Pagination、Modal、Toast 等关键 shared 组件的建设。

**关键产出**：
- `shared/ui/Table.vue`（含 checkbox 列、排序、空状态）
- `shared/ui/Pagination.vue`
- `shared/ui/Modal.vue`（弹窗壳层）
- `shared/ui/Toast.vue`
- `features/customer/` 成为列表页目录结构范式

**需新增 shared 组件**：Table、Pagination、Modal、Toast、BulkActionBar（或模式抽象）

**预计工作量**：中

---

#### ③ Leads（咨询线索）

**理由**：结构与 customers 高度同构（列表 + 筛选 + 批量 + 弹窗），可复用 Wave 2 建立的全部 shared 组件。增量只有去重提示面板。用于验证列表范式的通用性。

**关键产出**：
- 验证 customers 定义的列表范式能否零改动复用
- 去重提示面板为可复用的 inline-alert 模式

**需新增 shared 组件**：无（或 InlineAlert 提升到 shared）

**预计工作量**：中偏小

---

### Wave 3 — 复杂表单与向导

#### ④ Case（案件新建）

**理由**：四步向导（Stepper）是系统中唯一的多步创建流程，与之前的列表范式完全不同。它依赖 customers 模块的数据（主申请人选择），因此必须排在 customers 之后。

**关键产出**：
- `shared/ui/Stepper.vue`（多步向导通用壳层）
- 验证跨模块数据引用模式（案件引用客户）
- 模板切换 / 条件字段显隐的 ViewModel 设计范式

**需新增 shared 组件**：Stepper

**预计工作量**：中偏大

---

### Wave 4 — 高交互 + 独有布局

#### ⑤ Tasks（任务与提醒）

**理由**：引入独有的工作台侧边栏（Workbench Sidebar）+ 行内详情面板（Detail Panel）布局。脚本数量多（5 个），筛选/详情/提醒日志/弹窗都需从零构建。需要 case 模块数据（关联案件链接）。

**关键产出**：
- 侧边栏视图切换模式
- 行内详情面板展开/收起模式
- CancelReasonDialog 弹窗模式（原因码 select + 备注）
- 状态机约束在 ViewModel 层的实现范式

**需新增 shared 组件**：DetailPanel（或内化到 feature）

**预计工作量**：大

---

### Wave 5 — 财务 & 资料

#### ⑥ Billing（收费与财务）

**理由**：系统中最复杂的列表页之一。包含摘要卡、双视图（Segmented）、收费计划面板、回款登记弹窗、风险确认留痕、三段式批量结果反馈。跨模块写依赖（Case 风险确认字段 + AuditLog）。

**关键产出**：
- SummaryCards 聚合卡片模式
- Segmented 双视图切换在 Vue 中的实现
- 风险确认 + 审计留痕的前端协议
- 三段式结果反馈 UI 模式

**需新增 shared 组件**：SummaryCard（或内化）、CollectionResultPanel

**预计工作量**：大

---

#### ⑦ Documents（资料中心）

**理由**：区块数最多（10 个），弹窗种类最多（登记 / 审核 / 退回 / waive / 引用版本 / 风险面板），绝大部分从零构建。共享版本过期联动逻辑是系统中最复杂的级联规则之一。

**关键产出**：
- relative_path 校验在前端的规范实现
- 多弹窗并发管理模式
- 版本引用 + 过期联动的前端状态模型

**需新增 shared 组件**：无新 shared（复用已有）

**预计工作量**：极大

---

### Wave 6 — 系统配置

#### ⑧ Settings（系统设置）

**理由**：几乎完全从零构建（旧原型可复用价值极低），且包含子导航（SubNav）这一独有布局模式。但它提供 Group / OrgSetting 数据，被 customers、case、billing、documents 等模块消费。

**特殊考量 — 可并行轨道**：
- 如果 Group 选择器和 OrgSetting 开关值在其他模块中需要真实数据（而非 stub），建议将 settings 作为**平行轨道**，在 Wave 2–3（customers / leads / case）阶段同步推进。
- 如果其他模块可以暂用 hardcoded stub 代替 Group 数据，则 settings 放到最后不会阻塞。

**关键产出**：
- SubNav 子导航通用模式
- Group CRUD + 停用确认的管理员流程
- 可见性开关 + 根目录配置的所级设置模式

**需新增 shared 组件**：SubNav（或内化到 feature）

**预计工作量**：大

---

## 迁移路径总览

```
Wave 1 ─── ① dashboard ─────────────────────────────────────────────────►
Wave 2 ─── ② customers ─── ③ leads ─────────────────────────────────────►
Wave 3 ─── ④ case ──────────────────────────────────────────────────────►
Wave 4 ─── ⑤ tasks ─────────────────────────────────────────────────────►
Wave 5 ─── ⑥ billing ─── ⑦ documents ──────────────────────────────────►
Wave 6 ─── ⑧ settings ──────────────────────────────────────────────────►

可选并行轨道：
  ⑧ settings ──（Group CRUD 最小可用）──► 与 Wave 2–3 同步
```

## Shared UI 组件建设路线

以下是各 Wave 驱动新增 shared 组件的时间线：

| Wave | 驱动模块 | 新增 shared 组件 |
|------|---------|-----------------|
| 1 | dashboard | SegmentedControl, SkeletonBlock |
| 2 | customers | Table, Pagination, Modal, Toast, BulkActionBar |
| 3 | case | Stepper |
| 4–5 | tasks, billing | DetailPanel, SummaryCard（视情况内化或提升） |

到 Wave 3 完成时，shared/ui 应具备覆盖 70%+ 常规管理端结构的能力，与 Foundation 计划 Phase 3 的完成标准一致。

## 每个 Wave 的验收门禁

1. **页面可独立运行**：在 dev server 中新页面可正确渲染，shell 正常。
2. **路由已注册**：`router/index.ts` 包含新路由，SideNav 高亮正确。
3. **shared 组件无业务泄漏**：新增的 shared 组件不包含业务语义。
4. **fix + guard 通过**：`npm run fix && npm run guard` 零错误。
5. **最小测试覆盖**：新增 ViewModel / 共享组件有对应单测。
6. **回归清单对照**：对照 split-manifest.json 中 `regressionChecklist` 逐项确认。

## 风险提示

| 风险 | 缓解 |
|------|------|
| Dashboard 聚合数据需要多模块 API stub | Wave 1 使用 hardcoded mock 数据，不阻塞壳层验证 |
| Customers 驱动大量 shared 组件建设，单次投入大 | 拆分为 Table → Modal → BulkActionBar 子任务，逐步交付 |
| Settings 提供 Group 数据被多模块依赖 | 并行轨道策略：先做 Group 最小可用 API；或其他模块 stub |
| Documents 区块多、弹窗多、从零构建 | 排到 Wave 5，此时 shared 组件已齐备，降低增量复杂度 |
| Case 向导步骤间有复杂联动 | 先做 Stepper 壳层 + Step 1，逐步扩展到全部 4 步 |
