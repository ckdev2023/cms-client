## 1. P0 目标

P0 的目标是让 **1 家小型签证事务所**在 **2-4 周内**用系统跑通案件主链路，不再依赖 Excel 主表管理案件。

### 1.1 范围原则

- **主链路广度保留，单模块深度控制**：从咨询到回款的链路必须完整，但每个模块只做最小闭环。
- **能跑通优先于能配置化**：模板、规则、门户、自定义能力先做基础版。
- **安全审计不后置**：与可见性、审计、留痕相关的基础能力必须有最小可用版本。

### 1.2 试点约束

| 约束 | 值 |
|------|---|
| 试点事务所数量 | 1 家 |
| 试点角色 | 行政书士（主办人）、事务员/助理、可选财务 |
| 试点案件模板 | 家族滞在、工作签证（技人国） |
| 试点周期 | 2-4 周 |

## 2. P0 功能范围

### 2.1 咨询线索

- 新建线索（联系方式、来源/介绍人、意向类型、group、备注）
- 线索状态（新咨询→跟进中→待签约→已签约→已转案件 / 已流失）
- 线索转客户/转案件（基础去重提示：电话/邮箱优先；默认继承 Lead.group）

### 2.2 客户管理

- 个人客户创建与基础字段（识别名、フリガナ、介绍人/来源、所属分组、联系方式、写真、备注）
- 关联案件列表
- 关联人关系（客户-客户）：配偶/子女/父母/代理人的轻量维护与跳转
- 基础搜索与筛选

> **P0 不做**：企业客户主数据——工作类案件的雇主/受入机构信息沉淀在案件字段。

### 2.3 案件管理

- 创建案件：选择模板（家族滞在/技人国），自动生成资料清单
- 绑定客户、关联人（CaseParty）、雇主信息（工作类）
- 家族签/扶养类：批量建案，强制绑定关键关系人（扶养者/保证人）
- 案件编号、阶段状态机（S1–S9）、负责人/协作者
- 默认继承客户 group，支持跨组建案/转组留痕
- 截止日期、下一步动作（用于工作台聚合）

### 2.4 资料清单管理

- 案件模板驱动生成资料清单
- 预置家族滞在、工作签证（技人国）两类材料字典与模板
- 资料项标记提供方（主申请人/扶养者或保证人/事务所内部）
- 资料状态切换（待提交→已上传→审核中→通过/退回/豁免）
- 附件上传（版本追加，不覆盖历史）
- 审核意见
- 完成率展示

### 2.5 缺件催办

- 待催办列表
- 手动催办与催办记录留痕
- 基础自动提醒规则

### 2.6 沟通记录

- 手工记录电话/消息/面谈/邮件摘要
- 案件时间线
- 内部备注

### 2.7 模板文书生成

- 基础模板库（委任状、理由书、申请表等）
- 变量填充
- 导出 Word / PDF

### 2.8 提交前校验与提交包

- 提交前校验：硬性阻断项最小集合
- 校验报告展示
- 提交包：锁定资料版本/文书版本与校验结果，记录提交日期与回执
- 复核能力：模型预留，默认不强制

### 2.9 期限提醒

- 在留到期提醒、补件截止提醒、提交截止提醒
- 负责人提醒
- P0 以站内提醒为主，外部触达（邮件/短信）仅预留扩展空间

### 2.10 收费与回款

- 费用总额、收费节点、回款记录、欠款标记
- 提醒与风险标识
- 欠款不作为统一提交硬阻断（提示模式）

### 2.11 审计日志

- 关键操作记录：状态变更、上传/审核、提交包生成/导出、跨组变更、负责人交接、导出/下载/分享、欠款风险确认

### 2.12 Group 与协作归属

- Group 基础管理（创建/停用/重命名，入口在系统设置）
- 归属继承链：Lead.group → Customer.group → Case.group
- 跨组建案/转组留痕

### 2.13 最小权限与可见性

- 基础角色：管理员、主办人、助理、销售、财务
- 客户可见 / 内部可见的数据分域
- 用户与 group 的最小可见范围
- 敏感导出/下载/分享的留痕

### 2.14 工作台与全局检索

- 我的待办 / 今日到期 / 已逾期 / 待我处理
- 按案件编号、客户名、电话、邮箱的全局检索
- 按 group 的过滤与基础聚合视图
- 风险案件与待提交案件的基础聚合

## 3. P0 明确不做

以下内容**不属于 P0 范围**。即使在开发中看到相关需求，也应回主目录维护而非塞入 P0：

| 不做项 | 后置到 | 原因 |
|--------|-------|------|
| 客户门户（客户侧上传/查看/付款状态） | P1 | 首版聚焦内部作业闭环 |
| 企业客户主数据与批量案件 | P1/P2 | 首版以个人客户为主 |
| 高级校验规则配置化 | P1 | 首版用预置规则 |
| 双人复核全局强制 | P1 | 模型预留，默认关闭 |
| 文书版本管理（初稿/审核版/定稿流程） | P1 | 首版用基础生成+导出 |
| 模板与材料库事务所级自定义 | P1 | 首版用预置模板 |
| 多语言（中/日/英） | P1 | — |
| 高级报表（补正率/成功率/办结时长） | P2 | — |
| 自动化流程引擎与条件触发器 | P2 | — |
| 字段级权限、审批流、模板审批 | P2 | — |
| 外部集成（邮件/SMS/LINE/会计/电子签约） | P2 | — |
| Group 层级树与复杂组间审批 | P2 | — |

## 4. P0 冻结策略摘要

以下策略在 P0 中冻结，不因开发过程中的讨论而动摇（完整规则见 [03-业务规则与不变量](03-业务规则与不变量.md)）：

| 策略 | P0 口径 |
|------|---------|
| 案件模板 | 预置 2 类：家族滞在、工作签证（技人国） |
| 客户实体 | 个人客户为主；雇主信息沉淀在案件字段 |
| 双人复核 | 模型预留，默认不强制 |
| 收费阻断 | 提示模式（warn），不作为统一硬阻断 |
| 客户门户 | 不纳入 P0 |
| 外部触达 | 站内提醒为主，外部触达仅预留 |

## 5. P0 验收标准

P0 完成的判定标准——试点事务所和研发团队以此为验收依据：

### 5.1 必须通过

1. **主链路可跑通**：从"已签约线索"到"生成首个提交包"可在系统内完成，不依赖线下 Excel 补记关键节点
2. **可追溯**：任一案件的资料、文书、校验、提交、回款均能追溯到责任人、时间和最新状态
3. **角色视图**：不同角色登录后能看到与自身职责匹配的待办与风险聚合视图
4. **模板闭环**：至少 1 类高频案件模板（家族滞在或技人国）的资料清单→文书生成→提交包的完整闭环
5. **提交包不可覆盖**：已生成的提交包，其引用的资料/文书版本不被后续新增版本覆盖
6. **操作可审计**：导出/下载/分享与跨组变更动作均可在审计日志追溯

### 5.2 可执行验收矩阵

以下矩阵用于把“主链路可跑通”转成研发、QA、试点三方都能对齐的最小检查项：

| 验收主题 | 必须通过的检查项 | 不通过表现 |
|------|----------------|-----------|
| 咨询转化 | 能从 `Lead` 创建 `Customer` 与首个 `Case`，并继承 Group；去重提示可见 | 线索只能停留在线索层，或转化后归属错乱 |
| 资料闭环 | 必交资料项可从生成 → 上传 → 审核通过 / 退回补正 → 重新上传；`waived` 有留痕 | 资料状态无法驱动下一步，或补正后历史断链 |
| Gate-A / Gate-B / Gate-C | 三道 Gate 均能产生 `ValidationRun`；硬性阻断时不能继续敏感动作 | 只靠改阶段就能绕开规则，或校验无记录 |
| 提交包不可变 | 生成提交包后，后续新增资料/文书版本不覆盖历史引用 | 历史提交内容无法复盘，或引用指向“最新版本” |
| 补正链路 | 收到补正后能新增任务、更新资料/文书、重新校验并生成补正提交包 | 补正与首次提交混在一起，无法追溯 |
| 权限与可见性 | 非管理员默认只能看本组、本人负责或被协作授权对象；搜索结果遵循同一权限边界 | 菜单隐藏但详情/搜索/导出仍能看到不该看的数据 |
| 敏感动作留痕 | 导出、下载、分享、跨组变更、负责人交接均写入审计日志 | 发生敏感动作但日志中没有记录 |
| 欠款风险 | 欠款不统一阻断提交，但继续提交前必须完成风险确认留痕 | 欠款既无提示也无确认，或被错误硬阻断 |

### 5.3 试点成功标准

| 指标 | 标准 |
|------|------|
| 主链路完成 | 1 家事务所在 2-4 周内用系统完成至少 1 类案件从建档到提交包的全流程 |
| 不再依赖 Excel | 试点期间案件管理主要在系统内完成，Excel 仅用于历史数据参考 |
| 留存意愿 | 试点结束后事务所表达继续使用或付费意愿 |
