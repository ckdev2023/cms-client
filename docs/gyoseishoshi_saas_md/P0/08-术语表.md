## P0 先读概念

> 首版开发期间必须理解的核心对象。掌握后再进入规则（03）、流程（04）、页面规格（06）。

### 对象关系图

```
Lead ──转化──→ Customer ──建案──→ Case
(咨询线索)     (个人客户)          (案件)
                  │                  │
          CustomerRelation    ┌──────┼──────┐
          (关联人跳转)         │      │      │
                        CaseParty  DocReq  BillingPlan
                        (门槛校验)(资料项)(收费节点)
                                     │
                              DocFileVersion
                              (附件版本)
                                     │
                          SubmissionPackage ←── GenDocVersion
                          (提交包快照)          (文书版本)
```

### P0 必须理解的 7 个概念

| 概念 | 一句话定义 | P0 补充 |
|------|-----------|---------|
| **Customer** | 个人客户主数据，一个客户可关联多个案件 | P0 以个人客户为主体，不引入企业客户 |
| **Case** | 系统核心工作单元，一案一单 | P0 预置家族滞在与技人国两类模板 |
| **Group** | 内部归属团队，非企业客户 | P0 单层治理，不做层级树 |
| **Stage** | 案件管理层阶段（S1–S9），跨签证类型通用 | P0 冻结 9 个阶段，手动驱动推进；见「双层状态模型」 |
| **Step** | 业务子步骤（CaseWorkflowStep），模板专属细粒度节点 | **P1 启用**；P0 两类模板无 Step；见「双层状态模型」 |
| **Gate** | 阶段间的质量关卡（A/B/C 三道） | P0 用预置规则，不可配置 |
| **DocumentRequirement** | "需要什么材料"的要求（≠ 文件本身） | 与 DocumentFileVersion 是一对多 |
| **SubmissionPackage** | 一次对外提交的完整快照，生成后锁定 | P0 最关键的不可变对象 |
| **ResidencePeriod** | 客户每次成功取得在留资格后的期间记录 | **P1 启用**；P0 仅用 Customer.residence_expiry_date 单字段 |

### 最常混淆的概念

| 概念 A | 概念 B | 区别 |
|--------|--------|------|
| DocumentRequirement（资料项） | DocumentFileVersion（附件版本） | 资料项是"要什么"；附件版本是"实际交的文件" |
| CaseParty（案件关联人） | CustomerRelation（客户关系映射） | CaseParty 参与门槛校验与提交包锁定；CustomerRelation 仅用于客户侧导航跳转 |
| Group（归属团队） | Company（企业客户） | Group 是内部团队归属；Company 是外部企业主体（P0 不引入） |
| Stage（管理阶段 S1–S9） | Step（业务子步骤，CaseWorkflowStep） | Stage 是跨签证类型的通用协作视图；Step 是模板专属的操作节点（P1 启用）；两者并行不冲突 |
| Stage（阶段） | Gate（门槛） | Stage 标识案件在哪个阶段；Gate 是阶段间的质量关卡 |
| Template（文书模板） | CaseTemplate（案件模板） | Template 是文书生成模板；CaseTemplate 驱动资料清单/默认任务/校验规则 |

## 核心业务对象

### Lead（咨询线索）

| 项目 | 内容 |
|------|------|
| 定义 | 签约/立案前的早期信息与跟进记录。转化时生成 Customer 与 Case |
| P0 状态 | 新咨询 → 跟进中 → 待签约 → 已签约 → 已转案件；或 → 已流失 |
| 归属 | Lead.group → 转化时默认继承至 Customer.group → Case.group |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-p0-核心对象)、[03 §3.6](03-业务规则与不变量.md#36-线索状态) |

### Customer（个人客户）

| 项目 | 内容 |
|------|------|
| 定义 | 个人客户主数据记录。一个 Customer 可关联多个 Case |
| P0 约束 | 以个人客户为主体；雇主信息沉淀在案件字段，不引入企业客户实体 |
| 易混辨析 | ≠ Company（P0 不引入）；≠ ClientUser（P0 无客户门户） |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-p0-核心对象)、[07 §3.2](07-数据模型设计.md#32-customer个人客户) |

### Case（案件）

| 项目 | 内容 |
|------|------|
| 定义 | 系统核心工作单元。一案一单，拥有独立的阶段、资料清单、任务、期限、收费与审计记录 |
| P0 约束 | 预置家族滞在与技人国两类模板；Case.group 为归属快照，不因 Customer.group 变更而回写 |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-p0-核心对象)、[07 §3.5](07-数据模型设计.md#35-case案件) |

### Group（归属团队/项目组）

| 项目 | 内容 |
|------|------|
| 定义 | 内部归属维度，用于客户/案件的协作归属、检索筛选与可见性边界 |
| 易混辨析 | ≠ 企业客户（Company）；≠ 来源渠道（source_channel）；≠ 普通标签 |
| P0 约束 | 单层治理，不展开层级树/字段级权限/审批流。被引用的 Group 只能停用不可物理删除 |
| 继承链 | Lead.group → Customer.group → Case.group（默认继承，跨组建案/转组须留痕） |
| 权威来源 | [03 §12](03-业务规则与不变量.md#12-group-治理) |

### CaseParty（案件关联人）

| 项目 | 内容 |
|------|------|
| 定义 | 案件维度的关键关系人记录。角色包括主申请人、扶养者、保证人、配偶、子女、企业担当 |
| 用途 | 门槛校验、文书变量填充、责任边界、提交包快照锁定 |
| 易混辨析 | ≠ CustomerRelation（客户侧关联关系仅用于导航跳转，不作为门槛口径） |
| 权威来源 | [03 §2.1](03-业务规则与不变量.md#21-p0-核心对象)、[07 §3.7](07-数据模型设计.md#37-caseparty案件关联人) |

### CustomerRelation（客户关系映射）

| 项目 | 内容 |
|------|------|
| 定义 | 客户侧的关联人关系（配偶/子女/父母/代理人），用于展示、跳转、批量建案聚合 |
| P0 约束 | 不作为案件门槛口径，不随提交包锁定 |
| 权威来源 | [07 §3.2A](07-数据模型设计.md#32a-customerrelation客户关系映射) |

### CaseCollaborator（案件协作成员）

| 项目 | 内容 |
|------|------|
| 定义 | 案件绑定的协作成员记录。角色类型：principal（负责人）/ collaborator（协作者）/ viewer（仅查看） |
| 用途 | 多人协作与跨组授权，与 Group 可见性口径叠加 |
| 权威来源 | [07 §3.5A](07-数据模型设计.md#35a-casecollaborator案件协作成员) |

## 资料与文书

### DocumentRequirement（资料项）

| 项目 | 内容 |
|------|------|
| 定义 | 案件需要的材料要求。资料项是"要什么"，不是文件本身 |
| P0 状态 | not_sent → waiting_upload → uploaded_reviewing → approved / revision_required / waived / expired |
| 易混辨析 | ≠ DocumentFileVersion（附件版本 = 实际提交的文件版本） |
| 权威来源 | [03 §3.2](03-业务规则与不变量.md#32-资料项状态)、[03 §7](03-业务规则与不变量.md#7-资料项治理) |

### DocumentFileVersion（资料附件版本）

| 项目 | 内容 |
|------|------|
| 定义 | 实际上传的文件版本。每次上传生成新版本，不覆盖历史 |
| 不可变性 | 版本一旦创建不可覆盖；审核结果写在版本上 |
| 权威来源 | [03 §2.3](03-业务规则与不变量.md#23-资料项与附件版本分离)、[07 §3.10](07-数据模型设计.md#310-documentfileversion资料附件版本) |

### CaseTemplate（案件模板）

| 项目 | 内容 |
|------|------|
| 定义 | 按"签证类型 × 申请类型"驱动，生成资料清单、默认任务、校验规则集与文书模板建议 |
| P0 约束 | 预置家族滞在与工作签证（技人国）两类，不支持事务所自定义 |
| P1 扩展 | 新增 `workflow_steps_blueprint`、`extra_fields_schema`、`reminder_schedule_blueprint` 三个 JSON 字段，支持接入新签证类型（如经营管理签）而无需 DDL 变更 |
| 权威来源 | [07 §3.8](07-数据模型设计.md#38-casetemplate案件模板) |

### 双层状态模型（P1 可扩展性核心设计）

| 项目 | 内容 |
|------|------|
| 定义 | 将案件状态拆分为两层：管理层（Stage S1–S9，跨签证类型通用）与业务层（Step，CaseWorkflowStep，模板专属细粒度节点） |
| 为何这样设计 | 避免不同签证类型的专属步骤（如 COE_SENT、VISA_APPLYING）污染通用主阶段枚举；新签证类型只需新增 CaseTemplate 并配置 blueprint，不改任何现有逻辑 |
| P0 状态 | 业务层不启用；家族滞在/技人国只跑 S1–S9 |
| P1 激活条件 | CaseTemplate.workflow_steps_blueprint 非 null 时，该模板的案件自动启用业务步骤视图 |
| 权威来源 | [07 §1 设计原则](07-数据模型设计.md)、[07 §3.24](07-数据模型设计.md#324-caseworkflowstep业务子步骤记录p1-预定义) |

### CaseWorkflowStep（业务子步骤记录，P1）

| 项目 | 内容 |
|------|------|
| 定义 | 记录特定签证模板专属的业务操作节点（如 UNDER_REVIEW / COE_SENT / VISA_APPLYING / ENTRY_SUCCESS）的进入、完成、跳过历史 |
| 与 CaseStageHistory 区别 | CaseStageHistory 记录管理层 S1–S9 流转；CaseWorkflowStep 记录业务层操作节点；两者并行，职责不重叠 |
| P0 状态 | **不启用**，表已预建，P1 新签证模板配置后激活 |
| 权威来源 | [07 §3.24](07-数据模型设计.md#324-caseworkflowstep业务子步骤记录p1-预定义) |

### ResidencePeriod（在留期间历史，P1）

| 项目 | 内容 |
|------|------|
| 定义 | 客户每次成功取得在留资格后的完整期间记录，含在留期 start/end、产生来源案件 ID、是否已完成到期提醒批量生成 |
| 与 Customer.residence_expiry_date 区别 | 该字段只存当前到期日（单值）；ResidencePeriod 支持多次续签历史追溯 |
| 提醒联动 | 写入 ResidencePeriod 后，按 CaseTemplate.reminder_schedule_blueprint 批量生成 CaseDeadline 与 Reminder |
| P0 状态 | **不启用**，P0 到期日手动维护 Customer.residence_expiry_date |
| 权威来源 | [07 §3.25](07-数据模型设计.md#325-residenceperiod在留期间历史记录p1-预定义) |

### Template（文书模板）

| 项目 | 内容 |
|------|------|
| 定义 | 版本化的文书模板，含变量字典与格式化规则，用于生成 GeneratedDocumentVersion |
| P0 约束 | 预置模板，不支持事务所级自定义编辑 |
| 权威来源 | [03 §8](03-业务规则与不变量.md#8-文书规则)、[07 §3.14](07-数据模型设计.md#314-template文书模板) |

### GeneratedDocumentVersion（生成文书版本）

| 项目 | 内容 |
|------|------|
| 定义 | 模板填充生成的实际文书产物。每次修改产出新版本 |
| P0 状态 | draft → final → exported（三态流转，完整版本管理后置到 P1） |
| 不可变性 | 提交包引用固定版本 |
| 权威来源 | [03 §3.3](03-业务规则与不变量.md#33-文书状态)、[07 §3.15](07-数据模型设计.md#315-generateddocumentversion生成文书版本) |

### SubmissionPackage（提交包）

| 项目 | 内容 |
|------|------|
| 定义 | 一次对外提交的快照。锁定提交日期、受理机关、回执，以及引用的资料版本、文书版本、关键字段快照 |
| 不可变性 | 提交后不允许覆盖式替换历史引用。补正通过"新版本 + 新提交包"完成 |
| 权威来源 | [03 §2.4](03-业务规则与不变量.md#24-提交包锁定与不可覆盖)、[03 §13](03-业务规则与不变量.md#13-提交动作强规则) |

### 材料字典

| 项目 | 内容 |
|------|------|
| 定义 | 系统预置的资料项命名标准库，含默认说明/样例与时效建议。案件模板从材料字典引用标准项 |
| P0 约束 | 预置 2 类模板的材料字典，不支持事务所级覆盖 |
| 权威来源 | [03 §7.3](03-业务规则与不变量.md#73-p0-模板策略) |

## 流程与校验机制

### Stage（案件阶段）

| 项目 | 内容 |
|------|------|
| 定义 | 案件在主流程中所处的阶段，驱动工作台视图与默认任务 |
| P0 枚举 | S1 已建档、S2 资料收集中、S3 资料审核中、S4 文书制作中、S5 待校验、S6 待提交、S7 已提交审理中、S8 已出结果、S9 已归档 |
| 易混辨析 | Stage 用于管理协作；Gate 用于阻断关键错误 |
| 权威来源 | [03 §3.1](03-业务规则与不变量.md#31-案件阶段)、[04 §2](04-核心流程与状态流转.md#2-案件阶段s1s9) |

### Gate（门槛校验）

| 项目 | 内容 |
|------|------|
| 定义 | 流程中的关键关卡，用于阻断关键错误。每次执行沉淀为 ValidationRun |
| P0 枚举 | Gate-A（进入文书，S3→S4）、Gate-B（进入校验，S4→S5）、Gate-C（生成提交包前） |
| P0 约束 | 预置规则，不可配置化 |
| 权威来源 | [03 §4.3](03-业务规则与不变量.md#43-三道-gate)、[04 §3](04-核心流程与状态流转.md#3-gate-触发点与通过后动作) |

### 硬性阻断 / 软性提示

| 项目 | 内容 |
|------|------|
| 定义 | 校验结果的两个严重级别。**硬性阻断**：必须修复才可继续。**软性提示**：风险提示，不阻断流程 |
| 权威来源 | [03 §4.1](03-业务规则与不变量.md#41-硬性阻断p0-最小集合)、[03 §4.2](03-业务规则与不变量.md#42-软性提示p0-最小集合) |

### ValidationRun（校验运行记录）

| 项目 | 内容 |
|------|------|
| 定义 | 每次 Gate 执行产生的记录，区分硬性阻断与软性提示 |
| P0 状态 | pending / failed / passed |
| 权威来源 | [03 §4](03-业务规则与不变量.md#4-校验门槛gate规则)、[07 §3.16](07-数据模型设计.md#316-validationrun校验运行记录) |

### ReviewRecord（双人复核记录）

| 项目 | 内容 |
|------|------|
| 定义 | 校验通过后提交前的复核环节。驳回须给出原因 |
| P0 约束 | 模型预留，默认关闭。事务所或模板可按需开启 |
| 结果 | approved / rejected |
| 权威来源 | [03 §1.2](03-业务规则与不变量.md#12-预留但默认关闭)、[07 §3.17](07-数据模型设计.md#317-reviewrecord双人复核) |

### waived（无需提供/不适用）

| 项目 | 内容 |
|------|------|
| 定义 | 资料项被豁免的状态。必须记录原因（原因码 + 文本）、操作人与时间。从完成率分母剔除 |
| 权威来源 | [03 §7.2](03-业务规则与不变量.md#72-waived-治理) |

## 收费与财务

### BillingPlan（收费计划/收费节点）

| 项目 | 内容 |
|------|------|

## 业务领域术语（入管实务）

> 以下为 P0 试点涉及的入管实务概念。P0 预置家族滞在与技人国两类模板，以下术语在这两类案件中高频出现。

| 术语 | 日文 | 定义 |
|------|------|------|
| 行政书士 | 行政書士 | 日本国家资格，代理行政许可申请文书的作成与提出。系统核心用户 |
| 事务所 | 事務所 | 行政书士的执业机构 |
| 在留资格 | 在留資格 | 外国人在日本合法停留的资格类别 |
| 在留期限 | 在留期間 | 在留资格的有效期限。到期前需申请更新，是提醒的核心触发数据 |
| 在留卡 | 在留カード | 中长期在留外国人的身份证件，卡号用于去重与检索 |
| 认定 | 認定 | 在留资格认定证明书交付申请，通常为首次入境 |
| 变更 | 変更 | 在留资格变更许可申请 |
| 更新 | 更新 | 在留期间更新许可申请 |
| 家族滞在 | 家族滞在 | 在留资格之一，适用于在日就劳者的配偶与子女。**P0 预置模板之一** |
| 技人国 | 技術・人文知識・国際業務 | 在留资格之一，适用于专业技术/人文/国际业务就劳。**P0 预置模板之一** |
| 扶养者 | 扶養者 | 家族滞在案件的关键关系人（持就劳签的配偶/父母）。其在留信息是门槛校验必要条件 |
| 保证人 | 保証人 | 部分案件类型要求的身份保证人 |
| 受入机关 | 受入機関 | 接收外国人就劳/研修的企业或机关 |
| 入管 | 出入国在留管理庁 | 负责在留资格审批的日本政府机关 |
| フリガナ | フリガナ | 日文姓名的片假名读音标注，用于检索与官方表单 |
| 委任状 | 委任状 | 客户委托行政书士代理的授权文书 |
| 理由書 | 理由書 | 向入管提交的申请理由说明书 |
| 補正 | 補正 | 入管审查后要求补充/修正材料的通知。系统以"补正提交包"追踪 |
| 回执 | 受付票 | 案件提交后入管发放的受理凭证 |

## P0 不涉及的概念

以下概念在 P0 文档中不会主动使用。如需了解定义，请查阅主目录 [08-术语表](../08-术语表.md)：

| 概念 | 原因 |
|------|------|
| Company（企业客户） | P0 不引入企业客户实体 |
| ContactPerson（企业担当） | P0 不引入 |
| ClientUser（客户门户用户） | P0 无客户门户 |
| 永住 / 归化 | P0 模板不覆盖（家族滞在与技人国） |
| 法人番号 | P0 不引入 Company，法人番号不作为核心字段 |
| WorkflowRule（自动化规则） | P0 流程人工驱动 |
| NotificationTemplate（多语言消息模板） | P0 仅站内提醒 |
| CaseWorkflowStep（业务子步骤） | P1 启用；P0 字段/表已预建，P0 业务逻辑不使用 |
| ResidencePeriod（在留期间历史） | P1 启用；P0 用 Customer.residence_expiry_date 单字段存当前值 |
| CaseSurvey（独立问卷引擎） | P2；P1 用 DocumentRequirement.survey_data JSON 承载 |
| extra_fields / extra_fields_schema | P1 启用；P0 两类模板无专属扩展字段 |

## 附：易混概念速查

| 概念 A | 概念 B | 区别 |
|--------|--------|------|
| DocumentRequirement（资料项） | DocumentFileVersion（附件版本） | 要求 vs 实际文件 |
| CaseParty（案件关联人） | CustomerRelation（客户关系映射） | 门槛校验+提交锁定 vs 导航跳转 |
| Group（归属团队） | Company（企业客户） | 内部维度 vs 外部主体（P0 不引入） |
| Stage（管理阶段 S1–S9） | Step（业务子步骤） | 通用跨签证协作视图 vs 模板专属细粒度操作节点（P1） |
| Stage（阶段） | Gate（门槛） | 管理标识 vs 质量关卡 |
| 硬性阻断 | 软性提示 | 必须修复 vs 仅风险提示 |
| 可见（view） | 可导出（export） | 可见不等于可导出 |
| SubmissionPackage（提交包） | 补正提交包 | 都是独立提交包；补正包通过 related_submission_id 关联原包 |
| Template（文书模板） | CaseTemplate（案件模板） | 文书生成模板 vs 案件类型模板 |
| CaseStageHistory | CaseWorkflowStep | 管理层阶段流转记录（S1–S9）vs 业务子步骤记录（模板专属，P1） |
| Customer.residence_expiry_date | ResidencePeriod | 当前到期日单字段 vs 多次续签历史记录表（P1） |
