## 1. P0 版本策略冻结

P0 采用 **"宽链路、浅能力、强留痕"** 策略：从咨询到回款的主链路模块全部保留，但每个模块只做最小闭环，不在首版追求配置化或门户化。

### 1.1 冻结口径

| 策略 | P0 口径 |
|------|---------|
| 案件模板 | 预置 2 类：家族滞在、工作签证（技人国）。不支持事务所自定义模板 |
| 客户实体 | 个人客户为主；雇主信息沉淀在案件字段，不引入企业客户实体 |
| 双人复核 | 模型预留，默认不强制。事务所/模板可按需开启 |
| 收费阻断 | 提示模式（warn），不作为统一硬阻断 |
| 客户门户 | 不纳入 P0 |
| 外部触达 | 站内提醒为主，外部触达仅预留扩展空间 |
| Group 治理 | 单层治理，不做层级树、字段级权限、审批流 |

### 1.2 预留但默认关闭

以下能力在模型层预留开关，P0 默认关闭或降级：

| 能力 | P0 默认状态 | 开关位置 |
|------|-----------|---------|
| 双人复核 | 关闭；事务所或案件模板可按需开启 | CaseTemplate.review_required_flag |
| 收费阻断 | 提示模式（warn） | CaseTemplate.billing_gate_mode |
| 外部触达（邮件/短信/IM） | 不作为首版依赖 | 提醒渠道配置 |

## 2. 对象边界与不可变规则

### 2.1 P0 核心对象

| 对象 | 定义 | P0 补充 |
|------|------|---------|
| Customer | 个人客户，可关联多个案件 | P0 以个人客户为主体 |
| Case | 案件，一案一单，拥有独立的阶段、资料清单、收费、审计 | — |
| CaseParty | 案件关联人，用于门槛校验、文书变量、责任边界 | ≠ CustomerRelation（客户侧关联关系仅用于检索与跳转） |
| Group | 内部归属团队/项目组，用于协作归属与可见性边界 | ≠ 企业客户 ≠ 来源渠道 ≠ 普通标签 |
| Lead | 咨询线索，承接未签约/未立案的早期信息 | 转化时生成 Customer 与 Case |
| DocumentRequirement | 资料项 = 材料要求 | ≠ 附件（DocumentFileVersion） |
| DocumentFileVersion | 附件版本，每次新增版本不覆盖历史 | P0 默认不存 SaaS 文件本体；以“本地归档登记”为主，必须记录 `storage=local_server` 与 `relative_path`（相对路径，可复制） |
| SubmissionPackage | 提交包 = 一次对外提交的快照，引用具体版本并锁定 | — |
| ValidationRun | 校验运行记录，区分硬性阻断与软性提示 | — |

### 2.2 归属继承链

```
Lead.group → Customer.group → Case.group（默认继承）
```

- **Case.group 是案件归属快照**：Customer.group 的后续变更不回写覆盖历史案件。
- 跨组建案/转组必须留痕（操作人、时间、原因），写入审计日志。
- 线索转客户时默认继承 Lead.group；如转化时改组，必须记录原因、操作人与时间。

### 2.3 资料项与附件版本分离

- 资料项是"材料要求"，附件是"提交的证据版本"。两者是一对多关系。
- 同一资料项允许多版本追加；每次上传生成新版本，不覆盖历史版本。
- 文书与提交包引用必须指向"具体版本"，保证审计与复盘成立。
- 多案件可引用同一附件版本；提交后若需更新材料，应创建新版本用于新的提交包。

> **P0 落地口径（重要）**：P0 资料处理以事务所本地资料服务器为主，系统中的“上传/新增版本”应被理解为**登记资料版本**：
>
> - 系统不保存文件二进制；保存“版本元数据 + 状态流转 + 审核/催办留痕”
> - 每个版本至少记录：`storage=local_server`、`relative_path`（相对路径，不含盘符/根目录）、`file_name`（或描述）、操作人、时间
> - 本地根目录在事务所设置中配置；系统仅存相对路径，确保可迁移与权限隔离

### 2.4 提交包锁定与不可覆盖

这是 P0 最关键的不可变规则之一——提交版本不可追溯是首版要解决的核心痛点。

- 提交包引用"具体附件版本 / 文书版本 / 关键字段快照"。
- 提交后不允许覆盖式替换历史，只能新增版本并生成新提交包。
- 补正属于新一轮提交：形成"补正提交包"并与原提交包关联。
- 无论首次提交还是补正提交，只要产生新的对外提交，就必须创建新的提交包。

### 2.5 可见性标记

- 沟通记录、附件、文书必须标记 `visible_scope`：`internal_only` / `client_visible`。
- P0 无客户门户，但标记必须存在——内部备注与客户可见内容的分离从首版开始建立。

### 2.6 去重规则

- 去重不做物理覆盖删除：保留原记录，建立归并关系，历史数据不断链。
- 去重检查触发点：线索转客户、批量建档/建案时。
- 匹配优先级：电话/邮箱优先，其次姓名+生日（或证件号）；支持人工确认合并。

## 3. 状态机定义

> 各状态机的转移规则与触发条件详见 [04-核心流程与状态流转](04-核心流程与状态流转.md)。

### 3.1 案件阶段

| 阶段 | 代号 | 说明 |
|------|------|------|
| 已建档 | S1 | 已签约/已立案 |
| 资料收集中 | S2 | 对客户发出资料请求 |
| 资料审核中 | S3 | 内部审核/退回补正 |
| 文书制作中 | S4 | 模板生成、整理、定稿 |
| 待校验 | S5 | 进入提交前校验关卡 |
| 待提交 | S6 | 校验通过，等待对外提交 |
| 已提交审理中 | S7 | 含补正/追加提交 |
| 已出结果 | S8 | 许可/不许可 |
| 已归档 | S9 | 结案 |

### 3.2 资料项状态

| 状态 | 枚举值 | 说明 |
|------|--------|------|
| 未发送 | `not_sent` | 资料项已生成但尚未对客户发出请求 |
| 待提交 | `waiting_upload` | 已请求客户提交，可触发催办 |
| 已提交待审核 | `uploaded_reviewing` | 客户已上传最新版本，等待内部审核 |
| 审核通过 | `approved` | 满足要求，可用于文书/提交包引用 |

| 已过期需重交 | `expired` | 超过时效或证件有效期不足 |

### 3.3 文书状态

| 状态 | 枚举值 | 说明 |
|------|--------|------|
| 草稿 | `draft` | 模板生成或手动创建的初版 |

| 状态 | 说明 |
|------|------|

| 状态 | 说明 |
|------|------|
| 应收 | 已确认待付款 |
| 部分回款 | — |
| 已回款 | 已结清 |
| 欠款 | — |

### 3.6 线索状态

| 状态 | 说明 |
|------|------|
| 新咨询 | — |
| 跟进中 | — |
| 待签约 | — |
| 已签约 | — |
| 已转案件 | — |

### 3.7 校验结果状态

| 状态 | 枚举值 | 说明 |
|------|--------|------|
| 待校验 | `pending` | — |

## 4. 校验门槛（Gate）规则

校验分为两层：硬性阻断（必须修复才可继续）和软性提示（风险提示但不阻断）。目的是避免过度阻断影响效率，也避免完全不阻断导致质量失控。

### 4.1 硬性阻断（P0 最小集合）

- 必填字段完整
- 关键关系人（CaseParty）齐备（主申请人 / 扶养者绑定）
- 必交资料项齐备且通过（`approved`）
- 关键文书已定稿（`final`）
- （如启用）双人复核签名通过

### 4.2 软性提示（P0 最小集合）

- 日期逻辑 / 有效期风险
- 收费欠款风险
- 建议补充材料
- 附件版本过旧但未过期
- 关键信息疑似未更新（地址/在留期限等）

### 4.3 三道 Gate

P0 冻结 3 个 Gate。每次 Gate 执行都沉淀为一条校验运行记录（ValidationRun）与报告。

> Gate 的触发点与通过后动作详见 [04-核心流程](04-核心流程与状态流转.md) §1.4。

#### Gate-A：进入文书

- **硬性阻断**：关键关系人（CaseParty）齐备；关键资料项达到可做文书的最低门槛（状态为 `approved`）
- **软性提示**：附件版本过旧但未过期、关键信息疑似未更新、欠款风险

#### Gate-B：进入提交前校验

- **硬性阻断**：关键文书已定稿（`final`）；必交资料项齐备且通过；关键字段必填完成
- **软性提示**：日期逻辑/有效期风险、材料时效建议、收费节点异常

#### Gate-C：提交（生成提交包前）

- **硬性阻断**：最新一次校验结果为通过；如启用双人复核则复核通过；提交信息字段满足最小要求（提交日期/机关/回执等）
- **软性提示**：欠款风险、非关键材料建议补充
- **欠款特别规则**：若存在欠款风险且仍要提交，必须先完成风险确认留痕（确认人、原因，可选上传凭证）

### 4.4 重新校验触发条件

以下变化发生后，必须重新生成 ValidationRun，避免提交引用与校验结果不一致：

- 任一必交资料项新增了附件版本且被审核通过/退回补正
- 任一关键关系人（CaseParty）新增/删除/角色变更
- 任一关键字段发生变更（姓名、在留期限、住址、雇主信息等按模板定义）
- 任一关键文书发生变更或重新定稿
- 复核启用策略发生变化

## 5. 审计事件最小集合

案件时间线用于"单页可扫读"，审计日志用于"可追责"。两者事件口径一致，展示层次可不同。

### 5.1 P0 必须覆盖的事件

| 类别 | 事件 |
|------|------|
| 案件 | 创建、阶段变更、负责人/协作者变更、Group 变更、跨组建案/转组 |
| 资料 | 资料项状态变更、标记无需提供（含原因）、审核意见变更 |
| 附件 | 附件版本新增、下载/导出/分享 |
| 催办 | 催办发送、催办记录 |
| 提交与风控 | 校验执行与结果、复核签名（如启用）、提交包生成/导出、收费节点确认、回款记录新增、欠款风险确认 |
| 敏感动作 | 导出/下载/分享、负责人交接 |

### 5.2 审计日志治理

- 审计日志**默认不可删**。
- 敏感导出与跨案件批量操作需权限控制。
- 最小字段集合：对象类型、对象 ID、操作类型、操作原因（可选）、变更前后数据、操作人、操作时间。
- 审计载荷采用**白名单字段**策略：默认记录业务判断所需摘要，不默认记录完整证件号、完整地址、完整附件内容等高敏信息。
- 审计日志的查看权限应独立于业务对象查看权限，避免“为了留痕而扩大泄露面”。

## 6. 收费与欠款策略

- P0 默认**不将欠款作为统一提交硬阻断**。
- 收费节点与案件阶段绑定（例如"受理前/提交前/结果后"），到达节点触发内部提醒。
- 收费异常在 P0 以**风险提示和工作台暴露**为主，不统一阻断提交。
- 欠款状态下继续提交时，必须记录风险确认（确认人、原因，可选上传凭证），进入案件时间线与审计日志。
- 工作台应将未确认的欠款案件聚合为风险案件并提示补录。

## 7. 资料项治理

### 7.1 完成率口径

- 默认口径：`完成率 = 已审核通过的必交资料项数 / 必交资料项总数`
- **分母剔除**：标记为 `waived` 的资料项不计入总数，但必须记录原因、操作人与操作时间。
- 支持按提供方输出子完成率（主申请人 / 扶养者或保证人 / 事务所内部）。

### 7.2 waived 治理

- **必填留痕**：原因（结构化原因码 + 自由文本补充）、操作人、操作时间。
- **权限控制**：默认仅允许助理/主办人及以上角色执行。
- **生命周期约束**：已被提交包引用并完成对外提交的资料项，不可删除或覆盖历史记录。

### 7.3 P0 模板策略

- 系统预置 2 类模板（家族滞在、工作签证/技人国）的材料字典与默认说明/样例。
- 案件内允许手动追加资料项或标记无需提供（需留痕）。
- P0 不支持事务所级模板自定义——模板覆盖能力后置到 P1。

### 7.4 资料项标记要求

- 资料项必须标记提供方（主申请人 / 扶养者或保证人 / 事务所内部），支持按提供方展示与催办。
- 多对象/多案件允许引用同一附件版本；引用指向具体版本，避免共享可变文件。

## 8. 文书规则

- **变量字典**：每个变量需定义来源字段、缺失时策略（阻断/提示/留空）。
- **版本不可变**：文书导出关联到具体版本，便于审计与提交包锁定。导出后的改动不影响已锁定版本。
- **P0 文书流程**：草稿 → 定稿 → 导出。完整的初稿/审核版/定稿版本管理后置到 P1。
- **进入校验最低门槛**：关键资料项 `approved` 且关键文书 `final`。

---

## 9. 沟通记录规则

- **最小标准**：渠道、对象、摘要、结论、下一步动作（可一键转任务）。
- 内部备注与客户可见内容**必须分离**（`visible_scope` 标记），避免误发敏感信息。
- 沟通记录可见性变更属于审计事件。

## 10. 权限与可见性

### 10.1 P0 角色

管理员 / 主办人 / 助理 / 销售 / 财务。

### 10.2 可见性口径

- **管理员**：默认可见全量对象。
- **普通用户**：默认仅可见本组对象、本人负责对象及被显式协作授权的案件。

### 10.3 访问控制维度

内部侧对象级访问控制至少同时考虑三个维度：**角色**、**Group**（归属团队）、**负责人/协作者**。

### 10.4 可见 ≠ 可导出

- 权限表达拆分为：**可见（view）**、**可编辑（edit）**、**可导出/下载（export）**。
- 即使可见也不默认等于可导出。
- 导出、下载、分享链接等动作**必须记录到审计日志**。

### 10.5 最小权限矩阵

| 资源 / 动作 | 管理员 | 主办人 | 助理 | 销售 | 财务 |
|------|------|------|------|------|------|
| 线索查看 | 全量 | 本组 + 本人负责 | 本组 | 本组 | 不默认开放 |
| 客户查看 | 全量 | 本组 + 本人负责 | 本组 | 本组基础信息 | 不默认开放 |
| 案件查看 | 全量 | 本组 + 负责/协作案件 | 本组 + 协作案件 | 不默认开放 | 收费相关视图 |
| 资料 / 文书编辑 | 全量 | 负责案件 | 协作案件的执行字段 | 不可 | 不可 |
| 校验 / 提交包生成 | 可 | 负责案件 | 不可 | 不可 | 不可 |
| 收费记录编辑 | 可 | 可 | 不可 | 不可 | 可 |
| 导出 / 下载 / 分享 | 受控开放 | 受控开放 | 默认关闭 | 默认关闭 | 财务相关受控开放 |

> 上表是 P0 的最小口径。具体页面呈现可以更收缩，但不能比此表更宽松。

### 10.6 服务端强制口径

- 权限判断必须在服务端统一执行，不能只依赖前端隐藏按钮或菜单。
- 列表、详情、全局搜索、导出/下载、批量动作必须使用同一套授权条件：`role + org + group + 负责人/协作关系`。
- 即使用户已知对象 ID，只要不满足授权条件，也不得返回对象详情、附件、提交包或导出结果。
- 跨组协作属于显式授权，不得被“同角色”默认放大。

---

## 11. 提醒规则

- P0 以站内提醒为主。外部触达（邮件/IM）仅预留扩展空间，不作为首版依赖。
- 关键提醒类型：在留到期、补件截止、提交截止、收费节点。
- 催办支持基础升级：客户未响应 → 提醒负责人。
- 所有触达必须留痕。

---

## 12. Group 治理

- Group 统一定义为"内部归属团队/项目组"。不等同于企业客户、来源渠道、普通标签。
- P0 只做**单层治理**：创建/停用/重命名，入口在系统设置。
- 被 Customer/Case 引用过的 Group 仅允许停用，不做物理删除。
- 跨组建案/转组是例外动作，执行时必须留痕（操作人、时间、原因）。

---

## 13. 提交动作强规则

以下规则在 P0 冻结为强规则，不因开发讨论而动摇：

1. 提交包必须锁定引用：资料附件版本、文书版本、校验报告与复核记录（如存在）。
2. 提交后禁止覆盖式替换历史引用；后续补正必须通过"新版本 + 新提交包"完成。
3. 提交包导出、回执下载、对外分享等属于敏感动作，必须权限控制并写入审计日志。
4. 提交前必须引用"最新一次有效校验结果"；若关键资料版本、关键字段或定稿文书发生变化，应重新校验。
5. 阶段推进、校验执行、提交包生成必须由服务端重新校验前置条件，不能只依赖页面状态。
