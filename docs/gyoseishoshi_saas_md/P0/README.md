## P0 与主目录的关系

| 维度 | P0（本目录） | 主目录（上级） |

### 一份权威源，两种阅读视图

- **人读视图**：按 `00 → 01 → 02 → 03/04/05/06/07/08` 阅读，保留背景、解释、例子
- **AI 读视图**：先读 [09-结构化总索引与交叉映射](09-结构化总索引与交叉映射.md)，再跳转对应正文
- **执行视图**：原始 PRD / 讨论记录不可直接执行，必须先走“raw → extract → clarify → freeze → execute → verify / write-back”最小编译流水线；若目标是落地**已有原型交互**或**数据设计**，编译结果应优先回写到对应原型页的 `P0-CONTRACT*`（交互契约）与 `MIGRATION-MAPPING*`（数据契约），而不是继续扩展抽象文档
- **治理原则**：`09` 只做结构化导航，不新造规则；真正的规则仍以正文文档为准

## 核心治理规则

### R-1：P0 是首版权威源

研发、设计、QA 在首版开发期间，以 P0/ 下的文档为唯一执行依据。如果某需求只在主目录中存在而 P0/ 中没有，则该需求**不属于首版范围**。

### R-2：所有内容必须通过试点必要性测试

每个章节、每条规则、每个页面都必须能回答：**"若删掉这部分，首个试点是否仍可跑通？"** 若答案是"能"，则不应进入 P0。

### R-3：P1/P2 需求不回灌

后续发现的 P1/P2 需求不直接塞回 P0/，而是回主目录对应文档维护。P0/ 只做首版口径的澄清与纠错。

### R-4：与主目录的同步规则

- P0/ 中对规则的修正，应同步检查主目录对应文档是否需要更新。
- 主目录中的 P0 范围变更，应同步反映到 P0/ 对应文档。
- 同步动作记录在 [99-文档维护与版本记录](99-文档维护与版本记录.md) 中。

### R-5：文档必须重写而非复制

每篇 P0 文档都必须重写开头结论、目标对象、排除项和验收口径。禁止从主目录复制后仅做少量删减。

### R-6：先结构化，再执行

AI 或新成员在执行某项需求前，必须先通过 `09-结构化总索引与交叉映射` 确认：

- 任务属于哪一类问题
- 唯一权威正文是哪篇
- 该需求落在哪个页面、数据对象与验收项
- 页面、实体、规则之间的下游关系是什么
- 哪些内容在 P0 明确不做，禁止自动脑补

### R-7：Raw 输入不可直接执行

原始 PRD、会议纪要、聊天记录、示例截图只能先进入 `_raw/00-inbox.md`。在形成最小中间产物前，不得直接开工：

- `requirements.ir`：结构化需求条目；若仓库中已存在对应原型页，优先嵌入该页的 `P0-CONTRACT*` / `MIGRATION-MAPPING*`，不额外扩写平行文档
- `ambiguities`：歧义与未决问题清单
- `boundary`：冻结后的 in-scope / out-of-scope / acceptance

### R-8：完成必须可追踪

任何 P0 任务只有在能回写到“需求 ID → 页面 / 数据 / 验收 → 代码 / 测试 / 证据”时才算完成。没有 traceability 的交付，不得标记为 done。

## P0 需求编译流水线（最小闭环）

| 阶段 | 做什么 | 最小产物 | 强门禁 |
|------|--------|---------|--------|
| Ingest | 收集原始 PRD / 会议纪要 / 讨论结论 | `_raw/00-inbox.md` 条目 | 只追加，不执行 |
| Extract | 把自然语言拆成目标 / 规则 / 示例 / 不做 / 未决项 | `requirements.ir`（已有原型时优先落到 `P0-CONTRACT*` / `MIGRATION-MAPPING*`） | 示例不得自动升级为规则 |
| Clarify | 暴露并关闭 P0 / P1 歧义 | `ambiguities` | 有高优先级未决项，不得进入冻结 |
| Freeze | 明确做什么、不做什么、如何验收 | `boundary` | 缺少 `out_of_scope`，不得冻结 |
| Execute | 任务绑定需求 ID 并按冻结边界执行 | 任务卡 / 实现记录 / 原型契约回写 | 只认冻结契约，不认记忆化 PRD |
| Verify / Write-back | 回写需求 → 实现 → 验收证据 | `traceability`、`_output/00-outputs.md` | 无映射证据，不算完成 |
