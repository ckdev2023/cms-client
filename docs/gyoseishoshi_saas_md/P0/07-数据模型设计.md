## 1. P0 设计原则

- **以案件为中心**：阶段（Stage）、门槛（Gate）、快照（SubmissionPackage）拆开落地
- **客户资料可复用**：一个客户可多案；多案件可共享附件版本，但提交包引用必须锁定具体版本
- **资料项是要求，附件是版本**：每次上传生成新版本，不覆盖历史
- **文书版本不可变**：提交包引用固定版本
- **内部域与客户可见域隔离**：通过 `visible_scope` 标记从首版开始分离
- **全量关键事件可留痕**：审计日志覆盖所有敏感动作

### P0 策略对模型的直接影响

| 策略 | 对模型的影响 |
|------|------------|
| 不引入 Company 实体 | 工作类案件雇主信息沉淀在 Case.employer_* 字段 |
| 不引入 ClientUser | P0 无客户门户；`visible_scope` 标记仍必须存在 |
| 模板预置不可自定义 | CaseTemplate 预置 2 类，不做事务所级编辑 |
| 双人复核默认关闭 | ReviewRecord 预留；通过 CaseTemplate.review_required_flag 控制 |
| 收费阻断默认 warn | 通过 CaseTemplate.billing_gate_mode 控制 |

### 跨实体通用字段

所有核心业务实体统一具备：

- `org_id`：租户隔离
- `created_by`、`updated_by`、`created_at`、`updated_at`
- 编号类字段（`lead_no`、`customer_no`、`case_no`、`submission_no`）组织内唯一，保留人工可读格式

### 双层状态模型（可扩展性核心原则）

为保证 P1/P2 新签证类型（如经营管理签证）接入时不破坏任何 P0 现有逻辑，模型采用「双层状态」设计：

| 层 | 字段 | 颗粒度 | 用途 |
|---|---|---|---|
| **管理层** | `Case.stage`（S1–S9） | 粗，通用，跨签证类型 | 报表、仪表盘、SLA、Gate 触发 |
| **业务层** | `CaseWorkflowStep.step_code`（P1 预定义实体，见 §3.24） | 细，模板专属，每类签证自定义 | 承办人操作界面、签证类型专属流程节点 |

- **管理层不感知签证细节**：S1–S9 永远是 CaseTemplate 无关的通用协作视图
- **业务层由 CaseTemplate 驱动**：新签证类型 = 新增一条 CaseTemplate 并填写 `workflow_steps_blueprint`，无需修改主表枚举
- **P0 两类模板降级运行**：`workflow_steps_blueprint = null` 时系统只用 S1–S9，家族滞在 / 技人国不受影响

## 2. P0 实体总览

```
Lead ──转化──→ Customer ──建案──→ Case
                  │                 ├── CaseParty（关键关系人）
                  │                 ├── CaseCollaborator（协作成员）
                  │                 ├── CaseStageHistory（阶段记录）
                  │                 ├── CaseWorkflowStep（业务子步骤，P1 预定义）
                  │                 ├── DocumentRequirement → DocumentFileVersion
                  │                 ├── GeneratedDocumentVersion ← Template
                  │                 ├── ValidationRun → ReviewRecord（可选）
                  │                 ├── SubmissionPackage → SubmissionPackageItem
                  │                 ├── BillingPlan → PaymentRecord
                  │                 ├── Task
                  │                 ├── CaseDeadline
                  │                 ├── CommunicationLog
                  │                 └── Reminder/Notification
                  └── CustomerRelation
                  └──────────────→ ResidencePeriod（在留期间历史，P1 预定义）

Group ← UserGroupMembership → User / Role
CaseTemplate（预置模板）
AuditLog（全对象通用）
```

| 实体 | P0 角色 |
|------|---------|
| Group | 归属团队，访问控制核心维度 |
| UserGroupMembership | 用户与 Group 的多对多关系 |
| Lead | 咨询线索，签约前跟进 |
| Customer | 个人客户主数据 |
| CustomerRelation | 客户侧关联人关系（检索跳转用，不作为门槛口径） |
| Case | 核心工作单元，案件全生命周期 |
| CaseCollaborator | 案件协作成员与跨组授权 |
| CaseParty | 案件关联人（门槛校验与提交包锁定） |
| CaseStageHistory | 阶段流转记录 |
| **CaseWorkflowStep** | **业务子步骤记录（P1 预定义，见 §3.24）——承载模板专属流程节点** |
| CaseTemplate | 预置案件模板（驱动资料清单、校验规则、文书建议） |
| DocumentRequirement | 资料项 = 材料要求（含问卷类，P1 扩展 survey_data） |
| DocumentFileVersion | 资料附件版本（不可变） |
| CommunicationLog | 沟通记录（内部/客户可见分域） |
| Task | 任务（驱动案件流转） |
| CaseDeadline | 期限（统一建模） |
| Template | 文书模板（版本化） |
| GeneratedDocumentVersion | 生成文书版本（不可变） |
| ValidationRun | 校验运行记录 |
| ReviewRecord | 双人复核（预留，默认关闭） |
| SubmissionPackage | 提交包 = 对外提交快照 |
| SubmissionPackageItem | 提交包锁定引用 |
| BillingPlan | 收费计划 / 节点 |
| PaymentRecord | 回款记录 |
| Reminder / Notification | 提醒与触达记录（P0 仅站内） |
| AuditLog | 审计日志 |
| User / Role | 内部用户与角色 |
| **ResidencePeriod** | **在留期间历史记录（P1 预定义，见 §3.25）——支持多次续签追溯** |

## 3. P0 实体定义

### 3.0 Group（归属团队/项目组）

> 治理规则见 [03 §12](03-业务规则与不变量.md#12-group-治理)。

P0 角色：所有客户和案件的归属维度，驱动工作台筛选与访问控制。

- id
- group_no（组织内可读编号，可选）
- name
- description（可选）
- active_flag
- created_by / created_at / updated_by / updated_at

### 3.0A UserGroupMembership（用户与 Group 关系）

P0 角色：定义内部用户属于哪些 Group，用于工作台默认范围与可见性判断。

- id
- user_id
- group_id
- is_primary_group
- active_flag
- joined_at
- left_at（可选）

### 3.1 Lead（咨询线索）

P0 角色：承接签约前的早期信息与跟进，转化时生成 Customer 与 Case。

- id
- lead_no
- name
- phone / email
- source_channel（来源/介绍人/渠道）
- intended_case_type（意向业务类型）
- group_id（归属团队，转化时默认继承至 Customer → Case）
- status（→ [03 §3.6](03-业务规则与不变量.md#36-线索状态)）
- next_action / next_follow_up_at
- quote_amount（可选）
- note
- owner_user_id
- converted_customer_id / converted_case_id（转化后回填）
- lost_reason（可选）
- created_at / updated_at

### 3.2 Customer（个人客户）

> P0 以个人客户为主体，不引入企业客户实体。

- id
- customer_no
- display_name（对内识别名）
- name_cn / name_jp / name_en / name_kana（フリガナ）
- gender / nationality / birth_date
- phone / email / address
- passport_no（可选）
- residence_card_no（可选）
- current_status_of_residence（可选）
- residence_expiry_date（可选）
- profile_photo_url（可选）
- referrer_name（可选）
- group_id（归属团队，P0 强依赖）
- note
- owner_user_id
- created_at / updated_at

### 3.2A CustomerRelation（客户关系映射）

P0 角色：客户侧关联人展示、跳转、批量建案聚合。不作为案件门槛口径。

- id
- from_customer_id / to_customer_id
- relation_type（spouse / child / parent / agent / other）
- note（可选）
- created_by / created_at / updated_at

### 3.5 Case（案件）

> 案件阶段定义见 [03 §3.1](03-业务规则与不变量.md#31-案件阶段)。归属继承链见 [03 §2.2](03-业务规则与不变量.md#22-归属继承链)。

P0 角色：系统核心工作单元，承载从建档到归档的完整闭环。

- id
- case_no / case_name
- source_lead_id（来源线索，可选）
- case_type（家族滞在 / 工作签证-技人国）
- application_type（认定 / 变更 / 更新）
- customer_id（主申请人）
- group_id（归属快照，默认继承 Customer.group_id；转组须留痕）
- employer_name（工作类案件雇主名称，可选）
- employer_corporate_number（可选）
- employer_address（可选）
- employer_contact_name / employer_contact_phone（可选）
- stage（S1–S9，→ [03 §3.1](03-业务规则与不变量.md#31-案件阶段)）
- current_workflow_step（当前业务子步骤 code，冗余字段便于列表查询；源头为 CaseWorkflowStep 表；**P1 启用，P0 为 null**）
- extra_fields（JSON：存储模板专属字段值，由 CaseTemplate.extra_fields_schema 定义；如 `visa_plan`、`coe_issued_date`；**P1 启用，P0 为 null**）
- priority / risk_level
- principal_user_id（负责人）
- primary_assistant_user_id（主协作/助理，可选）
- source_channel
- signed_at（签约时间，可选）
- result_outcome（pending / approved / rejected / withdrawn，可选）
- close_reason / archive_reason / archived_at（可选）
- next_action / next_action_due_at（工作台聚合用）
- latest_validation_run_id / latest_validation_status（工作台聚合用，可选）
- has_blocking_issue_flag / blocking_count_cached / warning_count_cached（可选）
- next_deadline_due_at（关键期限最近到期，聚合用，可选）
- billing_unpaid_amount_cached（未结清金额，聚合用，可选）
- billing_risk_acknowledged_at / billing_risk_acknowledged_by（欠款风险确认，可选）
- created_at / updated_at

> P0 不启用 `company_id`（Company 实体不存在）。雇主信息通过 `employer_*` 字段沉淀。
> P0 中 `current_workflow_step` 与 `extra_fields` 写入 null，列表查询忽略这两列；P1 新增签证类型时同步启用。

### 3.5A CaseCollaborator（案件协作成员）

P0 角色：支持多人协作与跨组授权，叠加 Group 可见性。

- id
- case_id / user_id
- role_type（principal / collaborator / viewer）
- active_flag
- granted_by / granted_at
- reason（跨组协作原因，可选）
- created_at / updated_at

### 3.6 CaseStageHistory（阶段流转记录）

- id
- case_id
- from_stage（首条可空）/ to_stage
- reason（可选）
- changed_by / changed_at

### 3.7 CaseParty（案件关联人）

> 与 CustomerRelation 的区别见 [08-术语表 · CaseParty](08-术语表.md#caseparty)。

P0 角色：门槛校验依据、文书变量来源、提交包快照锁定对象。

- id
- case_id
- party_role（主申请人 / 扶养者 / 保证人 / 配偶 / 子女 / 企业担当等）
- customer_id（可选）
- is_primary
- required_flag（是否关键关系，影响硬性阻断）
- created_at / updated_at

### 3.8 CaseTemplate（案件模板）

P0 角色：预置 2 类模板（家族滞在、工作签证/技人国），驱动资料清单与校验规则。P1 起通过三个扩展 blueprint 字段承载新签证类型的业务流程与专属字段定义。

- id
- template_name
- case_type / application_type（可选）
- requirement_blueprint（JSON：资料项定义、提供方、是否必交、截止策略等）
- default_tasks_blueprint（JSON，可选）
- validation_ruleset_ref（JSON：校验项集合与严重性）
- review_required_flag（是否要求双人复核；P0 默认 false）
- billing_gate_mode（off / warn / block；P0 默认 warn）
- workflow_steps_blueprint（JSON：业务子步骤定义列表，含 step_code / label / parent_stage / can_loop_to / billing_gate 配置；**P1 启用，P0 两类模板为 null**）
- extra_fields_schema（JSON：模板专属案件字段定义，如 `visa_plan`（enum: 1YEAR/4PLUS1）/ `coe_issued_date` / `entry_date` 等；**P1 启用，P0 为 null**）
- reminder_schedule_blueprint（JSON：在留到期提醒间隔配置，如 `{"residence_expiry": [180, 90, 30]}`；**P1 启用，P0 手动设置 Reminder**）
- active_flag
- created_at / updated_at

> P1 新增经营管理签证模板时，三个 blueprint 字段示例：
> `workflow_steps_blueprint` 定义 COE_SENT 步骤并绑定 `billing_gate: {mode: block, milestone: final_payment}`；
> `extra_fields_schema` 定义 `visa_plan`（必填）、`coe_issued_date`、`entry_date`；
> `reminder_schedule_blueprint` 定义 `residence_expiry` 到期前 180/90/30 天批量生成 Reminder。

### 3.9 DocumentRequirement（资料项）

> 资料项治理规则见 [03 §7](03-业务规则与不变量.md#7-资料项治理)。

P0 角色：描述"需要什么材料"的要求，按提供方分组，驱动完成率统计与催办。P1 扩展支持问卷类资料项（category=questionnaire），用于承载经营管理签证等需要结构化问卷数据的场景。

- id
- case_id
- category（`client_doc`客户资料 / `dependent_doc`扶养者资料 / `internal_doc`事务所内部 / `questionnaire`问卷，**P1 新增**）
- item_name / item_code（模板内唯一 key）
- required_flag
- provided_by_role（主申请人 / 扶养者或保证人 / 事务所内部）
- assignee_user_id（责任人，可选）
- due_date（补件截止，可选）
- status（→ [03 §3.2](03-业务规则与不变量.md#32-资料项状态)）
- client_visible_flag / client_action_required
- latest_version_id（便于查询，可选）
- last_reminded_at（可选）
- review_comment_latest（可选）
- waive_reason_latest / waive_reason_code_latest / waived_by_user_id_latest / waived_at_latest（可选）
- survey_data（JSON：问卷类资料项的结构化答案，含 personal_info / education / work_experience / financial_info / visa_history / business_plan / family_info 等子块；category=questionnaire 时使用；**P1 启用，P0 为 null**）
- created_at / updated_at

> `questionnaire` 类资料项天然参与资料清单完成率、审核流程、提交包快照锁定，无需额外实体。P2 如需问卷引擎，可将 survey_data 迁移为独立 CaseSurvey 表。

### 3.14 Template（文书模板）

> 文书规则见 [03 §8](03-业务规则与不变量.md#8-文书规则)。

P0 角色：预置文书模板（委任状、理由书、申请表等），包含变量字典。

- template_name
- case_type / doc_type / language
- version_no
- content_body
- variables_schema（JSON：变量字典与格式化规则）

### 3.24 CaseWorkflowStep（业务子步骤记录，P1 预定义）

> **P0 不启用**（当前在留期存于 Customer.residence_expiry_date）。P1 新增，支持多次续签的完整历史追溯与自动提醒批量生成。

P1 角色：记录客户每次成功取得在留资格后的完整期间信息；写入时根据 CaseTemplate.reminder_schedule_blueprint 批量创建 CaseDeadline 与 Reminder，并同步更新 Customer.residence_expiry_date。

- id
- customer_id
- case_id（产生此次在留期的案件，外键可溯源）
- residence_status（在留资格类型，如 `BUSINESS_MANAGER`）
- period_start / period_end
- residence_years
- card_number（可选）
- entry_date

## 5. P0 关键关系说明

- 一个 User 通过 UserGroupMembership 关联一个或多个 Group（一个标记为主 Group）
- 一个 Group 可关联多个 Customer 与 Case，用于归属、筛选与可见性
- 一个 Lead 可转化为一个 Customer 并创建一个或多个 Case；Lead.group 作为默认归属来源
- 一个 Customer 对应多个 Case；Case.group 默认继承 Customer.group 作为归属快照，后续不回写
- 一个 Case 可有多个 CaseParty（主申请人 + 关键关系人），门槛校验依赖其完整性
- 一个 Case 可有多个 CaseCollaborator，用于协作授权与可见性叠加
- 一个 Case 基于 CaseTemplate 生成多条 DocumentRequirement 与默认 Task
- 一个 DocumentRequirement 可有多个 DocumentFileVersion；提交包只引用具体版本
- 一个 Case 可有多个 GeneratedDocumentVersion；提交包只引用具体版本
- 一个 Case 可有多条 ValidationRun 与 ReviewRecord；提交包锁定当次引用
- 一个 Case 可有多个 SubmissionPackage；补正包通过 related_submission_id 关联原包
- BillingPlan 与 PaymentRecord 关联到 Case，可与阶段/门槛联动
- **（P1）** 一个 Case 可有多条 CaseWorkflowStep，记录模板专属业务步骤的进入 / 完成历史；`Case.current_workflow_step` 是当前步骤的冗余快照
- **（P1）** 一个 Customer 可有多条 ResidencePeriod，记录历次在留期间；每条 ResidencePeriod 关联产生它的 Case，并批量触发 CaseDeadline 与 Reminder 生成
