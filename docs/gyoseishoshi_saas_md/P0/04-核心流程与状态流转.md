## 0. 结构化速查（人 / AI 共用）

### 0.1 P0 主链路

`咨询录入 → 客户建档 → 案件创建 → 资料收集与审核 → 文书生成与定稿 → 提交前校验 → 正式提交 → 审理跟踪 / 补正 → 出结果 → 归档`

### 0.2 管理层阶段（唯一权威）

| 阶段 | 代号 | 进入条件 |
|---|---|---|
| 已建档 | `S1` | 已签约/已立案，主申请人、案件类型、负责人已明确 |
| 资料收集中 | `S2` | 资料清单已生成并发出至少一轮资料请求 |
| 资料审核中 | `S3` | 已有客户上传/登记资料，内部开始审核 |
| 文书制作中 | `S4` | Gate-A 通过 |
| 待校验 | `S5` | Gate-B 通过 |
| 待提交 | `S6` | 最新校验通过，复核要求已满足（如启用） |
| 已提交审理中 | `S7` | Gate-C 通过，且已生成提交包 |
| 已出结果 | `S8` | 已登记结果与结果日期 |
| 已归档 | `S9` | 案件已结案并完成归档动作 |

### 0.3 Gate 触发摘要

| Gate | 触发点 | 通过后动作 |
|---|---|---|
| Gate-A | `S3 → S4` 或首次生成关键文书前 | 进入文书制作，生成/更新文书任务 |
| Gate-B | `S4 → S5` 或主动进入校验 | 生成 `ValidationRun` 校验报告 |
| Gate-C | 点击“正式提交” | 生成 `SubmissionPackage`，阶段进入 `S7` |

### 0.4 P0 / P1 边界摘要

- **P0 只使用管理层阶段 `S1–S9`**
- **P1 才启用业务子步骤 `CaseWorkflowStep`**，如 `COE_SENT` / `VISA_APPLYING`
- **P0 续签提醒以手动 Reminder 为主**；P1 才通过 `ResidencePeriod + reminder_schedule_blueprint` 批量生成

## 1. P0 主链路总览

P0 的核心是一条从咨询到归档的主链路，试点期间所有案件沿这条链路运行：

```text
咨询录入 → 客户建档 → 案件创建 → 资料收集与审核 → 缺件催办
                                                         ↓
归档 ← 出结果 ← 审理跟踪 ← 正式提交 ← 提交前校验 ← 文书生成与定稿
                   ↑                                     ↑
                   └──── 补正循环 ────────────────────────┘
```

### 1.1 三类落地机制

| 机制 | 作用 | 详情 |
|------|------|------|
| **案件阶段（Stage）** | `S1–S9`，驱动工作台视图与默认任务 | 本文 §2 |
| **门槛校验（Gate）** | Gate-A/B/C 三道关卡，阻断关键错误 | 本文 §3 |
| **提交包（SubmissionPackage）** | 固化每一次对外提交快照 | [03 §2.4](03-业务规则与不变量.md#24-提交包锁定与不可覆盖) |

### 1.2 P0 策略口径

- 双人复核默认不强制；仅在事务所或案件模板明确开启时作为 `S5 → S6` 的附加门槛
- 收费节点用于提醒与风险提示，不默认阻断提交动作
- 客户门户不纳入首版主流程，P0 资料收集由事务所内部代为上传或线下完成
- 首版模板固定为：家族滞在、工作签证（技人国）

## 2. 案件阶段（S1–S9）

> 阶段枚举正式定义见 [03 §3.1](03-业务规则与不变量.md#31-案件阶段)。

| 代号 | 阶段 | 进入条件 | 典型动作 |
|------|------|---------|---------|
| `S1` | 已建档 | 已签约/已立案，主申请人、案件类型、负责人已明确 | 确认客户信息，生成资料清单 |
| `S2` | 资料收集中 | 资料清单已生成并发出至少一轮资料请求 | 等待客户提交/内部登记，催办 |
| `S3` | 资料审核中 | 已有客户提交或内部登记资料，开始审核 | 审核/退回/标记豁免 |
| `S4` | 文书制作中 | **Gate-A 通过** | 生成模板文书，整理定稿 |
| `S5` | 待校验 | **Gate-B 通过** | 执行校验，生成校验报告 |
| `S6` | 待提交 | 最新校验通过，复核要求已满足（如启用） | 准备提交信息 |
| `S7` | 已提交审理中 | **Gate-C 通过**，已生成提交包 | 等待结果，处理补正 |
| `S8` | 已出结果 | 已登记结果与结果日期 | 通知客户，确认后续动作 |
| `S9` | 已归档 | 案件已结案 | 资料归档，续签提醒确认 |

阶段用于“管理与协作”，不直接等于“资料是否齐备”。阻断由门槛校验（Gate）负责。

---

## 3. Gate 触发点与通过后动作

> Gate 的硬性阻断/软性提示明细与重新校验触发条件见 [03 §4](03-业务规则与不变量.md#4-校验门槛gate规则)。

### 3.1 Gate-A：进入文书（`S3 → S4`）

- **触发点**：从 `S3 → S4`，或首次生成关键文书前
- **通过后动作**：生成/更新“文书制作”相关默认任务；案件概览显示“可进入文书”状态

### 3.2 Gate-B：进入提交前校验（`S4 → S5`）

- **触发点**：从 `S4 → S5`，或用户主动进入校验
- **通过后动作**：生成校验报告（`ValidationRun`），作为复核与提交包的引用来源

### 3.3 Gate-C：提交（生成提交包前）

- **触发点**：点击“正式提交”动作
- **通过后动作**：生成提交包并锁定引用版本与关键字段快照；阶段进入 `S7`

## 4. 分阶段流程剧本

### 4.1 咨询转案件

1. 新建咨询线索，录入基础信息与归属 Group
2. 跟进咨询，推进到待签约/已签约
3. 签约后从线索创建正式案件：选择案件类型/签证类型、主申请人、负责人
4. 系统自动生成资料清单和初始任务

关键规则：

- 线索转客户：电话/邮箱优先去重匹配，支持人工确认合并
- 家族签/扶养类案件：必须同时绑定关键关系人（扶养者/保证人）作为 `CaseParty`
- `Case.group` 默认继承 `Customer.group` 作为案件归属快照

### 4.2 资料收集与审核

1. 案件创建后，系统根据“签证类型 × 申请类型”匹配模板并生成资料清单
2. 标记各资料项的提供方（主申请人 / 扶养者或保证人 / 事务所内部）
3. 对客户发出资料请求（资料项进入 `waiting_upload`）
4. 客户/内部提交资料，事务所完成扫描归档，并在系统**登记资料版本**
5. 助理审核：通过 → `approved`；不通过 → `revision_required`
6. 不通过的资料项触发催办提醒
7. 客户补交后回到审核环节，循环直到资料齐备

资料项状态主路径：

```text
not_sent → waiting_upload → uploaded_reviewing → approved
                                                → revision_required → waiting_upload
任一状态 → waived
approved → expired
```

### 4.3 文书生成与定稿

1. **Gate-A 通过**后进入 `S4`
2. 系统基于模板和案件变量生成文书初稿
3. 助理整理，行政书士确认
4. 定稿（文书状态 → `final`）
5. 进入提交前校验

### 4.4 提交前校验与提交

1. **Gate-B 通过**后进入 `S5`
2. 系统执行校验，生成校验报告（`ValidationRun`）
3. 校验通过：
   - 如未启用复核 → 直接进入 `S6`
   - 如启用复核 → 等待复核签名通过后进入 `S6`
4. **Gate-C 通过**后生成提交包
5. 提交包锁定：本次提交引用的资料版本、文书版本、关键字段快照、校验报告、复核记录（如有）
6. 阶段进入 `S7`

串联规则：

- 提交前必须引用“最新一次有效校验结果”
- 关键资料版本、关键字段或定稿文书变化后应重新校验
- 无论首次还是补正提交，都必须创建新的提交包

### 4.5 审理跟踪与补正

1. 案件进入 `S7` 已提交审理中
2. 等待入管局审理结果
3. 如收到补正通知：
   - 创建补正任务，更新补正截止日
   - 补正资料审核与文书更新
   - 补正完成后重新经过 **Gate-B → Gate-C** 生成补正提交包
4. 补正提交包与原提交包关联，形成可追溯链路
5. 许可 / 不许可 → 进入 `S8`

冻结口径：

- 收到补正通知后，主阶段默认仍保持在 `S7`
- 补正是已提交案件中的一次新循环，不回退成“未提交案件”
- 补正提交包必须通过 `related_submission_id` 关联原提交包

### 4.6 归档与续签提醒

- `S8` 出结果后确认后续动作 → 进入 `S9`
- 归档前确认：资料归档完成、续签提醒是否需要、收费是否结清
- 系统根据在留到期日在关键节点触发提醒（P0 以站内提醒为主）
- P0 不做续签商机自动创建——后置到 P1

### 4.7 收费流程

1. 案件创建时设定费用总额与收费节点
2. 到节点触发内部收款提醒
3. 财务登记实收、上传凭证
4. 标记回款/欠款状态

P0 口径：

- 收费异常以风险提示为主，不阻断提交
- 欠款状态下继续提交需风险确认留痕
- 工作台聚合暴露欠款风险案件

## 5. 服务端执行口径（AI / 开发实现必须遵守）

- 用户在页面上点击“推进阶段”“执行校验”“生成提交包”时，服务端必须重新校验 Gate 条件
- 即使页面缓存仍显示“可提交”，只要关键资料版本、关键字段或文书已变化，服务端也必须拒绝直接生成提交包并要求重新校验
- 阶段只是协作视图，不得成为绕开 Gate 的替代通道
- P0 不得通过伪造阶段值跳过 `ValidationRun` 或 `SubmissionPackage`

## 6. P0 明确不做（流程层）

| 不做项 | 后置到 | 理由 |
|--------|-------|------|
| 双人复核全局强制 | P1 | 模型预留，默认关闭 |
| 自动化流程引擎与条件触发器 | P2 | P0 流程推进由用户手动驱动 |
| 客户门户交互流程 | P1 | P0 聚焦内部作业闭环 |
| 续签商机自动创建 | P1 | P0 仅做到期提醒 |
| 文书初稿/审核版/定稿完整版本管理流程 | P1 | P0 用三态流转 |
| 高级催办自动升级（多级主管） | P1 | P0 仅支持基础升级 |
| 看板视图与高级工作台 | P1 | P0 用列表 + 基础聚合 |
| 业务子步骤视图（CaseWorkflowStep） | P1 | P0 只用 `S1–S9` 管理阶段；新签证类型上线时再通过 `CaseTemplate.workflow_steps_blueprint` 驱动 |
| 在留期间历史自动提醒批量生成 | P1 | P0 仅做手动 Reminder；P1 通过 `ResidencePeriod + reminder_schedule_blueprint` 自动触发 |
