## 1. P0 主链路总览

P0 的核心是一条从咨询到归档的主链路，试点期间所有案件沿这条链路运行：

```
咨询录入 → 客户建档 → 案件创建 → 资料收集与审核 → 缺件催办
                                                         ↓
归档 ← 出结果 ← 审理跟踪 ← 正式提交 ← 提交前校验 ← 文书生成与定稿
                   ↑                                     ↑
                   └──── 补正循环 ────────────────────────┘
```

### 1.1 三类落地机制

流程通过三类机制落地为可执行系统：

| 机制 | 作用 | 详情 |
|------|------|------|
| **案件阶段（Stage）** | S1–S9，驱动工作台视图与默认任务 | 本文 §1.2 |
| **门槛校验（Gate）** | Gate-A/B/C 三道关卡，阻断关键错误 | 本文 §1.4 |
| **提交包（Submission Package）** | 固化每一次对外提交快照 | [03 §2.4](03-业务规则与不变量.md#24-提交包锁定与不可覆盖) |

### 1.2 案件阶段（S1–S9）

> 阶段枚举正式定义见 [03 §3.1](03-业务规则与不变量.md#31-案件阶段)。

| 代号 | 阶段 | 进入条件 | 典型动作 |
|------|------|---------|---------|
| S1 | 已建档 | 已签约/已立案，主申请人、案件类型、负责人已明确 | 确认客户信息，生成资料清单 |
| S2 | 资料收集中 | 资料清单已生成并发出至少一轮资料请求 | 等待客户上传，催办 |
| S3 | 资料审核中 | 已有客户上传资料，内部开始审核 | 审核/退回/标记豁免 |
| S4 | 文书制作中 | **Gate-A 通过** | 生成模板文书，整理定稿 |
| S5 | 待校验 | **Gate-B 通过** | 执行校验，生成校验报告 |
| S6 | 待提交 | 最新校验通过，复核要求已满足（如启用） | 准备提交信息 |
| S7 | 已提交审理中 | **Gate-C 通过**，已生成提交包 | 等待结果，处理补正 |
| S8 | 已出结果 | 已登记结果与结果日期 | 通知客户，确认后续动作 |
| S9 | 已归档 | 案件已结案 | 资料归档，续签提醒确认 |

阶段用于"管理与协作"，不直接等于"资料是否齐备"。阻断由门槛校验（Gate）负责。

### 1.3 P0 策略口径

> 详细冻结口径见 [03 §1](03-业务规则与不变量.md#1-p0-版本策略冻结)。

- 双人复核默认不强制；仅在事务所或案件模板明确开启时作为 S5→S6 的附加门槛
- 收费节点用于提醒与风险提示，不默认阻断提交动作
- 客户门户不纳入首版主流程，P0 资料收集由事务所内部代为上传或线下完成
- 首版模板固定为：家族滞在、工作签证（技人国）

### 1.4 Gate 触发点与通过后动作

> Gate 的硬性阻断/软性提示明细与重新校验触发条件见 [03 §4](03-业务规则与不变量.md#4-校验门槛gate规则)。
> 本节仅定义 Gate 在流程中的触发点与通过后动作。

#### Gate-A：进入文书（S3 → S4）

- **触发点**：从 S3 → S4，或首次生成关键文书前
- **通过后动作**：生成/更新"文书制作"相关默认任务；案件概览显示"可进入文书"状态

#### Gate-B：进入提交前校验（S4 → S5）

- **触发点**：从 S4 → S5，或用户主动进入校验
- **通过后动作**：生成校验报告（ValidationRun），作为复核与提交包的引用来源

#### Gate-C：提交（生成提交包前）

- **触发点**：点击"正式提交"动作
- **通过后动作**：生成提交包并锁定引用版本与关键字段快照；阶段进入 S7

## 2. 咨询转案件

### 流程步骤

1. 新建咨询线索，录入基础信息（联系方式、来源/介绍人、意向类型）与归属 Group
2. 跟进咨询，推进到待签约/已签约
3. 签约后从线索创建正式案件：选择案件类型/签证类型、主申请人（可多选批量生成）、负责人
4. 系统自动生成资料清单和初始任务

### 转化规则

- 线索转客户：电话/邮箱优先去重匹配，支持人工确认合并
- 线索转客户：默认继承 Lead.group；如改组，必须记录原因
- 家族签/扶养类案件：必须同时绑定关键关系人（扶养者/保证人）作为案件关联人（CaseParty）
- Case.group 默认继承 Customer.group 作为案件归属快照

> 线索状态枚举见 [03 §3.6](03-业务规则与不变量.md#36-线索状态)。
> 归属继承规则见 [03 §2.2](03-业务规则与不变量.md#22-归属继承链)。

## 3. 资料收集与审核

### 流程步骤

1. 案件创建后，系统根据"签证类型 × 申请类型"匹配模板并生成资料清单
2. 标记各资料项的提供方（主申请人 / 扶养者或保证人 / 事务所内部）
3. 对客户发出资料请求（资料项进入 `waiting_upload`）
4. 客户/内部上传资料（进入 `uploaded_reviewing`）
5. 助理审核：通过 → `approved`；不通过 → `revision_required`（必须填写补正原因）
6. 不通过的资料项触发催办提醒
7. 客户补交后回到审核环节，循环直到资料齐备
8. 可在案件内手动追加资料项或标记无需提供（`waived`，需留痕）

### 资料请求已发出的认定标准

P0 无客户门户，首版允许通过系统外渠道（电话 / 邮件 / IM / 面谈）向客户催取资料，但要想让资料项进入 `waiting_upload`，必须至少满足以下之一：

- 在资料项上执行“发出请求”动作并记录责任人、时间；
- 创建一条关联沟通记录，且明确对应资料项或资料类别；
- 由系统基于模板首次批量生成资料请求任务并留痕。

换言之，**现实沟通可以在线下发生，但状态变化不能完全脱离系统留痕**。

### 资料项状态转移

> 状态枚举见 [03 §3.2](03-业务规则与不变量.md#32-资料项状态)。

```
not_sent → waiting_upload → uploaded_reviewing → approved
                                                → revision_required → waiting_upload（补正循环）
任一状态 → waived（需留痕原因）
approved → expired（时效不足时重新收集）
```

### 关键规则

- 每次上传生成新附件版本，不覆盖历史
- 标记 `waived` 必须记录原因、操作人、操作时间
- 完成率 = 已审核通过的必交资料项数 / 必交资料项总数（`waived` 剔除分母）

> 完成率口径与 waived 治理规则见 [03 §7](03-业务规则与不变量.md#7-资料项治理)。

## 4. 文书生成与定稿

### 流程步骤

1. **Gate-A 通过**后进入 S4 文书制作中
2. 系统基于模板和案件变量生成文书初稿
3. 助理整理，行政书士确认
4. 定稿（文书状态 → `final`）
5. 进入提交前校验

### P0 文书状态转移

> 状态枚举见 [03 §3.3](03-业务规则与不变量.md#33-文书状态)。

```
draft → final → exported
```

P0 简化为三态流转：草稿 → 定稿 → 导出。定稿后的文书版本用于提交包锁定引用。文书导出后的改动不影响已锁定版本。

## 5. 提交前校验与提交

### 校验流程

1. **Gate-B 通过**后进入 S5 待校验
2. 系统执行校验，生成校验报告（ValidationRun）
3. 校验通过：
   - 如未启用复核 → 直接进入 S6 待提交
   - 如启用复核 → 等待复核签名通过后进入 S6
4. 校验不通过 → 修复问题后重新校验

### 复核（P0 可选）

- 复核由事务所/案件模板决定是否开启，P0 默认关闭
- 复核内容：校验报告 + 关键材料清单 + 文书版本号
- 通过 → 进入 S6 待提交
- 驳回 → 根据原因回退到 S3（资料问题）、S4（文书问题）或 S5（重新校验）

### 提交流程

1. **Gate-C 通过**后生成提交包
2. 提交包锁定：本次提交引用的资料版本、文书版本、关键字段快照、校验报告、复核记录（如有）
3. 记录提交日期、受理机关/窗口、回执/凭证
4. 阶段进入 S7 已提交审理中

### 校验、复核与提交的串联规则

- 提交前必须引用"最新一次有效校验结果"
- 关键资料版本、关键字段或定稿文书变化后应重新校验
- 复核驳回后不应停留在 S6，应根据原因回退
- 无论首次还是补正提交，都必须创建新的提交包

### 服务端执行口径

- 用户在页面上点击“推进阶段”“执行校验”“生成提交包”时，服务端必须重新校验 Gate 条件。
- 即使页面缓存仍显示“可提交”，只要关键资料版本、关键字段或文书已变化，服务端也必须拒绝直接生成提交包并要求重新校验。
- 阶段只是协作视图，不得成为绕开 Gate 的替代通道。

> 重新校验触发条件见 [03 §4.4](03-业务规则与不变量.md#44-重新校验触发条件)。
> 提交动作强规则见 [03 §13](03-业务规则与不变量.md#13-提交动作强规则)。

## 6. 审理跟踪与补正

### 流程步骤

1. 案件进入 S7 已提交审理中
2. 等待入管局审理结果
3. 如收到补正通知：
   - 创建补正任务，更新补正截止日
   - 补正资料审核与文书更新
   - 补正完成后重新经过 Gate-B → Gate-C 生成补正提交包
4. 补正提交包与原提交包关联，形成可追溯链路
5. 审理期间支持记录来函、追加材料等事件，沉淀到案件时间线

### 补正操作剧本（P0 冻结口径）

- 收到补正通知后，案件主阶段默认仍保持在 **S7 已提交审理中**；补正是已提交案件中的一次新循环，不回退成“未提交案件”。
- 系统必须创建至少一条补正任务，并记录补正截止日、责任人、通知来源。
- 若补正影响资料项，则相关资料项进入 `revision_required` / 新增资料项；若补正影响文书，则需生成新文书版本并重新定稿。
- 补正完成后，重新经过 **Gate-B → Gate-C**，生成 `submission_kind = supplement` 的新提交包。
- 补正提交包必须通过 `related_submission_id` 关联原提交包，保证首次提交与补正链路可追溯。

### 结果处理

- 许可 / 不许可 → 进入 S8 已出结果
- 记录结果日期、结果详情

### 异常结案

- 客户撤回、改委托、长期中止等异常情况：记录结果状态、结案原因，不额外膨胀主阶段
- 异常结案后应自动关闭未完成的内部任务，并确认是否保留后续提醒
- 如需重新启动，应基于原案件派生新案件，不直接覆盖历史阶段与提交记录

## 7. 归档与续签提醒

### 归档

- S8 出结果后确认后续动作 → 进入 S9 已归档
- 归档前确认：资料归档完成、续签提醒是否需要、收费是否结清

### 续签提醒

- 系统根据在留到期日在关键节点触发提醒（P0 以站内提醒为主）
- 如客户确认续签，可基于历史资料和基础信息创建新案件
- P0 不做续签商机自动创建——后置到 P1

## 8. 收费流程

### P0 最小收费流程

1. 案件创建时设定费用总额与收费节点
2. 到节点触发内部收款提醒
3. 财务登记实收、上传凭证
4. 标记回款/欠款状态

### P0 口径

- 收费异常以风险提示为主，不阻断提交
- 欠款状态下继续提交需风险确认留痕（确认人、原因）
- 工作台聚合暴露欠款风险案件

> 收费状态枚举见 [03 §3.5](03-业务规则与不变量.md#35-收费状态)。
> 收费与欠款策略见 [03 §6](03-业务规则与不变量.md#6-收费与欠款策略)。

## 9. P0 明确不做（流程层）

以下流程能力不纳入 P0：

| 不做项 | 后置到 | 理由 |
|--------|-------|------|
| 双人复核全局强制 | P1 | 模型预留，默认关闭 |
| 自动化流程引擎与条件触发器 | P2 | P0 流程推进由用户手动驱动 |
| 客户门户交互流程 | P1 | P0 聚焦内部作业闭环 |
| 续签商机自动创建 | P1 | P0 仅做到期提醒 |
| 文书初稿/审核版/定稿完整版本管理流程 | P1 | P0 用三态流转 |
| 高级催办自动升级（多级主管） | P1 | P0 仅支持基础升级 |
| 看板视图与高级工作台 | P1 | P0 用列表+基础聚合 |
