# 99 文档维护与版本记录（P0 首版）

> 本文档记录 P0 文档体系的治理规则、结构变更和同步规范。
> 不承载业务内容，只承载维护动作。
> 主目录的维护规范见 [99-文档维护与版本记录](../99-文档维护与版本记录.md)。

## P0 文档治理核心约束

### G-1：P0 是首版权威源

`P0/` 只回答"首个成交版本做什么、不做什么"。研发、设计、QA 在首版开发期间以 P0/ 为执行依据。

### G-2：P1/P2 不回灌

后续 P1/P2 需求不直接塞回 P0/，而是回主目录维护。P0/ 只接受首版口径的澄清与纠错。

### G-3：与主目录的同步规则

| 场景 | 动作 |
|------|------|
| P0/ 中发现规则错误并修正 | 检查主目录对应文档是否需同步修正 |
| 主目录中 P0 范围发生变更 | 同步反映到 P0/ 对应文档 |
| P0/ 新增文档 | 在本文"变更日志"中记录，并检查主目录 README 是否需更新 |

### G-4：文档必须重写不可复制

每篇 P0 文档都必须重写开头结论、目标对象、排除项和验收口径。禁止从主目录复制后仅做少量删减。

### G-5：试点必要性测试

每个章节、每条规则必须能回答：**"若删掉这部分，首个试点是否仍可跑通？"** 若答案是"能"，则不应进入 P0。

### G-6：一份权威源，两种阅读视图

P0 不维护“两套相互独立的权威文档”。统一规则如下：

- 各正文文档（00/01/02/03/04/05/06/07/08）是**权威源**
- `09-结构化总索引与交叉映射.md` 是**结构化导航层**
- 人先读正文，AI 可先读 `09` 再跳正文，但**不得**把 `09` 当成新的规则来源

### G-7：先结构化抽取，再执行

凡是 AI 或新成员要根据 P0 文档执行任务，必须先完成以下动作：

1. 在 `09` 中定位任务类别
2. 找到唯一权威正文
3. 确认 P0 明确不做项
4. 确认页面 / 流程 / 数据 / 术语的交叉约束

### G-8：Raw 输入不可直接执行

原始 PRD、会议纪要、IM 讨论、示例截图只能先进入 `_raw/00-inbox.md`。AI 或新成员不得直接基于 raw 内容开工，必须先完成结构化抽取、歧义清单和边界冻结。

若仓库中已存在对应原型页面，则冻结后的执行输入应优先回写到该页面的 `P0-CONTRACT*`（交互契约）与 `MIGRATION-MAPPING*`（数据契约），而不是继续平行扩写抽象文档。

### G-9：未确认项不得实现

任何 `OPEN_QUESTION`、`P0 / P1` 级歧义、待确认假设都不得由 AI 以常识补齐。示例默认不是强约束，只有在冻结边界中显式升级后才可执行。

### G-10：无 Traceability 不算完成

P0 任务完成必须能回写到 `requirement_id → task → code / test / evidence`。若只有口头说明、截图或零散记录，而无法反查需求编号，则不算 done。

## 新增 P0 文档前先问三件事

1. **这个需求是否属于首版？** — 如果不是，回主目录维护。
2. **主目录中是否已有对应文档？** — P0 文档应以主目录对应文档为基底重写，不凭空创建。
3. **这篇 P0 文档给谁用、回答什么问题？** — 明确读者和核心问题后再动笔。

## P0 需求编译流水线（最小闭环）

| 步骤 | 输入 | 输出 | 门禁 |
|------|------|------|------|
| Ingest | PRD / 会议纪要 / 聊天记录 / 原型说明 | `_raw/00-inbox.md` 条目 | 只追加，不执行 |
| Extract | raw 条目 + 权威正文 | `requirements.ir`（已有原型时优先回写 `P0-CONTRACT*` / `MIGRATION-MAPPING*`） | 目标 / 规则 / 示例 / 不做 / 未决项必须分开 |
| Clarify | `requirements.ir` | `ambiguities` | 存在 `P0 / P1` 未决项时停止 |
| Freeze | 已关闭的歧义 + 权威正文 | `boundary` | 缺少 `out_of_scope` 或 `acceptance` 不得冻结 |
| Execute | 冻结边界 + 需求 ID | 任务卡 / 实现记录 | 任务必须绑定 `REQ-P0-*`，不得直接引用 raw 口语描述 |
| Verify / Write-back | 实现 / 测试 / 验收证据 | `traceability`、`_output/00-outputs.md` | 没有映射证据，不算完成 |

### P0 允许的最小落地方式

P0 阶段不强制先做平台。`requirements.ir / ambiguities / boundary / traceability` 可以先以文档表格、任务系统字段、PR 模板或结构化文本存在，只要字段齐全、可审阅、可回链即可。

若页面已有原型，优先使用现成载体承接：

- `packages/prototype/admin/*/P0-CONTRACT*.md`：交互契约、状态、按钮、反馈、回归清单
- `packages/prototype/admin/*/MIGRATION-MAPPING*.md`：字段映射、domain/data/model/ui 落点、生产实现边界

## 文档模板规范

P0 文档沿用主目录的三类模板（详见主目录 [99 §文档模板规范](../99-文档维护与版本记录.md#文档模板规范)）：

| 模板 | 适用范围 | P0 额外要求 |
|------|---------|-----------|
| 模板 A（入口文档） | 00、02-全景图 | 必须明确"P0 不做什么" |
| 模板 B（门禁正文） | 02-版本范围、03、04、05、07、08 | 统一头部表必须包含"不负责什么"并链接到主目录；头部后必须补 `0.结构化速查（人 / AI 共用）` |
| 模板 C（页面规格） | 06-页面规格/ | MVP/P1 边界必须明确标注；概览表后必须补 `0.结构化速查（人 / AI 共用）` |

结构化导航层补充规则：

- `09` 必须只做索引、映射、冻结边界汇总，不得新增独立规则
- 当 02/03/04/05/06/07/08 任一篇发生结构变化时，必须同步检查 `09`
- 当 02 的功能范围/验收矩阵、06 的页面承载、07 的数据对象发生变化时，必须同步回写 `09` 中的“需求 → 页面 → 数据 → 验收”矩阵
- 执行型需求必须先在 `09 §7` 绑定 `REQ-P0-*` 编号，再进入任务拆解与实现
- 若正文和 `09` 冲突，以正文为准，并在本日志记录修复

P0 额外写作约束：
- 每篇文档开头必须有一句话说明"本文档只覆盖 P0 范围"
- 排除项必须显式列出，不可省略
- 必须标注与主目录对应文档的关系（链接）
- 需要兼容人读与 AI 读：正文保留背景与解释，结构化边界统一回收到 `09`

## 最小中间产物模板（P0）

| 产物 | 最少字段 | 用途 |
|------|---------|------|
| `requirements.ir` | `id`、`type`、`statement`、`source`、`priority`、`status` | 把自然语言编译成机器可校验条目；已有原型时可嵌入 `P0-CONTRACT*` / `MIGRATION-MAPPING*` |
| `ambiguities` | `id`、`question`、`severity`、`owner`、`status` | 暴露未决问题，阻止 AI 脑补 |
| `boundary` | `goal`、`in_scope`、`out_of_scope`、`acceptance` | 冻结边界，定义执行契约 |
| `traceability` | `requirement_id`、`task_id`、`code_ref`、`test_ref`、`status` | 证明需求已实现并验证 |

### 字段使用约束

- `requirements.ir.type` 最少使用：`OBJECTIVE` / `RULE` / `CONSTRAINT` / `EDGE_CASE` / `OUT_OF_SCOPE` / `EXAMPLE` / `OPEN_QUESTION`
- `EXAMPLE` 默认不是规则；只有在 `boundary` 中升级后才可执行
- `traceability.requirement_id` 必须使用 `09 §7` 中的 `REQ-P0-*`
- `boundary.out_of_scope` 是冻结前必填项；缺失则视为门禁失败

## 固定检查表

每次变更 P0 文档后，回答以下问题：

- [ ] 变更内容是否仍属于 P0 首版范围？（若不属于，应回主目录维护）
- [ ] 是否在本文的"变更日志"中记录了本次变更？
- [ ] P0/README 中的文档目录和状态是否需要更新？
- [ ] 是否需要同步检查主目录对应文档？
- [ ] 若变更影响需求、页面、数据或验收，是否同步更新了 `09` 的回写矩阵？
- [ ] raw 输入是否已进入 `_raw/00-inbox.md`，且没有被直接当作执行输入？
- [ ] 是否已形成 `requirements.ir / ambiguities / boundary` 最小中间产物？
- [ ] 若对应页面已有原型，是否已把冻结结果回写到对应 `P0-CONTRACT* / MIGRATION-MAPPING*`？
- [ ] 是否还有 `P0 / P1` 歧义未关闭却进入执行？
- [ ] `out_of_scope` 是否已显式冻结？
- [ ] 任务 / 代码 / 测试 / 验收是否都引用了 `REQ-P0-*`？
- [ ] 新增文档是否按模板规范编写（开头一句话 P0 范围说明、排除项、主目录链接）？
- [ ] 是否通过了试点必要性测试（"删掉这部分，试点是否仍可跑通"）？
