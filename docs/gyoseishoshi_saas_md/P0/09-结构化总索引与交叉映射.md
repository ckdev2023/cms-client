## 1. 使用规则

### 1.1 一份权威源，两种阅读视图

- **人读视图**：仍以各正文文档的背景、为什么、流程解释、例子为主
- **AI 读视图**：优先读取本文的结构化索引，再跳转到对应正文锚点
- **禁止**把本文当成独立规则源；若正文与本文冲突，以正文为准，并应立即修正文档索引

### 1.2 AI 执行前最小读取顺序

1. 先读本文，确定任务属于哪一类
2. 再读对应权威正文
3. 最后读落点页面或数据定义

### 1.3 执行型需求的强门禁

- 原始 PRD / 会议纪要 / IM 讨论 / 示例截图 **不得直接实现**，必须先进入 `_raw/00-inbox.md`
- 开工前必须至少形成 `requirements.ir`、`ambiguities`、`boundary` 三类中间产物；若仓库中已有对应原型页，则应优先把冻结后的交互和数据要求回写到 `packages/prototype/admin/*/P0-CONTRACT*.md` 与 `MIGRATION-MAPPING*.md`
- 任何 `P0 / P1` 级未决问题未关闭，或 `out_of_scope` 未冻结，均不得进入任务执行

### 1.4 四份中间产物与本文的关系

| 中间产物 | 作用 | 在 P0 中如何承接 |
|---|---|---|
| `requirements.ir` | 把自然语言需求拆成机器可校验条目 | 用本文 §2 / §3 / §7 作为 requirement 定位和回链种子；已有原型时优先承接到对应 `P0-CONTRACT*` / `MIGRATION-MAPPING*` |
| `ambiguities` | 显式暴露未决问题，禁止 AI 脑补 | 每条歧义都必须回链到对应正文或本文的排除项 / 术语项 |
| `boundary` | 冻结 `in_scope / out_of_scope / acceptance` | 以本文 §3、§7、§9 为冻结边界的统一索引入口 |
| `traceability` | 证明需求已被实现并验证 | 以本文 §7 为主表，回写 task / code / test / evidence |

### 7.1 `REQ-P0-01` 原型锚点（按现有原型落地）

> 用途：不再新增抽象模板，而是把 `REQ-P0-01` 直接冻结到现有原型文件，供 AI 按交互契约与数据契约落地。

- `Lead` 侧交互起点：
  - `packages/prototype/admin/leads-message/detail.html`
  - `packages/prototype/admin/leads-message/P0-CONTRACT-DETAIL.md`
  - `packages/prototype/admin/leads-message/MIGRATION-MAPPING-DETAIL.md`
- `Customer` 侧承接点（建档）：
  - `packages/prototype/admin/customers/P0-CONTRACT.md`
  - `packages/prototype/admin/customers/MIGRATION-MAPPING.md`
- `Case` 侧承接点（首个案件创建）：
  - `packages/prototype/admin/case/create.html`
  - `packages/prototype/admin/case/P0-CONTRACT.md`
  - `packages/prototype/admin/case/MIGRATION-MAPPING.md`

AI 落地 `REQ-P0-01` 时，最小执行顺序固定为：

1. 在 Lead 详情页 `转化信息` Tab 完成去重提示与人工确认；
2. 先创建 Customer，继承 `Lead.group`，如改组则记录原因；
3. 再创建首个 Case，预填主申请人并继承 `Customer.group`；
4. 转化结果必须回填可跳转的 Customer / Case 入口，并保留 traceability。
