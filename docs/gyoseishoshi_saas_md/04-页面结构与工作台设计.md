# 04 页面结构与工作台设计

## 1. 信息架构总览

建议系统采用以下一级导航：

1. 仪表盘
2. 咨询线索（可选：如需覆盖销售/前台咨询转化）
3. 客户
4. 案件
5. 资料中心
6. 文书中心
7. 任务与提醒
8. 收费与财务
9. 报表
10. 系统设置
11. 客户门户

### 1.1 分阶段导航建议

为避免完整信息架构与 MVP 范围冲突，建议按版本分层展示导航：

- P0 主导航：仪表盘、咨询线索、客户、案件、任务与提醒、收费与财务、系统设置（基础）
- P0 次级入口：资料中心、文书中心可先作为案件详情内强入口，减少首版导航负担
- P1/P2 扩展导航：报表、客户门户、独立资料中心/文书中心、高级设置

### 1.2 全局交互原则
- 全局搜索应常驻顶部，至少支持案件编号、客户名、电话、邮箱、公司名检索
- 列表页统一支持：筛选、排序、保存视图、批量操作、导出权限控制
- 页面中的“内部可见/客户可见”信息必须有明确视觉区分，避免误发与误读
- 空状态页要明确下一步动作，例如“去建案”“去催办”“去生成模板文书”

### 1.3 首版页面收敛原则
- P0 以内部后台工作台为主，不把客户门户作为首版主导航核心模块
- P0 强调“从案件详情完成主要操作”，资料中心、文书中心可以先通过案件页强入口承载
- 复杂报表、独立高级中心、门户高级交互统一后置，避免导航完整但能力不完整
- 首版模板演示与页面字段设计优先围绕“家族滞在”“工作签证（技人国）”两类高频案件展开

### 2.1 面向对象
- 行政书士
- 负责人
- 助理
- 财务

### 2.2 核心卡片
- 今日待办
- 即将到期案件
- 待补件案件
- 待提交案件
- 审理中案件
- 待回款案件
- 本月新增案件
- 风险案件

### 2.3 视图建议
- 列表视图
- 看板视图
- 日历视图

### 2.4 工作台聚合口径
- 今日待办：到期在今天的任务、今日需跟进的线索、今日需处理的补正/审核动作
- 即将到期案件：7/30 天内存在关键期限（在留到期、补件截止、预约日、提交截止）的案件
- 待提交案件：已完成提交前校验，且满足复核要求（如启用）并处于 S6 待提交阶段的案件
- 风险案件：存在硬性阻断、关键资料逾期、补正临近到期、欠款影响提交或被标记高风险的案件
- 待回款案件：收费节点已到但未结清的案件

### 2.5 角色化默认视图
- 行政书士/主办人：待提交、风险案件、双人复核（如启用）、审理中案件
- 助理：资料审核、缺件催办、今日到期、待补正
- 销售/前台：新线索、待签约、待转案件
- 财务：待回款、开票状态、异常付款记录

## 4. 案件模块

### 4.0 创建/建案体验（关键）
建案时选择签证类型/申请类型后，系统应自动生成该类型的通用资料清单（资料项），并在建案完成后立即展示，避免用户逐条手工新增。
建议交互：
- 建案表单必选：签证类型、申请类型（认定/变更/更新）
- 创建完成落地页默认进入“资料清单”Tab，并提示“已按模板生成 N 项通用材料”
- 支持两类兜底动作：新增资料项（特殊材料）、标记不适用/无需提供（需原因留痕）

- 案件类型

- 企业客户

- 负责人
- 资料完成率
- 最近更新时间

## 9. 客户门户（P1）

客户门户作为 P1 优先增强模块上线，不纳入 P0 主导航与主流程。

### 9.1 主要入口
- 我的案件
- 资料上传
- 缺件清单
- 通知提醒
- 消息中心
- 付款中心
- 下载中心

### 9.2 客户体验原则
- 少专业术语
- 明确下一步动作
- 明确资料示例
- 明确谁在负责
- 明确是否已收到和审核通过
补充：
- 默认只展示客户需要采取的动作与结果确认（资料项状态、退回原因、截止日、付款状态）
- 内部风险/审计明细不对客户展示，避免误解与敏感信息泄露
- P1 首批页面入口建议仅保留“我的案件 / 资料上传 / 缺件清单 / 付款中心”，复杂消息和下载能力后续增强
